# Выпуск релизов PopBot

Релизы собираются GitHub Actions на **macOS, Windows и Linux** и публикуются в GitHub Release этого репозитория. Каждая платформа собирается на своём собственном раннере — нативные модули (`better-sqlite3`, `node-pty`) должны компилироваться под ABI Electron для каждой ОС, так что кросс-компиляция не вариант.

## Выпуск релиза

Из чистого рабочего дерева на `main`:

```bash
npm run release            # patch bump (default)
npm run release -- minor   # minor bump
npm run release -- major   # major bump
```

`scripts/release.sh` повышает версию, коммитит, создаёт аннотированный тег `vX.Y.Z` и пушит оба. Запушенный тег запускает workflow **Build**, который собирает все три платформы и публикует GitHub Release с прикреплёнными артефактами. Наблюдайте за этим через `gh run watch` или вкладку Actions.

Следующая версия вычисляется из последнего тега `v*`, повышенного согласно аргументу выше. До существования какого-либо тега используется откат к версии в `package.json` (так что первый релиз — это следующее повышение поверх неё). Скрипт отказывается запускаться из любой ветки, кроме `main` (переопределяется через `RELEASE_BRANCH=<name>`).

## Что производится

| Платформа | Артефакты |
|----------|-----------|
| macOS    | `.dmg`, `.zip`, `latest-mac.yml`, `.blockmap` |
| Windows  | NSIS-установщик `.exe`, `.zip`, `latest.yml`, `.blockmap` |
| Linux    | `.deb` (без автообновления — см. заметку про Linux ниже) |

Файлы `latest*.yml` + `.blockmap` — это метаданные electron-updater (генерируются конфигурацией [`electron-builder.yml`](../../electron-builder.yml) `publish: github`). Автообновлятель внутри приложения использует их для обнаружения, загрузки и подготовки обновлений — см. раздел «Автообновление» ниже.

Workflow: [`.github/workflows/build.yml`](../../.github/workflows/build.yml).

## Триггеры CI

- **Push тега `v*`** → сборка всех платформ (подписанная, если заданы секреты) + публикация GitHub Release.
- **Pull request в `main`** (не только документация) → только валидационная сборка, **всегда неподписанная**; артефакты прикрепляются к запуску, ничего не публикуется, секреты не используются.
- **Вручную** → «Run workflow» (workflow_dispatch), неподписанная.

Подписание запускается только при push тега `v*`, что может сделать только владелец репозитория. GitHub никогда не раскрывает секреты для запусков PR, триггерённых форком, так что PR от контрибьюторов не могут дотянуться до сертификатов подписи.

## Подписание кода

Подписание управляется через **секреты GitHub Actions** (Settings → Secrets and variables → Actions). Они зашифрованы, никогда не находятся в git-дереве и маскируются в логах. Без заданных секретов сборки по тегу производят неподписанные бинарники (macOS Gatekeeper / Windows SmartScreen предупреждают при первом запуске), и CI всё равно проходит.

### macOS (подпись + нотаризация)

| Секрет | Значение |
|--------|-------|
| `MAC_CSC_LINK` | base64 вашего `.p12` "Developer ID Application" (`base64 -i cert.p12 \| pbcopy`) |
| `MAC_CSC_KEY_PASSWORD` | пароль для этого `.p12` |
| `APPLE_ID` | email Apple ID, используемый для нотаризации |
| `APPLE_APP_SPECIFIC_PASSWORD` | специфичный для приложения пароль с appleid.apple.com |
| `APPLE_TEAM_ID` | ID команды Apple Developer |

Сборка по тегу подписывает + нотаризует только тогда, когда присутствует **полный набор** — `MAC_CSC_LINK`, `APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD` **и** `APPLE_TEAM_ID` (плюс `MAC_CSC_KEY_PASSWORD` для сертификата). Если чего-то не хватает, сборка производится неподписанной, а не проваливает нотаризацию позже — так что наполовину настроенный набор секретов не сломает CI.

### Windows (опционально)

| Секрет | Значение |
|--------|-------|
| `WIN_CSC_LINK` | base64 вашего `.pfx` для подписи кода |
| `WIN_CSC_KEY_PASSWORD` | пароль для этого `.pfx` |

Сборка по тегу подписывает, когда присутствует `WIN_CSC_LINK`; иначе — неподписанная.

## Автообновление

Автообновление внутри приложения подключено через **electron-updater** ([`src/main/updates/autoUpdate.ts`](../../src/main/updates/autoUpdate.ts)). В упакованных сборках оно опрашивает релизы этого репозитория, **тихо загружает** более новую версию в фоне и показывает тост **"Restart to install"**, когда она подготовлена — клик по нему завершает работу и перезапускает в новую версию. Оно читает метаданные `latest*.yml` + `.blockmap`, прикрепляемые workflow релиза; конфигурация `publish: github` в `electron-builder.yml` встраивает `app-update.yml`, нужный клиенту.

**Подписание требуется для шага установки.** macOS отклоняет неподписанные обновления, так что установка внутри приложения работает только после того, как релизы подписаны + нотаризованы (путь сборки по тегу с заданными секретами Apple). До тех пор — и всякий раз, когда автообновлятель сталкивается с ошибкой (нет метаданных, сбой сети) — он **откатывается** к ручному тосту «Download», открывающему страницу релиза, управляемому лёгкой проверкой GitHub в [`src/main/updates/check.ts`](../../src/main/updates/check.ts). Та же лёгкая проверка также обслуживает кнопку «Check for updates» по запросу в диалоге «О программе» и работает везде, включая режим разработки и неподписанные сборки.

Чтобы что-либо из этого проявило релиз, workflow должен публиковать **недрафтовые, не пререлизные** релизы с прикреплёнными установщиками для платформ — что он и делает. Автообновление отключено в режиме разработки.

### Проверка автообновления (первый сквозной тест)

Путь автообновления можно проверить только против **двух реальных подписанных релизов** — не в режиме разработки (там оно отключено) и не против одного релиза (нет ничего новее, что можно было бы подтянуть). Сделайте это один раз, после настройки подписания:

1. **Подтвердите, что подписание включено.** Добавьте секреты macOS (и опционально Windows) из таблицы выше. Первый подписанный релиз должен пройти успешно — на macOS неподписанные/ненотаризованные сборки могут загружаться, но **не могут установиться**, так что весь этот тест бессмыслен без подписи.
2. **Выпустите релиз N**, например `npm run release` → `v0.0.18`. Дождитесь, пока workflow опубликует релиз с ассетами + `latest*.yml`.
3. **Установите N из опубликованного релиза** на каждой поддерживаемой ОС (macOS `.dmg`, Windows `.exe`, Linux `.deb`). Запустите его — убедитесь, что Help ▸ About показывает правильную версию.
4. **Выпустите релиз N+1**, например `npm run release` → `v0.0.19`.
5. **Оставьте установку N запущенной.** В течение ~30 с после запуска (а затем каждые 6 ч) она проверяет обновления; на подписанной сборке она тихо загружает N+1, затем показывает тост **"Restart to install"**. Кликните по нему.
6. **Подтвердите, что произошёл перезапуск в N+1** — Help ▸ About теперь показывает новую версию. Это доказывает, что загрузка → подготовка → quitAndInstall → перезапуск работают на этой ОС.

Заметки для каждой платформы:
- **macOS:** Squirrel.Mac применяет обновление из ассета `.zip` (не `.dmg`); оба должны быть в релизе. Gatekeeper отклоняет неподписанное/ненотаризованное обновление — если «Restart to install» ничего не делает, перепроверьте нотаризацию сборки.
- **Linux:** `.deb` **не** самообновляется — electron-updater автообновляет только AppImage на Linux. Обновляйте, устанавливая новый `.deb` (`sudo dpkg -i …` / `sudo apt install ./…`). Так что пропустите шаги автообновления (4–6) для Linux; просто установите N+1 поверх N и подтвердите через About. Чтобы восстановить автообновление внутри приложения на Linux, добавьте обратно `AppImage` в `linux.target` в `electron-builder.yml`.
- **Windows:** установка NSIS обновляется на месте; SmartScreen может предупреждать, пока сборка не подписана `WIN_CSC_LINK`.

Если шаг 5 вместо этого показывает тост **«Download»** (открывающий страницу релиза), значит автообновлятель внутри приложения столкнулся с ошибкой и откатился — проверьте диагностический лог (записи `update.error` / `update.check.failed`) на предмет причины, чаще всего это неподписанная сборка macOS.
