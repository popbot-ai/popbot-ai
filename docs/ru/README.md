## Зачем PopBot

Запустить одного AI-агента для написания кода несложно. Запуск многих одновременно порождает проблемы, которых нет у одного агента: нужно изолировать их работу, чтобы они не перезаписывали друг друга, реально тестировать то, что они создают, проверять это ревью и блокировать необратимые действия, чтобы ни один агент не совершил их без надзора.

PopBot — это уровень оркестрации именно для этого. Он превращает тикеты и запросы на ревью в сессии агента в один клик, даёт каждому агенту изолированное рабочее пространство (собственную рабочую копию — а для игровых проектов ещё и собственный запущенный экземпляр тестируемого приложения), запускает их автономно по умолчанию с человеческим контролем на рискованных действиях и собирает каждую стенограмму, диф, терминал и лог в одном окне. Оператор пробегает взглядом по колонкам, одобряет действия, требующие подтверждения, и отправляет изменения.

Он был создан небольшой командой в **Proof of Play** и использовался ежедневно на реальном, насыщенном ассетами продакшен-проекте, который вышел в релиз. Именно в такой среде он и был проверен: многие гигабайты ассетов, настоящая система контроля версий, реальные дедлайны. Модель слотов — тёплые, изолированные рабочие пространства с copy-on-write — это то, что сделало параллельный запуск агентов практичным там, и это увеличило объём работы, который команда могла выполнять одновременно. Мы публикуем и поддерживаем PopBot как референсную реализацию: не готовый продукт для использования «как есть», а форму, которую нужно взять и переработать под свой стек и рабочий процесс. Это отражает взгляд на то, как лучше всего создавать ПО в эпоху ИИ — что командам, управляющим флотами агентов, лучше владеть инструментом и переделывать его, чем брать готовый. Он распространяется под лицензией MIT и организован так, чтобы его было удобно форкать; см. [Сделайте его своим](#сделайте-его-своим).

## Ключевые возможности

### Мультичат-вид с живыми превью

Каждый открытый чат остаётся на экране — полоса **живых превью** над расположенными рядом **колонками**. Каждое превью — это реальный, обновляющийся вид этого чата (а не просто индикатор статуса), окрашенный по состоянию: выполняется, готово, ждёт вас, ошибка. Одним взглядом вы видите, *чем занят каждый агент* и кому нужны вы — и вы можете **поймать неверный путь на раннем этапе**, перенаправив агента прежде, чем он потратит время и токены впустую. Один человек управляет целым флотом из одного окна.

### Тёплые слоты — параллельные агенты без налога на повторный импорт

Каждый работающий чат арендует **слот** — постоянную рабочую копию плюс собственное тёплое состояние сборки, созданное один раз и используемое многократно. Для игрового движка это значит, что слот хранит собственный горячий кеш ассетов (`Library` у Unity, DDC у Unreal) и может держать редактор запущенным, так что переключение агента обратно в его слот занимает **секунды, а не многоминутный повторный импорт**. Десять агентов работают в настоящей изоляции веток, не забивая единый кеш импорта.  [Как работают слоты →](GUIDE.md#слоты-тёплые-изолированные-одноразовые-рабочие-пространства)

### Неограниченное число копий на диске одного репозитория

Рабочее пространство слота — это **папка с copy-on-write**: каждый слот использует один общий базовый образ и хранит только то, что меняет. Поэтому свежая, живая, полная копия дерева игрового проекта **терабайтного масштаба** готова за **секунды** — это настоящие редактируемые файлы, а не поверхностное представление — а неограниченное число копий стоит места одного репозитория на диске. Это работает на **Windows, macOS и Linux** и именно это позволяет огромным деревьям Perforce вообще присоединиться к флоту. [Почему это важно →](GUIDE.md#copy-on-write-неограниченное-число-копий-на-диске-одного-репозитория)

### Git и Perforce со встроенным ревью

Система контроля версий — это **провайдер** за единым интерфейсом: **Git** (worktree, ветки, PR через `gh`) и **Perforce** (стримы поверх теневых рабочих пространств, чейнджлисты, ревью в **Helix Swarm**) — оба реализованы как полноценные варианты. Панель контроля версий, привязанная к *собственному рабочему пространству каждого чата*, показывает статус, коммиты и подиффайловые различия именно для этой ветки. Шаблонные действия в один клик (**Commit**, **Push PR**, **Make ready**, **Address CR**, **Rebase onto base**) отправляют предзаполненную инструкцию агенту этого чата, с уже подставленными `${branch}` / `${ticket}` / `${prnum}`.

### Один инбокс, много источников

Весь цикл в одном месте: ваш **инбокс** — назначенные тикеты из **Linear**, **Jira** и **GitHub Issues**, плюс ревью, ожидающие вас, в виде **GitHub PR** и **чейнджлистов Swarm** → работа агента **в процессе** в изолированных слотах → **push** и открытие PR / ревью → **архивация** завершённого чата → **повторное открытие и перезапуск** позже с полной историей. Кликните по тикету — и PopBot назовёт ветку, арендует слот, переведёт тикет в *In Progress* и подготовит агента — а затем проведёт его через весь путь до смёрженного изменения и обратно. [Пошаговые разборы workflow →](GUIDE.md#сквозные-workflow)

## Дополнительные возможности

- **Настоящие Claude Code и Codex — а не их переизобретение.** Каждый чат управляет *настоящим* агентом через его официальный SDK — теми же CLI-инструментами `claude` и `codex`, что вы запускаете в терминале, со всеми их инструментами, скиллами и MCP-серверами в неприкосновенности. Выбирайте модель (Opus / Fable / GPT) и уровень рассуждений для каждого чата, переключайтесь на лету или перезапускайте свежую сессию, подготовленную историей чата.
- **Агенты, которые тестируют собственную работу.** Слот может запустить настоящее приложение — для Unity и Unreal это живой редактор + вспомогательный сервер на втором дисплее, управляемый агентом через MCP-сервер в редакторе на **отдельном для каждого слота порту** — так что агент кликает по интерфейсу, читает логи и проверяет свои изменения вместо того, чтобы гадать. Пользовательские движки тоже поддерживаются.
- **Постоянные, архивируемые чаты.** Каждый чат — это долговечная стенограмма; закройте его, чтобы освободить слот, и откройте позже с сохранённой полной историей.
- **Терминал для каждого чата и кликабельный код.** Встроенный терминал, привязанный к рабочему пространству чата, и ссылки вида `file.ts:42`, открывающиеся в VS Code или Cursor.
- **Автономны, но никогда безрассудны.** Агенты автоматически выполняют безопасную работу внутри своего слота и приостанавливаются, ожидая вас, на всём рискованном — `git push` / `p4 submit`, открытие PR, что-либо за пределами рабочего пространства, сетевые вызовы. Разрешения выдаются на уровне чата, они долговечны и отзываемы — включая MCP-серверы.
- **Полностью локализован.** Весь интерфейс поставляется на восьми языках (английский, испанский, французский, немецкий, японский, корейский, упрощённый китайский, бразильский португальский), переключаемых в любой момент из меню языка.
- **Мультирепозиторий.** Управляйте несколькими репозиториями одновременно, у каждого свой пул слотов, цвет, провайдер и соглашения о ветках.

## Чем PopBot отличается

Инструменты для агентного написания кода обычно делятся на несколько категорий. PopBot занимает другое место: **локальная кабина управления для запуска множества *настоящих* агентов параллельно, с тёплым состоянием сборки и живым человеческим надзором.**

| Вместо… | …PopBot |
|---|---|
| **Одного агента в терминале или IDE** — одна задача в одном рабочем дереве за раз | **Много агентов одновременно**, каждый изолирован в своём тёплом слоте, все видны как живой флот, которым вы управляете из одного окна |
| **Асинхронных облачных агентов** — непрозрачных и удалённых; отправляете задачу, ждёте PR | **Локально и вживую** — вы наблюдаете за работой каждого агента и ловите неверный путь на раннем этапе, а агент управляет *вашим настоящим приложением* (редактором движка на втором экране) для полноценного сквозного тестирования |
| **Самодельного жонглирования `tmux` + worktree** — параллельно, но вручную, и каждый свежий чекаут платит многоминутный налог движка на повторный импорт | **Управляемых тёплых слотов** — переиспользуемых рабочих пространств с copy-on-write, которые держат кеш ассетов горячим, с жизненным циклом веток/рабочих пространств, панелью SCM и ревью кода, о которых уже позаботились |
| **Фреймворков для оркестрации агентов** — наборов инструментов для *создания* агентных систем | **Готового, продуманного приложения**, подключённого к вашему инбоксу и циклу ревью — human-in-the-loop по замыслу, а не библиотека для самостоятельной сборки |

И что критично: PopBot не заменяет Claude Code или Codex — он их **запускает**. Вы получаете именно тех агентов (и именно те версии CLI), которым уже доверяете, просто по многу одновременно, с оркестрацией, изоляцией и надзором, обёрнутыми вокруг них.

## Анатомия рабочего пространства

![PopBot UI anatomy](../../images/anatomy.png)

| Область | Что это |
|---|---|
| **Инбокс — тикеты и ревью** | Назначенные тикеты (Linear / Jira / GitHub Issues) и ожидающие вас ревью (GitHub PR / чейнджлисты Swarm), ранжированные по приоритету. Один клик порождает чат. |
| **Слоты** | Пул тёплых, изолированных рабочих пространств — рабочая копия с copy-on-write *плюс* постоянное состояние сборки (для игрового движка — собственный горячий кеш ассетов). Чат арендует слот на время работы и возвращает его при закрытии. |
| **Архив чатов** | Каждый прошлый чат, доступный для поиска и повторного открытия с полной историей. |
| **Превью чатов** | Живая полоса всех открытых чатов — окрашенная по статусу (выполняется / готово / нужны вы / ошибка). |
| **Чаты** | Сфокусированные сессии агентов: текст, вызовы инструментов и встроенные диффы кода, транслируемые вживую. |
| **Терминал для каждого чата** | Встроенный терминал, указывающий на рабочее пространство этого чата, для ручных команд. |
| **Панель SCM** | Статус рабочего дерева / чейнджлиста, коммиты, файловые диффы и действия commit / push / PR / review в один клик. |

## День с PopBot

**Тикет с функциональностью.** Тикет попадает в ваш инбокс. Кликните по нему → PopBot открывает чат на ветке `you/eng-123-…`, арендует слот, переводит тикет в *In Progress* и передаёт агенту полное описание. Агент пишет код, запускает приложение в своём слоте для проверки и приостанавливается, ожидая вашего ОК перед push. Вы просматриваете дифф в панели SCM и нажимаете **Push PR**.

**Баг — параллельно.** Пока это выполняется, приходит отчёт об ошибке. Порождаете второй чат — свой слот, своя ветка — и оба агента работают одновременно, ни разу не касаясь дерева друг друга. Полоса превью показывает оба: один зелёный (готово), один синий (выполняется).

**Запрос на ревью.** PR коллеги (или чейнджлист Swarm) появляется во вкладке «Ревью». Кликните по нему → мгновенно открывается **безрепозиторный** чат ревью, агент читает дифф *и* окружающий код, ищет реальные баги и публикует встроенный комментарий на GitHub или в Swarm — пока ваши два чата сборки продолжают работать.

**Продолжите завтра.** Закройте завершённые чаты, чтобы освободить их слоты. На следующее утро откройте чат с функциональностью из архива, чтобы обработать отзывы ревью — агент возобновляет работу со всей беседой и рабочим пространством в неприкосновенности.

→ Полные разборы (сценарии для функций, багов и ревью, а также как под капотом работают слоты, рабочие пространства с copy-on-write и повторное открытие) — в **[Руководстве по функциям и workflow](GUIDE.md)**.

## Установка

Подписанные, готовые к использованию установщики доступны на **[popbot.app](https://popbot.app)**:

- **macOS** — подписанный и нотаризованный `.dmg` (Apple silicon)
- **Windows** — подписанный установщик `.exe`
- **Linux** — пакет `.deb`

Приложение автоматически обновляется из своего канала релизов. Чтобы вместо этого собрать свою сборку, см. [Сборка из исходников](#сборка-из-исходников).

## Сборка из исходников

```bash
npm install
npm run dev        # запустить приложение в режиме разработки
npm run package    # собрать подписанный установщик для вашей платформы
```

**Требования**

- **macOS, Windows или Linux.** macOS — наиболее проверенная платформа (workflow с тестируемым приложением на втором дисплее опирается на macOS Accessibility API); Windows и Linux поддерживаются и поставляются — см. [WINDOWS.md](WINDOWS.md) для заметок по настройке Windows/WSL.
- **Node 20+** (Node 20 / 22 позволяют избежать перекомпиляции нативных модулей; см. заметки по Windows).
- CLI **`claude`** и/или **`codex`** (бэкенды агентов), плюс **`git`** и, для GitHub-сценариев, **`gh`**. Для Perforce — CLI **`p4`**.
- Учётные данные (Linear, Jira, GitHub, Helix Swarm) хранятся **локально на вашей машине**, в собственной базе данных приложения — никогда в этом репозитории.
- Опционально: редактор Unity или Unreal для игровых проектов; VS Code / Cursor; iTerm.

## Сделайте его своим

PopBot публикуется как референсная реализация, предназначенная для форка и адаптации, а не для использования «как есть». Его форма универсальна — **агенты + изолированные, тёплые слоты с copy-on-write + инбокс-как-очередь + тестируемое приложение** — а код организован как *провайдеры за небольшими общими интерфейсами*, так что команда может заменить одну часть, не трогая остальное. Он распространяется по лицензии **MIT**. Общий подход — сохранить базовые идеи и заменить конкретные реализации:

- **Замените тестируемое приложение.** Unity и Unreal — это две реализации идеи «пусть агент запускает и проверяет приложение». Хук для пользовательского движка уже передаёт идентичность слота в вашу команду запуска — направьте её на своё веб-приложение, CLI или тестовый харнесс. *(`src/shared/gameEngine.ts`, `src/main/ipc/apps.ts`)*
- **Направьте инбокс в другое место.** Linear, Jira и GitHub Issues — рабочие примеры; добавьте трекер, реализовав один интерфейс и зарегистрировав его. *(`src/main/tickets/`)*
- **Добавьте или замените систему контроля версий.** Расширьте базовый класс провайдера рядом с Git и Perforce; вызывающий код определяет ветвление по *возможностям*, никогда по идентификатору провайдера. *(`src/main/scm/`)*
- **Перенастройте действия и промпты.** Соглашения о ветках, потоки PR/ревью и каждый затравочный промпт — редактируемые шаблоны в настройках, без единой строки кода.
- **Сохраните ядро.** Тёплые слоты, рабочие пространства с copy-on-write, постоянные чаты, жёстко закодированный порог разрешений и кабина управления параллельными агентами — это устойчивый стержень.
