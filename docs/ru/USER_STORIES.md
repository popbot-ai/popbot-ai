# Пользовательские истории

Справочник «как выглядит успех» для PopBot. Зафиксировано 2026-05-01. Каждое
решение по реализации должно прослеживаться до одной из них.

Пользователь — единственный разработчик (Бен), запускающий PopBot на
собственной машине. «Я» ниже — это он.

> **Статус (аннотация добавлена в 2026-07, на момент публикации).** Истории
> ниже — это *основополагающие* пользовательские истории, зафиксированные в
> 2026-05, сохранённые здесь как оригинальная запись замысла дизайна. С тех
> пор PopBot был обобщён далеко за пределы того первоначального однопользова-
> тельского охвата Unity/Linear/Slack/GitHub — теперь он охватывает Git и
> Perforce, Unity и Unreal, Linear/Jira/GitHub Issues, GitHub PR и Helix
> Swarm, и поставляется локализованным на несколько языков под лицензией MIT.
> Этот документ намеренно *не* подгоняется под это; относитесь к нему как к
> истории, а актуальный набор функций см. в [GUIDE.md](GUIDE.md). Истории
> US-1..US-9 и снимок 2026-05 не изменены.

## US-1 · Осведомлённость об очереди внимания

> *«Я должен быть осведомлён о высокоприоритетных задачах, сообщениях Slack и
> других PR, на которые мне нужно обратить внимание».*

Три источника, отображаемые вместе в верхней части окна:

- **Тикеты Linear**, назначенные на меня, ранжированные по приоритету + сроку.
- **Сообщения Slack**, адресованные мне (личные сообщения, @упоминания,
  каналы, которыми я владею). _Новое требование; отсутствовало в
  оригинальном дизайне — см. [Отклонения](#отклонения-и-дополнения)._
- **PR GitHub**, запрашивающие моё ревью.

Каждая строка показывает достаточно с первого взгляда, чтобы отсортировать
без клика (заголовок, источник, возраст, индикатор приоритета).
Высокоприоритетные элементы визуально выделяются на фоне низкоприоритетных.

**Соответствует:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)
(панели Tickets / Reviews — расширить панелью Slack).

## US-2 · Активация в один клик

> *«Я должен иметь возможность легко инициировать активность по любому из
> них и открыть чат, чтобы начать работу».*

Клик по любой строке в очереди внимания порождает новый чат, засеянный для
этой работы:

- Тикет Linear → чат, засеянный телом тикета, ветка названа по ключу тикета,
  промпт агента предзаполнен.
- Сообщение Slack → чат, засеянный контекстом беседы, готовый набросать
  ответ или начать реальную работу.
- PR → чат, засеянный диффом и чек-листом ревью.

Никакого трения между «я вижу нечто, что нужно обработать» и «агент уже
работает над этим».

**Соответствует:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)
(«Клик по строке → порождение чата, засеянного для этой работы»).

## US-3 · Настоящее тестирование игры в чате

> *«Чаты должны иметь возможность задействовать экземпляр Unity и запускать
> unity/server, когда это нужно, чтобы тестировать и отлаживать работу».*

Когда чату нужно проверить поведение в реальной игре, чат приобретает слот,
порождает Unity (размещённый на экране 2) и опционально порождает
вспомогательный сервер. Агент управляет игрой через MCP внутри редактора —
входя в режим Play, кликая по интерфейсу, делая скриншоты, читая логи,
проверяя состояние.

Приобретение слота — самая медленная часть в первый раз (~15-30 с холодного
старта); последующая активность «липкая» (~50 мс).

**Соответствует:** [POPBOT_DESIGN.md → Chat types](POPBOT_DESIGN.md#chat-types)
(Client Test / Server Test), [Slots](POPBOT_DESIGN.md#slots--the-durable-unit),
[MCP automation surface](POPBOT_DESIGN.md#mcp-automation-surface).

## US-4 · Автономное сквозное завершение с доказательством

> *«Агенты должны иметь возможность работать полностью автономно, и
> исправлять/отлаживать и завершать целый тикет, включая предоставление
> доказательства, что исправление/изменение сработало так, как требовалось,
> в markdown-документе, который можно проверить».*

В автономном режиме агент выполняет полный цикл читать → воспроизвести →
исправить → проверить без вмешательства и в конце пишет артефакт
`proof.md`. Доказательство содержит:

- **Repro** — точные шаги, продемонстрировавшие баг.
- **Before** — скриншоты + отфильтрованные дампы логов из сломанного
  состояния.
- **Root cause** — диагноз агента.
- **Fix** — дифф или резюме изменений.
- **After** — скриншоты + чистые дампы логов из исправленного состояния.
- **Verification** — повторный прогон воспроизведения, теперь успешный.

Я могу открыть `proof.md` и решить, хороша ли работа, не перезапуская
ничего сам. Пауза для ревью нужна только для рискованных операций (`git
push`, `gh pr create` и т. д.).

**Соответствует:** [POPBOT_DESIGN.md → Autonomous mode](POPBOT_DESIGN.md#autonomous-mode),
[Proof artifacts](POPBOT_DESIGN.md#proof-artifacts-agent-debug-deliverable).

## US-5 · Лёгкая многозадачность через превью

> *«Я должен иметь возможность легко переключаться между агентами, кликая по
> превью».*

Полоса превью — основная поверхность навигации для параллельной работы. Ряд
компактных предпросмотров — по одному на чат — позволяет мне мгновенно
переключаться между агентами. Клик по превью выводит этот чат на передний
план; остальные чаты продолжают работать в фоне.

Само превью сообщает состояние, а не только идентичность. См. US-6.

**Соответствует:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)
(ряд превью), фаза 3 в [PHASING.md](PHASING.md).

## US-6 · Статус с первого взгляда

> *«Я должен иметь возможность легко понять, что делает агент, и нужна ли
> ему помощь или указания от меня, с первого взгляда».*

Каждое превью чата показывает своё текущее состояние без необходимости
кликать внутрь:

| Цвет | Значение |
|---|---|
| Синий | Выполняется |
| Зелёный | Задача завершена |
| **Жёлтый** | **Приостановлено — нужен я** |
| Красный | Ошибка |
| Серый | Простой / не запущен |

Жёлтый — тот, что требует внимания. Просмотр ряда превью должен ответить на
вопрос «кто-то застрял?» менее чем за секунду. Помимо цвета, превью
показывает краткую подсказку прогресса (последнее действие, текущий шаг),
чтобы я мог решить, стоит ли погружаться.

**Соответствует:** [POPBOT_DESIGN.md → Status colors](POPBOT_DESIGN.md#status-colors-chat-thumbnail).

## US-7 · Восстановление и продолжение откуда угодно

> *«Я должен легко иметь возможность восстановить и продолжить работу с
> тикетами, даже теми, что больше не активны, с того места, на котором я
> остановился».*

Чат устойчив. Даже после того, как я его закрою, перезапущу PopBot или
перезагружу компьютер, я могу заново открыть любой прошлый чат и продолжить
ровно с того места, где остановился:

- Полная стенограмма воспроизводится в колонке чата.
- Слот приобретается заново (или поднимается с нуля) на той же ветке, на
  которой я был.
- Состояние Unity + вспомогательного сервера восстанавливается до
  релевантной фикстуры / блоба сохранения, если он был задан.
- Агент повторно читает недавнюю стенограмму перед ответом на моё следующее
  сообщение — контекст не теряется при перезапуске.

Закрытие чата освобождает его слот; повторное открытие приобретает заново.
Чат — устойчивая запись; слот — временная инфраструктура.

**Соответствует:** [POPBOT_DESIGN.md → Slots](POPBOT_DESIGN.md#slots--the-durable-unit)
(жизненный цикл слота vs. чата), [Tech stack → better-sqlite3](POPBOT_DESIGN.md#tech-stack)
(персистентность стенограммы). Схема записи для каждого чата живёт в
`src/main/persistence/`.

## US-8 · Инспекция по тикету: чат + Unity + логи + доказательство

> *«Я должен иметь возможность легко посмотреть на прогресс тикета,
> увидев содержимое, работающий экземпляр сервера/Unity, релевантные логи,
> артефакт завершения (markdown)».*

Для любого чата (активного или приостановленного) один клик выводит всё,
что мне нужно, чтобы оценить прогресс:

- **Содержимое чата** — работающая стенограмма с рассуждением агента,
  вызовами инструментов и выводами.
- **Статус сервера / Unity** — поднят ли слот, какая ветка, каков стек
  экранов, в режиме ли Play Unity.
- **Релевантные логи** — консоль Unity + вспомогательный сервер,
  отфильтрованные по сессии чата, синхронно прокручиваемые.
- **Артефакт завершения** — `proof.md` (и вспомогательные `before/`,
  `after/`, `diff.patch`), который создал агент, отрендеренный встроенно.

Это вид «покажи мне, что произошло». Не сырой поток, а курируемый срез,
отвечающий на вопрос «сделано ли это хорошо?».

**Соответствует:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)
(колонка чата + нижняя панель логов), [Proof artifacts](POPBOT_DESIGN.md#proof-artifacts-agent-debug-deliverable).
Рендерер доказательства живёт в `src/renderer/chat/ProofViewer.tsx`
(планируется).

## US-9 · Разрешения по требованию, в нужный момент

> *«Я должен легко иметь возможность давать агентам разрешение делать
> различные вещи, которые им не следует позволять делать полностью
> автономно».*

Когда агент хочет сделать что-то из списка «всегда пауза» (`git push`, `gh
pr create`, `rm` за пределами слота, сетевые вызовы к неразрешённым хостам и
т. д.), PopBot приостанавливается и спрашивает меня. Поток предоставления
разрешения:

- Появляется модальное окно с тем, **что** агент хочет сделать, **почему**
  (заявленная агентом причина) и **командой / аргументами**.
- Я могу **разрешить один раз**, **разрешить для этого чата / сессии**,
  **всегда разрешать** (постоянное правило для конкретного инструмента,
  конкретной цели) или **запретить**.
- Правила разрешения накапливаются для каждого чата, отображаются в панели
  настроек чата, чтобы я мог их отозвать.
- Жёстко закодированный список запрета никогда не переопределяется из UI —
  см. [adr/0004](../adr/0004-canusetool-policy-boundary.md).

Суть: автономность — это значение по умолчанию, но я могу без трения
одобрить конкретное рискованное действие, не открывая терминал и не
присматривая за агентом.

**Соответствует:** [POPBOT_DESIGN.md → Autonomous mode](POPBOT_DESIGN.md#autonomous-mode),
[adr/0004 — canUseTool policy boundary](../adr/0004-canusetool-policy-boundary.md).
Хранилище разрешений живёт в `src/main/agents/policy/`.

## Отклонения и дополнения

Этот раздел отмечает места, где пользовательские истории расходятся с
зафиксированным дизайном. При реализации используйте пользовательские
истории как источник истины и обновляйте документ дизайна.

### Slack как третий источник внимания (US-1)

Оригинальный дизайн охватывает тикеты Linear и непроверенные PR. Сообщения
Slack не входили в область охвата. Чтобы соблюсти US-1:

- Добавить **панель Slack** в группу вкладок сверху слева рядом с Tickets и
  Reviews.
- Источник: личные сообщения Slack, @упоминания и сообщения в каналах,
  которыми я владею. Правила фильтрации TBD применительно к workflow
  порождения чата.
- Аутентификация: Slack OAuth (токен в связке ключей через `keytar`).
- Порождение чата из сообщения Slack засеивает агента контекстом беседы.

Это **совершенно новая подсистема** — клиент API Slack в `src/main/slack/`,
панель в `src/renderer/panels/slack/`. Поместите её в фазу 3 в
[PHASING.md](PHASING.md) рядом с другими панелями, но относитесь к ней как к
полноценному равноправному участнику, а не второстепенной идее.
