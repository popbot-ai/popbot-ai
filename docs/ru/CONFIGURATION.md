# Настройка PopBot

Всё в PopBot настраивается внутри приложения через **Настройки** (шестерёнка в заголовке окна, или `⌘,`) — конфигурационных файлов для ручного редактирования нет. Это руководство проходит по каждой панели в том порядке, в котором их перечисляет навигация, что примерно соответствует порядку их настройки в первый раз.

> Учётные данные, которые вы вводите (Linear, Jira, GitHub, Perforce и т. д.), хранятся **локально на вашей машине** в собственной базе данных приложения — никогда в этом репозитории.

## Integrations

Здесь живут две независимые группы: **источник тикетов**, питающий очередь Tickets, и **игровые движки**, которые может запускать слот.

![Integrations — Linear](../../images/preferences_integrations1.png)

### Источник тикетов

Единственный активный трекер задач питает очередь Tickets. Выберите его в селекторе в верхней части панели; форма конфигурации под ним поменяется соответственно. Активен только один трекер за раз.

- **Linear** — вставьте API-ключ (из *linear.app → Settings → API*). Опционально задайте **Team key** (например, `ENG`), чтобы ограничить ленту тикетов одной командой, и выберите **Project**, чтобы сузить её ещё больше. Сохранение проверяет ключ и показывает, от чьего имени произошло подключение.
- **Jira** — введите URL вашего сайта (`https://your-domain.atlassian.net`), email аккаунта и API-токен (из *id.atlassian.com → Security → API tokens*). Опционально ограничьте область до **Project** и добавьте фильтр **JQL** (например, `labels = backend`). Сохранение проверяет учётные данные перед их сохранением.
- **GitHub** — GitHub Issues здесь не требует учётных данных: провайдер обращается к CLI `gh`, который вы уже авторизовали для ревью и git-действий, а очередь охватывает те же репозитории, что настроены в разделе [Repositories](#repositories). Форма — это проверка статуса, подтверждающая, что `gh` установлен и авторизован, и сообщающая, сколько репозиториев она охватывает.

Каждый трекер с учётными данными проверяет их при **сохранении**, прежде чем сохранить, и показывает индикатор статуса *Connected / Not connected*.

### Игровые движки

В отличие от одиночного выбора источника тикетов, движки **независимы** — вы можете включить Unity, Unreal и пользовательский движок одновременно. Каждый включённый движок добавляет кнопку **Run** в панель чата, запускающую его редактор из рабочего пространства слота чата.

- **Enabled** — чекбокс для каждого движка, показывающий (или скрывающий) кнопку Run этого движка в панели чата.
- **Detected installs / Editor binary** *(Unity, Unreal)* — PopBot сканирует установленные редакторы (установки Unity Hub / Epic) со ссылкой **rescan**; выберите обнаруженную версию или введите абсолютный путь **Editor binary**, чтобы переопределить выпадающий список.
- **Run command** *(Custom)* — произвольная команда оболочки, запускаемая в каталоге проекта, с отдельными вариантами для **macOS / Linux** и **Windows**, так что одна конфигурация работает кросс-платформенно. У пользовательского движка нет автоопределения; PopBot передаёт идентичность слота в вашу команду через переменную окружения `POPBOT_SLOT`, так что вы можете настроить собственный поток «запустить и проверить».
- **Project subpath** — путь проекта движка относительно корня рабочего пространства (папка проекта Unity; папка, содержащая `.uproject`; или рабочий каталог, в котором выполняется пользовательская команда). Оставьте пустым, если корень рабочего пространства *и есть* проект.
- **Use MCP + Base MCP port** *(Unity, Unreal)* — когда чекбокс **Use MCP** включён, редактор запускается с указанием на MCP-сервер внутри редактора, чтобы агент мог им управлять. Каждый слот получает **собственный порт**, так что параллельные слоты никогда не конфликтуют: порт — это `basePort + (slotId − 1)` (слот 1 → база, слот 2 → база + 1, …). Поле **Base MCP port** задаёт порт слота 1; по умолчанию — **8000 для Unreal** и **8080 для Unity** (соответствует умолчанию MCP-плагина каждого движка) и восстанавливается к этому значению при очистке.
- **Show project path in title bar** *(Unity)* — кнопка **Install title-bar script**, которая помещает небольшой скрипт редактора в ваш проект Unity, чтобы каждый открытый редактор показывал полный путь проекта в заголовке окна, упрощая различение окон слотов. Скрипт безопасно коммитить.

> **Slack** и **Sentry** остаются заготовками подключения, а не подключёнными источниками инбокса, поэтому сегодня они не показаны здесь как панели. Их можно снова включить без структурных изменений; см. заметку в конце [Руководства по функциям и workflow](GUIDE.md).

## Agents

**Уровень рассуждений** модели по умолчанию для вновь создаваемых чатов (существующие чаты сохраняют свой собственный, пока вы не измените его в композере чата).

![Agents](../../images/preferences_agents.png)

- Задавайте уровень усилий независимо для **Claude** и **Codex**, и отдельно для:
  - **New chats** — обычные чаты и чаты по тикетам.
  - **Code reviews** — чаты ревью PR, резервные чаты повторного ревью и уведомления о ревью.

Больше усилий означает более глубокое рассуждение и более тщательное использование инструментов, за более высокую цену и задержку. Ревью часто требует иной глубины, чем сборка функциональности — отсюда и разделение.

## Runtime & slots

Эта панель управляет **удержанием вложений**. (Настройка размера пула слотов теперь выполняется для каждого репозитория отдельно и находится в разделе [Repositories](#repositories) — см. заметку там.)

![Runtime & slots](../../images/preferences_slots.png)

- **Keep attachments for** — как долго файлы и изображения, прикреплённые к чату, хранятся в собственном хранилище PopBot (по умолчанию 60 дней, диапазон 1–365). Вложения копируются в хранилище PopBot, чтобы они продолжали открываться из истории чата, даже если оригинал перемещён; очистка при запуске удаляет копии старше этого окна, чтобы папка не могла расти бесконечно.

> Скриншот выше может относиться к периоду до разделения настройки размера пула слотов на пореповый поток.

## Repositories

Каждый чат живёт в **репозитории**. Эта панель перечисляет ваши репозитории, и именно здесь настраиваются система контроля версий, слоты и рабочие пространства с copy-on-write для каждого репозитория.

![Repositories](../../images/preferences_repositories.png)

- **Add Repository** открывает мастер, ориентированный на папку: выберите папку, и PopBot **обнаружит её систему контроля версий** (Git или Perforce) и разветвится соответственно. Затем вы задаёте id, акцентный цвет, префикс слота и количество слотов.
  - Репозитории **Git** выбирают режим **slots** (переиспользуемый пул рабочих пространств — по умолчанию, отображается как `slots × N`) или **ephemeral** (свежее рабочее пространство на каждый чат). Режим slots держит кеши сборки тёплыми между чатами.
  - Репозитории **Perforce** всегда в режиме slot. Мастер фиксирует подключение P4, выполняет **проверку диска перед запуском** и строит замороженный **базовый образ** синхронизированного дерева; слоты затем создаются как дочерние copy-on-write этой базы (см. ниже).
- **Рабочие пространства с copy-on-write.** Рабочее пространство слота — это папка с copy-on-write, использующая один общий **базовый образ** репозитория и хранящая только те блоки, которые изменяет, через `shado` (слой теневых рабочих пространств PopBot): **дифференцирующий VHDX** на Windows, нативный copy-on-write (APFS / reflink) на macOS и Linux. Десять слотов на дереве терабайтного масштаба стоят примерно диска одного репозитория плюс небольшая дельта каждого слота — именно это позволяет большим деревьям Perforce вообще участвовать. Базовый образ строится один раз, как шаг мастера Add-Repository.
- **Режим постоянен.** Режим репозитория slots-vs-ephemeral фиксируется при создании; переключение осиротило бы рабочие пространства активных чатов.
- **Edit** репозиторий, чтобы изменить его акцентный цвет, базовую ветку по умолчанию (Git) или рабочий каталог агента Perforce, а также **изменить размер слотов** (увеличивать или уменьшать пул по одному рабочему пространству за раз, при условии, что все чаты в этом репозитории закрыты).
- **Delete** репозиторий; подтверждение предупредит вас, если чаты всё ещё ссылаются на него.

Несколько репозиториев работают бок о бок, у каждого свой пул слотов и акцентный цвет (цвет тонирует «таблетки» слотов этого репозитория, чтобы вы могли различать чаты с первого взгляда). Каждая карточка репозитория показывает своего провайдера системы контроля версий и режим.

## Source control

Глобальные настройки системы контроля версий и редактируемые шаблоны действий. Панели Git и Perforce показаны бок о бок, потому что провайдер репозитория определяется для каждой папки, и обе системы могут использоваться одновременно.

![Source control](../../images/preferences_source_control.png)

- **Change-view file limit** *(общий)* — максимальное число файлов, показываемых в виде изменений, прежде чем список будет ограничен. Применяется как к Git, так и к Perforce.

**Git**

- **Branch username** — префикс для новых веток: `<username>/<ticket>-<slug>`.
- **Action templates** — промпты, которые панель SCM отправляет агенту для **Commit**, **Push PR**, **Push draft PR**, **Make ready**, **Address CR** и **Rebase onto base**. Каждый поддерживает макросы `${name}` (`${branch}`, `${baseBranch}`, `${ticket}`, `${prnum}`, `${prurl}`…).

**Perforce**

- **Connection defaults** — путь к бинарнику `p4`, порт сервера по умолчанию и пользователь по умолчанию, которые предзаполняют шаг подключения Perforce мастера Add-Repository.
- **Transfer / submit options** — количество параллельных потоков синхронизации и следует ли откатывать неизменённые файлы при submit.
- **Swarm review poll interval** — как часто панель Reviews опрашивает Helix Swarm на предмет чейнджлистов, ожидающих вашего ревью. Это **независимо от опроса GitHub** и имеет **нижний порог 30 секунд**; увеличьте значение, чтобы снизить нагрузку на общий сервер Perforce/Swarm при масштабе.
- **Perforce action templates** — промпты, которые панель Perforce отправляет агенту для **CR** (открыть/обновить ревью Helix Swarm), **Run tests** и **Review & commit**, каждый с макросами `${name}`.

## External apps

Десктопные приложения, которые PopBot запускает из строки иконок чата, все указывают на рабочее пространство слота этого чата.

![External apps](../../images/preferences_external_apps.png)

- **Terminal** — какой терминал открывает лаунчер с иконкой терминала (например, iTerm2).
- **Terminal shell (Windows)** — оболочка, используемая панелью терминала внутри приложения: PowerShell, Command Prompt или PowerShell 7. Применяется к терминалам, открытым после изменения.
- **Code editor** — VS Code или Cursor; также используется для кликабельных ссылок `file.ts:42` в строках инструмента Edit.
- **Git client** — по умолчанию GitHub Desktop.
- **Chrome profile for URLs** — закрепите открытие ссылок за конкретным профилем Chrome (по имени его *каталога* профиля), чтобы они всегда попадали в ваш рабочий аккаунт.

> Бинарники движков и их опции MCP настраиваются в разделе [Integrations → Game engines](#integrations), не здесь.

## Prompt templates

Первое сообщение, которое PopBot отправляет при порождении чата. Каждый шаблон редактируем, со справочной карточкой доступных ему макросов `${name}`. (Шаблоны действий панели SCM находятся в разделе [Source control](#source-control).)

![Prompt templates](../../images/preferences_prompt_templates.png)

- **Start ticket** — срабатывает, когда вы порождаете чат из тикета, независимо от источника (Linear, Jira или GitHub Issues). Макросы включают `${ticketid}`, `${tickettitle}`, `${markdown}`, `${branch}` и `${slot}`.
- **Start code review** — срабатывает, когда вы порождаете чат из ревью — GitHub PR или чейнджлиста Helix Swarm. По умолчанию направляет агента использовать скилл ревью, читать окружающий код (не только дифф) и относиться к чату как только для чтения.
- **Re-review** — срабатывает, когда вы повторно проверяете существующий чат ревью; ограничивает агента только новыми коммитами.

Настройте их, чтобы закодировать соглашения, чек-листы и тон вашей команды.

## Code reviews

Управление инбоксом **Reviews**. Очередь показывает GitHub PR и чейнджлисты Helix Swarm, ожидающие вашего ревью; PR, которые вы уже проверили, автоматически удаляются.

![Code reviews](../../images/preferences_code_reviews.png)

- **Search cache window** — на сколько дней назад пикер **+ Add** нечётко сопоставляет недавние тикеты и PR (больше = больше возможностей поиска, немного медленнее обновление и больше бюджета API). Тикеты, назначенные на вас, всегда включены независимо от этого порога.
- **Ignore by title** — подстроки (по одной на строку, без учёта регистра), которые исключают PR из очереди.
- **Ignore by GitHub author** — логины ботов/авторов (по одному на строку, например `renovate[bot]`) для игнорирования.

> Частота **опроса** ревью настраивается для каждого провайдера отдельно, не здесь: интервал опроса Helix Swarm находится в разделе [Source control → Perforce](#source-control), независимо от опроса GitHub, так что общий сервер Perforce/Swarm можно защитить, не замедляя GitHub.

## Notifications

Как проявляются оповещения.

![Notifications](../../images/preferences_notifications.png)

- **VIP names** — люди, чьи сообщения всегда повышаются до срочного приоритета. Сопоставляются как подстроки отображаемого имени без учёта регистра, так что держите имена достаточно конкретными.
- **Toast placement** — *по центру сверху, улетает к колокольчику при закрытии* (по умолчанию), или классические тосты в верхнем правом углу. Переключатель применяется немедленно.
- **Test new-item flow** — временно помечает несколько реальных элементов очереди как NEW, чтобы предпросмотреть поведение чипа/индикатора (ничего не сохраняется). Это временное средство для разработки.

## Permissions

Глобальное значение по умолчанию для каждого инструмента агента и порог под автономным режимом.

![Permissions](../../images/preferences_permissions.png)

- Для каждого инструмента (**Bash**, **Read**, **Write**, **Edit**, **Grep**, **Glob**, **WebFetch**, **WebSearch**, …): **Ask** (спрашивать каждый раз — по умолчанию), **Allow** (автоматически одобрять) или **Deny** (автоматически отклонять).
- **Разрешения для каждого MCP-сервера.** MCP-серверу редактора слота (Unity, Unreal или любому MCP-серверу, который загружает агент) можно давать разрешение теми же тремя способами. Разрешение редакторского MCP слота однажды запоминается, и это разрешение видно и отзываемо здесь — отображается как `unityEditor → all tools` / `unrealEditor → all tools`, а не как сырое пространство имён. PopBot включает MCP-серверы редакторов Unity и Unreal именно так автоматически; правило для отдельного инструмента, отличающееся от wildcard, сохраняется как переопределение.
- Правила для конкретного чата (задаваемые в карточке разрешения через *Allow this chat* / *Deny this chat*) переопределяют эти глобальные настройки, так что отдельный чат может заблокировать инструмент, который вы иначе разрешили везде.

> Жёсткий порог запрета — `git push` / `p4 submit`, сеть к неразрешённым хостам, что-либо за пределами рабочего пространства — живёт в коде и **не** переопределяется здесь, так что неправильно настроенное правило не может позволить агенту самостоятельно влить изменения в мейнлайн.

## Language

Интерфейс PopBot полностью локализован.

- **Display language** — переключите локаль интерфейса из меню языка, которое перечисляет каждый язык на его собственном названии. Поставляемые локали — английский, испанский, французский, немецкий, китайский (упрощённый), японский, корейский и португальский (бразильский). Большая часть текста и меню обновляются сразу же; несколько системных строк завершают обновление после перезапуска. Новые окна и меню приложения также используют этот язык.
