# 사용자 스토리

PopBot의 "성공이란 어떤 모습인가"에 대한 참조 문서입니다. 2026-05-01에 작성되었습니다. 모든 구현 선택은 이 중 하나로 거슬러 올라갈 수 있어야 합니다.

사용자는 자신의 기기에서 PopBot을 운영하는 단일 개발자(Ben)입니다. 아래의 "나"는 그를 가리킵니다.

> **상태(발행 시점인 2026-07에 추가된 주석).** 아래의 스토리들은 2026-05에 작성된 *창립* 사용자 스토리이며, 설계 의도의 원본 기록으로 여기 보존되어 있습니다. PopBot은 그 이후 최초의 단일 사용자, Unity/Linear/Slack/GitHub 범위를 훨씬 넘어 일반화되었습니다 — 이제 Git과 Perforce, Unity와 Unreal, Linear/Jira/GitHub Issues, GitHub PR과 Helix Swarm을 아우르며, MIT 라이선스 아래 여러 언어로 현지화되어 출시됩니다. 이 문서는 의도적으로 이에 맞춰 개작되지 *않았습니다*. 역사적 자료로 취급하고, 현재 기능 목록은 [GUIDE.md](GUIDE.md)를 참고하세요. US-1..US-9 스토리와 2026-05 시점의 기록은 변경되지 않았습니다.

## US-1 · 관심이 필요한 큐에 대한 인지

> *"나는 처리해야 할 우선순위 높은 이슈, Slack 메시지, 그 외 PR을 인지하고 있어야 한다."*

세 개의 소스가 창 상단에 함께 나타납니다.

- 나에게 할당된, 우선순위 + 마감일로 순위가 매겨진 **Linear 티켓**.
- 나에게 온 **Slack 메시지**(DM, @멘션, 내가 소유한 채널). _새로운 요구 사항; 원래 설계에는 없었음 — [편차](#편차와-추가-사항)를 참고._
- 내 리뷰를 요청하는 **GitHub PR**.

각 행은 클릭 없이도 분류할 수 있을 만큼 한눈에 충분한 정보(제목, 소스, 경과 시간, 우선순위 표시기)를 보여줍니다. 우선순위가 높은 항목은 낮은 항목과 시각적으로 구분됩니다.

**대응 문서:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#앱-레이아웃)(티켓/리뷰 패널 — Slack 패널로 확장).

## US-2 · 원클릭 활성화

> *"나는 이것들 중 어느 것에 대해서든 쉽게 작업을 시작하고, 채팅을 열어 작업을 시작할 수 있어야 한다."*

관심 큐의 어떤 행을 클릭하든 그 작업을 위해 채워진 새 채팅이 생성됩니다.

- Linear 티켓 → 티켓 본문으로 채워지고, 티켓 키로 이름 붙여진 브랜치, 미리 채워진 에이전트 프롬프트를 가진 채팅.
- Slack 메시지 → 대화 맥락으로 채워져 답변을 작성하거나 실제 작업을 시작할 준비가 된 채팅.
- PR → diff와 리뷰 체크리스트로 채워진 채팅.

"내가 처리해야 할 무언가를 발견했다"와 "에이전트가 그것을 처리하고 있다" 사이에 설정 마찰이 없습니다.

**대응 문서:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#앱-레이아웃)("행을 클릭 → 그 작업을 위해 채워진 채팅 생성").

## US-3 · 채팅 안에서의 실제 게임 테스트

> *"채팅은 필요할 때 Unity 인스턴스를 실행하고 unity/서버를 실행하여 작업을 테스트하고 디버그할 수 있어야 한다."*

채팅이 실제 게임에서 동작을 검증해야 할 때, 채팅은 슬롯을 획득하고, Unity를 생성하며(2번 화면에 배치), 선택적으로 사이드카 서버를 생성합니다. 에이전트는 에디터 내 MCP를 통해 게임을 구동합니다 — Play 모드 진입, UI 클릭, 스크린샷 촬영, 로그 읽기, 상태 검증.

슬롯을 획득하는 것은 처음에는 느린 부분입니다(콜드 상태로 약 15~30초). 이후의 활동은 고정(sticky)됩니다(약 50ms).

**대응 문서:** [POPBOT_DESIGN.md → Chat types](POPBOT_DESIGN.md#채팅-유형)(Client Test / Server Test), [Slots](POPBOT_DESIGN.md#슬롯--지속되는-단위), [MCP automation surface](POPBOT_DESIGN.md#mcp-자동화-표면).

## US-4 · 증거와 함께하는 자율적인 엔드투엔드 완료

> *"에이전트는 완전히 자율적으로 작업할 수 있어야 하며, 검토 가능한 마크다운 문서에 수정/변경이 요구된 대로 작동했다는 증거를 전달하는 것을 포함하여, 티켓 전체를 수정/디버그하고 완료할 수 있어야 한다."*

자율 모드에서 에이전트는 개입 없이 완전한 읽기 → 재현 → 수정 → 검증 사이클을 실행하고, 끝에 `proof.md` 아티팩트를 작성합니다. 증거 문서는 다음을 담습니다.

- **재현** — 버그를 보여준 정확한 단계.
- **이전** — 고장 난 상태의 스크린샷 + 필터링된 로그 덤프.
- **근본 원인** — 에이전트의 진단.
- **수정** — 변경 사항의 diff나 요약.
- **이후** — 수정된 상태의 스크린샷 + 깨끗한 로그 덤프.
- **검증** — 이제 통과하는, 재현의 재실행.

나는 `proof.md`를 열어 스스로 아무것도 다시 실행하지 않고도 작업이 잘 되었는지 판단할 수 있습니다. 검토를 위한 일시 정지는 위험한 작업(`git push`, `gh pr create` 등)에서만 필요합니다.

**대응 문서:** [POPBOT_DESIGN.md → Autonomous mode](POPBOT_DESIGN.md#자율-모드), [Proof artifacts](POPBOT_DESIGN.md#증빙-아티팩트-에이전트-디버그-산출물).

## US-5 · 썸네일을 통한 쉬운 멀티태스킹

> *"나는 썸네일을 클릭함으로써 에이전트 사이를 쉽게 멀티태스킹할 수 있어야 한다."*

썸네일 스트립은 병렬 작업을 위한 주요 내비게이션 표면입니다. 채팅당 하나씩, 작고 압축된 미리보기의 행이 나를 즉시 에이전트 사이로 이동시켜줍니다. 썸네일을 클릭하면 그 채팅이 전면으로 나오고, 다른 채팅들은 백그라운드에서 계속 실행됩니다.

썸네일 자체는 정체성뿐 아니라 상태도 전달합니다. US-6을 참고하세요.

**대응 문서:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#앱-레이아웃)(썸네일 행), [PHASING.md](PHASING.md)의 Phase 3.

## US-6 · 한눈에 알 수 있는 상태

> *"나는 에이전트가 무엇을 하고 있는지, 그리고 나로부터 도움이나 지시가 필요한지를 한눈에 쉽게 알 수 있어야 한다."*

모든 채팅 썸네일은 내가 클릭해서 들어가지 않아도 현재 상태를 보여줍니다.

| 색 | 의미 |
|---|---|
| 파랑 | 실행 중 |
| 초록 | 작업 완료 |
| **노랑** | **일시 정지 — 나를 필요로 함** |
| 빨강 | 오류 발생 |
| 회색 | 유휴 / 시작되지 않음 |

노랑이 주의를 요구하는 색입니다. 썸네일 행을 훑어보는 것만으로 "누군가 막혀 있는가?"에 1초 안에 답할 수 있어야 합니다. 색을 넘어서, 썸네일은 짧은 진행 힌트(마지막 동작, 현재 단계)를 보여주어 내가 들어가볼지 결정할 수 있게 합니다.

**대응 문서:** [POPBOT_DESIGN.md → Status colors](POPBOT_DESIGN.md#상태-색상-채팅-썸네일).

## US-7 · 어디서든 복구하고 이어가기

> *"나는 티켓을, 더 이상 활성 상태가 아닌 것들까지도, 멈춘 곳에서 쉽게 복구하고 이어갈 수 있어야 한다."*

채팅은 지속됩니다. 그것을 닫거나, PopBot을 재시작하거나, 재부팅한 후에도, 나는 어떤 과거 채팅이든 다시 열어 정확히 멈춘 곳에서 이어갈 수 있습니다.

- 전체 트랜스크립트가 채팅 컬럼에 다시 재생됩니다.
- 슬롯이 내가 있던 것과 같은 브랜치에서 다시 획득됩니다(또는 콜드 상태로 새로 생성됩니다).
- Unity + 사이드카 상태는 설정되어 있었다면 관련 픽스처/세이브 블롭으로 복원됩니다.
- 에이전트는 내 다음 메시지에 응답하기 전에 최근 트랜스크립트를 다시 읽습니다 — 재시작을 거쳐도 맥락이 사라지지 않습니다.

채팅을 닫으면 그 슬롯이 해제됩니다. 다시 열면 다시 획득합니다. 채팅이 지속되는 기록이고, 슬롯은 일시적인 인프라입니다.

**대응 문서:** [POPBOT_DESIGN.md → Slots](POPBOT_DESIGN.md#슬롯--지속되는-단위)(슬롯 대 채팅 생명주기), [Tech stack → better-sqlite3](POPBOT_DESIGN.md#기술-스택)(트랜스크립트 영속성). 채팅별 레코드 스키마는 `src/main/persistence/`에 있습니다.

## US-8 · 티켓별 검사: 채팅 + Unity + 로그 + 증거

> *"나는 콘텐츠, 실행 중인 서버/Unity 인스턴스, 관련 로그, 완료 아티팩트(마크다운)를 보여줌으로써 티켓의 진행 상황을 쉽게 살펴볼 수 있어야 한다."*

어떤 채팅이든(활성 상태든 일시 정지 상태든) 클릭 한 번으로 진행 상황을 평가하는 데 필요한 모든 것이 나타납니다.

- **채팅 내용** — 에이전트의 추론, 도구 호출, 출력을 담은 실행 중인 트랜스크립트.
- **서버/Unity 상태** — 슬롯이 떠 있는가, 어떤 브랜치인가, 화면 스택은 어떤가, Unity가 Play 모드인가.
- **관련 로그** — 채팅의 세션으로 필터링되어 동기화 스크롤되는 Unity 콘솔 + 사이드카 서버.
- **완료 아티팩트** — 에이전트가 만든 `proof.md`(그리고 함께 제공되는 `before/`, `after/`, `diff.patch`)가 인라인으로 렌더링됩니다.

이것은 "무슨 일이 있었는지 보여줘" 뷰입니다. 원시 데이터의 홍수가 아니라, "이것이 잘 되었는가?"에 답하는 엄선된 단면입니다.

**대응 문서:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#앱-레이아웃)(채팅 컬럼 + 하단 로그 패널), [Proof artifacts](POPBOT_DESIGN.md#증빙-아티팩트-에이전트-디버그-산출물). 증거 렌더러는 `src/renderer/chat/ProofViewer.tsx`에 있습니다(예정).

## US-9 · 적시(Just-in-time) 권한 승인

> *"나는 에이전트에게 완전히 자율적으로 허용되어서는 안 될 다양한 작업에 대한 권한을 쉽게 줄 수 있어야 한다."*

에이전트가 항상 일시 정지 목록에 있는 무언가(`git push`, `gh pr create`, 슬롯 바깥의 `rm`, 허용 목록에 없는 호스트로의 네트워크 호출 등)를 하고 싶을 때, PopBot은 일시 정지하고 나에게 묻습니다. 승인 플로우는 다음과 같습니다.

- 모달이 나타나 에이전트가 **무엇을** 하고 싶은지, **왜**(에이전트가 밝힌 이유)인지, 그리고 **명령/인자**를 보여줍니다.
- 나는 **한 번 허용**, **이 채팅/세션에 대해 허용**, **항상 허용**(도구별, 대상별 지속되는 규칙), 또는 **거부**를 선택할 수 있습니다.
- 허용 규칙은 채팅별로 누적되며, 채팅 설정 패널에 표시되어 내가 철회할 수 있습니다.
- 하드코딩된 거부 목록은 UI에서 결코 재정의할 수 없습니다 — [adr/0004](../adr/0004-canusetool-policy-boundary.md)를 참고하세요.

핵심은 이것입니다. 자율성이 기본값이지만, 나는 터미널을 열거나 에이전트를 일일이 지켜보지 않고도 특정 위험 작업을 마찰 없이 승인할 수 있습니다.

**대응 문서:** [POPBOT_DESIGN.md → Autonomous mode](POPBOT_DESIGN.md#자율-모드), [adr/0004 — canUseTool policy boundary](../adr/0004-canusetool-policy-boundary.md). 승인 저장소는 `src/main/agents/policy/`에 있습니다.

## 편차와 추가 사항

이 섹션은 사용자 스토리가 확정된 설계와 갈라지는 지점을 표시합니다. 구현 시, 사용자 스토리를 진실 공급원으로 사용하고 설계 문서를 업데이트하세요.

### 세 번째 관심 소스로서의 Slack (US-1)

원래 설계는 Linear 티켓과 리뷰되지 않은 PR을 다룹니다. Slack 메시지는 범위에 없었습니다. US-1을 충족시키기 위해:

- 티켓, 리뷰와 나란히 좌상단 탭 그룹에 **Slack 패널**을 추가합니다.
- 소스: 내가 소유한 채널의 Slack DM, @멘션, 메시지. 채팅 생성 워크플로별 필터링 규칙은 추후 결정.
- 인증: Slack OAuth(`keytar`를 통해 키체인에 토큰 저장).
- Slack 메시지로부터 채팅을 생성하면 대화 맥락으로 에이전트가 채워집니다.

이는 **완전히 새로운 서브시스템**입니다 — `src/main/slack/`의 Slack API 클라이언트, `src/renderer/panels/slack/`의 패널. [PHASING.md](PHASING.md) Phase 3에 다른 패널들과 함께 단계적으로 도입하되, 부차적인 것이 아니라 1급 대등한 존재로 취급하세요.
