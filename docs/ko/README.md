## 왜 PopBot인가

AI 코딩 에이전트 하나를 실행하는 것은 간단합니다. 하지만 여러 개를 동시에 실행하면 단일 에이전트에는 없던 문제들이 생깁니다. 서로의 작업을 덮어쓰지 않도록 격리를 유지하는 것, 실제로 빌드된 결과물을 테스트하는 것, 이를 리뷰하는 것, 그리고 되돌릴 수 없는 작업은 어떤 에이전트도 감독 없이 수행하지 못하도록 게이트를 거는 것입니다.

PopBot은 이를 위한 오케스트레이션 레이어입니다. 티켓과 리뷰 요청을 원클릭 에이전트 세션으로 전환하고, 각 에이전트에 격리된 워크스페이스(자신만의 작업 사본 — 게임 프로젝트의 경우 테스트 대상 앱의 실행 중인 사본까지)를 제공하며, 기본적으로 자율적으로 실행하되 위험한 작업에는 사람의 게이트를 두고, 모든 트랜스크립트·diff·터미널·로그를 하나의 창에 모읍니다. 운영자는 컬럼을 훑어보고, 게이트가 걸린 작업을 승인하고, 배포합니다.

이 도구는 **Proof of Play**의 소규모 팀이 만들어, 실제로 출시된 방대한 에셋 기반의 프로덕션 프로젝트에서 매일 사용했습니다. 그것이 바로 이 도구가 검증된 환경입니다. 수 기가바이트에 달하는 에셋, 실제 소스 컨트롤, 실제 마감일. 웜(warm)하고 격리된 카피-온-라이트(copy-on-write) 워크스페이스라는 슬롯 모델이야말로 그 환경에서 에이전트를 병렬로 실행하는 것을 실용적으로 만든 요소였고, 팀이 한 번에 처리할 수 있는 작업량을 늘려주었습니다. 우리는 PopBot을 참조 구현(reference implementation)으로 공개하고 지원합니다. 있는 그대로 소비하는 완제품이 아니라, 여러분 자신의 스택과 워크플로에 맞게 가져다가 다시 빚어내는 하나의 형태로서 말입니다. 이는 AI 시대에 소프트웨어를 가장 잘 만드는 방법에 대한 하나의 관점을 반영합니다 — 에이전트 무리를 운영하는 팀은 고정된 도구를 채택하기보다 도구를 직접 소유하고 수정하는 편이 더 낫다는 관점입니다. MIT 라이선스이며 포크하기 쉽도록 구성되어 있습니다. [나만의 것으로 만들기](#나만의-것으로-만들기)를 참고하세요.

## 핵심 기능

### 실시간 썸네일이 있는 멀티챗 뷰

열려 있는 모든 채팅이 화면에 유지됩니다 — 나란히 배치된 **컬럼** 위에 **실시간 썸네일** 스트립이 자리합니다. 각 썸네일은 단순한 상태 점이 아니라 해당 채팅의 실제로 업데이트되는 뷰이며, 상태에 따라 색으로 구분됩니다: 실행 중, 완료, 사용자 대기 중, 오류. 한눈에 *모든 에이전트가 무엇을 하고 있는지*와 누가 여러분을 필요로 하는지 알 수 있고, **잘못된 방향을 조기에 포착**하여 시간과 토큰을 낭비하기 전에 방향을 바꿀 수 있습니다. 한 사람이 하나의 창에서 전체 에이전트 무리를 감독합니다.

### 웜 슬롯 — 재임포트 비용 없는 병렬 에이전트

작업 중인 각 채팅은 **슬롯**을 대여합니다 — 영구적인 작업 사본에 더해 한 번 생성되어 재사용되는 자신만의 웜 빌드 상태입니다. 게임 엔진의 경우 이는 슬롯이 자신만의 핫 에셋 캐시(Unity의 `Library`, Unreal의 DDC)를 유지하고 에디터를 계속 실행 상태로 둘 수 있음을 의미하며, 그 덕분에 에이전트를 슬롯으로 다시 전환하는 데 몇 분에 걸친 재임포트가 아니라 **몇 초**밖에 걸리지 않습니다. 열 개의 에이전트가 단 하나의 임포트 캐시를 두고 다투는 일 없이 진정한 브랜치 격리 속에서 실행됩니다. [슬롯의 작동 방식 →](GUIDE.md#슬롯-웜하고-격리되고-폐기-가능한-워크스페이스)

### 하나의 리포지토리 디스크로 무제한 복사

슬롯의 워크스페이스는 **카피-온-라이트(copy-on-write) 폴더**입니다. 모든 슬롯은 하나의 베이스 이미지를 공유하고, 자신이 변경한 부분만 저장합니다. 그래서 **테라바이트급** 게임 트리의 새롭고 살아있는 완전한 복사본이 **몇 초 만에** 준비됩니다 — 얕은 뷰가 아니라 실제로 편집 가능한 파일이며, 무제한 복사본을 만들어도 디스크 사용량은 리포지토리 하나 분량에 그칩니다. **Windows, macOS, Linux**에서 모두 동작하며, 거대한 Perforce 트리가 애초에 이 체계에 합류할 수 있게 해주는 요소이기도 합니다. [왜 이것이 중요한가 →](GUIDE.md#카피-온-라이트-하나의-리포지토리-디스크로-무제한-복사)

### 리뷰 기능이 내장된 Git과 Perforce

소스 컨트롤은 하나의 인터페이스 뒤에 있는 **프로바이더**입니다. **Git**(워크트리, 브랜치, `gh`를 통한 PR)과 **Perforce**(섀도 워크스페이스 위의 스트림, 체인지리스트, **Helix Swarm** 리뷰) 모두 1급 지원됩니다. *각 채팅 고유의 워크스페이스*에 범위가 한정된 소스 컨트롤 패널이 정확히 그 브랜치의 상태, 커밋, 파일별 diff를 보여줍니다. 원클릭 템플릿 작업(**커밋**, **PR 푸시**, **리뷰 준비 완료**, **CR 반영**, **베이스로 리베이스**)은 `${branch}` / `${ticket}` / `${prnum}`이 채워진 사전 작성 지시문을 해당 채팅의 에이전트에게 전송합니다.

### 하나의 인박스, 여러 소스

전체 루프가 한곳에 모여 있습니다 — **인박스**(**Linear**, **Jira**, **GitHub Issues**에서 할당된 티켓, 그리고 **GitHub PR**과 **Swarm 체인지리스트**로서 여러분을 기다리는 리뷰) → 격리된 슬롯에서 **진행 중**인 에이전트 작업 → **푸시**하고 PR/리뷰 열기 → 완료된 채팅을 **아카이브** → 나중에 전체 히스토리와 함께 **재개**하기. 티켓을 클릭하면 PopBot이 브랜치 이름을 정하고, 슬롯을 대여하고, 티켓을 *진행 중*으로 옮기고, 에이전트에 작업 내용을 심어줍니다 — 그리고 이를 병합된 변경 사항까지, 그리고 다시 그 이후까지 이끌어갑니다. [워크플로 둘러보기 →](GUIDE.md#엔드투엔드-워크플로)

## 추가 기능

- **재구현이 아닌, 진짜 Claude Code와 Codex.** 각 채팅은 공식 SDK를 통해 *실제* 에이전트를 구동합니다 — 터미널에서 실행하는 것과 동일한 `claude`와 `codex` CLI이며, 모든 도구, 스킬, MCP 서버가 그대로 유지됩니다. 채팅마다 모델(Opus / Fable / GPT)과 추론 강도를 선택하고, 세션 도중 전환하거나, 채팅의 히스토리를 바탕으로 새 세션을 다시 시작할 수 있습니다.
- **자신의 작업을 직접 테스트하는 에이전트.** 슬롯은 실제 앱을 실행할 수 있습니다 — Unity와 Unreal의 경우, 두 번째 디스플레이에서 실행되는 라이브 에디터 + 사이드카 서버를, 슬롯별 포트를 통한 에디터 내 MCP 서버로 에이전트가 구동합니다 — 그래서 에이전트는 추측하는 대신 UI를 클릭하고, 로그를 읽고, 변경 사항을 검증합니다. 커스텀 엔진도 지원됩니다.
- **영속적이고 아카이브 가능한 채팅.** 모든 채팅은 지속되는 트랜스크립트입니다. 닫으면 슬롯이 해제되고, 나중에 전체 히스토리가 그대로 보존된 채로 다시 열 수 있습니다.
- **채팅별 터미널과 클릭 가능한 코드.** 채팅의 워크스페이스에 고정된 내장 터미널, 그리고 VS Code나 Cursor에서 열리는 `file.ts:42` 링크.
- **자율적이되 결코 무모하지 않음.** 에이전트는 자신의 슬롯 안에서 안전한 작업을 자동으로 실행하고, 위험한 작업 — `git push` / `p4 submit`, PR 열기, 워크스페이스 바깥의 모든 것, 네트워크 호출 — 에서는 여러분을 위해 일시 정지합니다. 권한 부여는 채팅별로 이루어지며, 지속되고, 철회 가능합니다 — MCP 서버도 포함됩니다.
- **완전히 현지화됨.** 전체 인터페이스가 8개 언어(영어, 스페인어, 프랑스어, 독일어, 일본어, 한국어, 중국어 간체, 브라질 포르투갈어)로 제공되며, 언어 메뉴에서 언제든 전환할 수 있습니다.
- **멀티 리포지토리.** 여러 리포지토리를 나란히 운영할 수 있으며, 각각 고유한 슬롯 풀, 색상, 프로바이더, 브랜치 규칙을 가집니다.

## PopBot이 다른 점

에이전틱 코딩 도구는 대체로 몇 가지 부류로 나뉩니다. PopBot은 다른 자리에 있습니다 — **웜 빌드 상태와 실시간 사람의 감독을 갖춘, 다수의 *실제* 에이전트를 병렬로 운영하기 위한 로컬 콕핏**입니다.

| 이것 대신… | …PopBot은 |
|---|---|
| **터미널이나 IDE의 단일 에이전트** — 한 번에 하나의 작업 트리에서 하나의 작업 | **한 번에 여러 에이전트**, 각각 자신만의 웜 슬롯에 격리되어, 하나의 창에서 조종하는 실시간 무리로 모두 보임 |
| **비동기 클라우드 에이전트** — 불투명하고 원격이며, 작업을 제출하고 PR을 기다림 | **로컬이고 실시간** — 각 에이전트가 작업하는 모습을 지켜보며 잘못된 방향을 조기에 포착하고, 진정한 엔드투엔드 테스트를 위해 *여러분의 실제 앱*(두 번째 화면의 엔진 에디터)을 구동함 |
| **직접 만든 `tmux` + 워크트리 저글링** — 병렬적이지만 수동적이며, 매번 새로 체크아웃할 때마다 엔진의 몇 분짜리 재임포트 비용을 치름 | **관리되는 웜 슬롯** — 재사용되는 카피-온-라이트 워크스페이스가 에셋 캐시를 뜨겁게 유지하며, 브랜치/워크스페이스 라이프사이클, SCM 패널, 코드 리뷰까지 대신 처리함 |
| **에이전트 오케스트레이션 프레임워크** — 에이전트 시스템을 *구축*하기 위한 툴킷 | **여러분의 인박스와 리뷰 루프에 연결된, 완성된 자기 주장이 뚜렷한 앱** — 조립해야 할 라이브러리가 아니라 설계 단계부터 인간 개입을 전제로 함 |

그리고 결정적으로, PopBot은 Claude Code나 Codex를 대체하지 않습니다. 그것들을 **실행**합니다. 여러분은 이미 신뢰하는 바로 그 에이전트(그리고 정확히 그 버전의 CLI)를, 한 번에 여러 개, 오케스트레이션과 격리와 감독으로 감싸서 얻게 됩니다.

## 워크스페이스 해부

![PopBot UI anatomy](../../images/anatomy.png)

| 영역 | 내용 |
|---|---|
| **인박스 — 티켓과 리뷰** | 순위가 매겨진, 할당된 티켓(Linear / Jira / GitHub Issues)과 여러분을 기다리는 리뷰(GitHub PR / Swarm 체인지리스트). 클릭 한 번으로 채팅이 생성됩니다. |
| **슬롯** | 웜하고 격리된 워크스페이스의 풀 — 카피-온-라이트 작업 사본 *더하기* 영속적인 빌드 상태(게임 엔진의 경우 자신만의 핫 에셋 캐시). 채팅이 작업하는 동안 하나를 대여하고, 닫으면 반환합니다. |
| **채팅 아카이브** | 과거의 모든 채팅, 검색 가능하며 전체 히스토리와 함께 재개 가능. |
| **채팅 썸네일** | 열려 있는 모든 채팅의 실시간 스트립 — 상태(실행 중 / 완료 / 사용자 필요 / 오류)에 따라 색으로 구분. |
| **채팅** | 포커스된 에이전트 세션: 실시간으로 스트리밍되는 텍스트, 도구 호출, 인라인 코드 diff. |
| **채팅별 터미널** | 수동 명령을 위해 해당 채팅의 워크스페이스를 가리키는 내장 터미널. |
| **SCM 패널** | 작업 트리/체인지리스트 상태, 커밋, 파일 diff, 원클릭 커밋/푸시/PR/리뷰 작업. |

## PopBot과 함께하는 하루

**기능 티켓.** 인박스에 티켓이 도착합니다. 클릭하면 → PopBot이 `you/eng-123-…`에서 채팅을 열고, 슬롯을 대여하고, 티켓을 *진행 중*으로 옮기고, 에이전트에게 전체 설명을 건네줍니다. 에이전트는 코드를 작성하고, 슬롯에서 앱을 실행해 검증한 뒤, 푸시하기 전 여러분의 승인을 위해 일시 정지합니다. SCM 패널에서 diff를 리뷰하고 **Push PR**을 누릅니다.

**병렬로 진행되는 버그.** 그것이 진행되는 동안 버그 리포트가 들어옵니다. 두 번째 채팅을 생성합니다 — 자신만의 슬롯, 자신만의 브랜치 — 그리고 두 에이전트는 서로의 트리를 건드리지 않고 동시에 작업합니다. 썸네일 스트립에는 둘 다 표시됩니다: 하나는 초록(완료), 하나는 파랑(실행 중).

**리뷰 요청.** 동료의 PR(또는 Swarm 체인지리스트)이 여러분의 리뷰 탭에 나타납니다. 클릭하면 → **리포지토리 없는(repoless)** 즉석 리뷰 채팅이 열리고, 에이전트는 diff *와* 주변 코드를 읽고, 실제 버그를 찾아내고, GitHub나 Swarm에 인라인 리뷰를 게시합니다 — 그동안 여러분의 두 빌드 채팅은 계속 진행됩니다.

**다음 날 다시 시작하기.** 완료된 채팅을 닫아 슬롯을 해제합니다. 다음 날 아침, 아카이브에서 기능 채팅을 다시 열어 리뷰 피드백을 반영합니다 — 에이전트는 전체 대화와 워크스페이스가 그대로 유지된 채로 재개됩니다.

→ 전체 둘러보기(기능, 버그, 리뷰 플로우, 그리고 슬롯, 카피-온-라이트 워크스페이스, 재개하기가 내부적으로 어떻게 작동하는지)는 **[기능 및 워크플로 가이드](GUIDE.md)**에 있습니다.

## 설치

서명된 사전 빌드 설치 파일은 **[popbot.app](https://popbot.app)**에서 받을 수 있습니다.

- **macOS** — 서명 및 공증된 `.dmg` (Apple 실리콘)
- **Windows** — 서명된 `.exe` 설치 파일
- **Linux** — `.deb` 패키지

앱은 릴리스 채널로부터 자동 업데이트됩니다. 직접 빌드하려면 [소스에서 빌드하기](#소스에서-빌드하기)를 참고하세요.

## 소스에서 빌드하기

```bash
npm install
npm run dev        # 개발 모드로 앱 실행
npm run package    # 여러분의 플랫폼용 서명된 설치 파일 빌드
```

**요구 사항**

- **macOS, Windows, 또는 Linux.** macOS가 가장 많이 검증된 플랫폼입니다(두 번째 디스플레이에서 테스트 대상 앱을 구동하는 워크플로가 macOS 접근성 API에 의존합니다). Windows와 Linux도 지원되며 출시되어 있습니다 — Windows/WSL 설정 노트는 [WINDOWS.md](WINDOWS.md)를 참고하세요.
- **Node 20+** (Node 20 / 22는 네이티브 모듈 재컴파일을 피할 수 있습니다. Windows 노트를 참고하세요).
- **`claude`** 및/또는 **`codex`** CLI(에이전트 백엔드), 그리고 **`git`**, GitHub 플로우를 위한 **`gh`**. Perforce의 경우 **`p4`** CLI.
- 자격 증명(Linear, Jira, GitHub, Helix Swarm)은 앱 자체 데이터베이스에 **여러분의 컴퓨터에 로컬로** 저장되며 — 이 리포지토리에는 절대 저장되지 않습니다.
- 선택 사항: 게임 프로젝트를 위한 Unity 또는 Unreal 에디터; VS Code / Cursor; iTerm.

## 나만의 것으로 만들기

PopBot은 있는 그대로 채택하기보다 포크하고 개조하도록 의도된 참조 구현으로 공개되어 있습니다. 그 형태는 일반적입니다 — **에이전트 + 격리되고 웜하고 카피-온-라이트인 슬롯 + 큐로서의 인박스 + 테스트 대상 앱** — 그리고 코드는 *작은 공통 인터페이스 뒤에 있는 프로바이더*로 구성되어 있어, 팀은 나머지를 건드리지 않고 한 부분만 교체할 수 있습니다. **MIT 라이선스**입니다. 일반적인 접근 방식은 핵심 아이디어는 유지하고 구체적인 구현만 교체하는 것입니다.

- **테스트 대상 앱을 교체하세요.** Unity와 Unreal은 "에이전트가 여러분의 앱을 실행하고 검증하도록 하는" 두 가지 구현입니다. 커스텀 엔진 훅은 이미 슬롯 아이덴티티를 여러분의 실행 명령까지 전달하므로 — 여러분의 웹 앱, CLI, 테스트 하네스를 가리키기만 하면 됩니다. *(`src/shared/gameEngine.ts`, `src/main/ipc/apps.ts`)*
- **인박스를 다른 곳으로 향하게 하세요.** Linear, Jira, GitHub Issues는 예시일 뿐입니다. 인터페이스 하나를 구현하고 등록하여 트래커를 추가하세요. *(`src/main/tickets/`)*
- **소스 컨트롤을 추가하거나 교체하세요.** Git과 Perforce처럼 프로바이더 베이스 클래스를 확장하세요. 호출부는 프로바이더 id가 아니라 *기능(capability)*에 따라 분기합니다. *(`src/main/scm/`)*
- **액션과 프롬프트를 재배선하세요.** 브랜치 규칙, PR/리뷰 플로우, 그리고 심어지는 모든 프롬프트는 환경설정에서 코드 작성 없이 편집 가능한 템플릿입니다.
- **핵심은 유지하세요.** 웜 슬롯, 카피-온-라이트 워크스페이스, 영속적인 채팅, 하드코딩된 권한 하한선, 병렬 에이전트 콕핏이 이 도구의 변치 않는 척추입니다.

**[기능 및 워크플로 가이드](GUIDE.md)**는 각 이음매 뒤의 논리를 설명하고, **[아키텍처](ARCHITECTURE.md)** 문서는 코드에서 그것을 어디서 찾을 수 있는지 지도로 보여줍니다.
