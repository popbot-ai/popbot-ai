# User Stories

PopBot にとっての「成功とは何か」を示すリファレンスです。2026-05-01 に記録されました。すべての実装判断は、これらのいずれかに遡れるはずです。

ユーザーは自分のマシンで PopBot を走らせている一人の開発者（Ben）です。以下の「私」は彼を指します。

> **ステータス（公開時、2026-07 に追記）。** 以下のストーリーは 2026-05 に記録された*創設期*のユーザーストーリーであり、設計意図の当初の記録としてここに保存されています。PopBot はその後、この最初の単一ユーザー、Unity/Linear/Slack/GitHub というスコープを大きく超えて一般化されました — 現在では Git と Perforce、Unity と Unreal、Linear/Jira/GitHub Issues、GitHub PR と Helix Swarm にまたがり、MIT ライセンスの下で複数言語にローカライズされて出荷されています。この文書は意図的にそれへ合わせて改変されて**いません** — 歴史として扱ってください。現在の機能セットについては [GUIDE.md](GUIDE.md) を参照してください。US-1..US-9 のストーリーと 2026-05 時点の記録は変更されていません。

## US-1 · アテンションキューの把握

> *「優先度の高いイシュー、Slack メッセージ、対応が必要な他の PR に気づけるようにしたい。」*

3 つのソースがウィンドウ上部にまとめて表示されます。

- **Linear チケット** — 自分に割り当てられたもの、優先度 + 期日でランク付け。
- **Slack メッセージ** — 自分宛のもの（DM、@mention、自分がオーナーのチャンネル）。_新規要件。オリジナルの設計にはなし — [Deviations](#deviations-and-additions) を参照。_
- **GitHub PR** — 自分にレビューを依頼しているもの。

各行はクリックせずにトリアージできるだけの情報を一目で示します（タイトル、ソース、経過時間、優先度インジケーター）。優先度の高い項目は低いものと視覚的に区別されます。

**対応先:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)（Tickets / Reviews パネル — Slack パネルで拡張）。

## US-2 · ワンクリックでの着手

> *「これらのどれについても簡単に活動を開始でき、作業を始めるためのチャットを開けるようにしたい。」*

アテンションキュー内の任意の行をクリックすると、その作業のために種付けされた新しいチャットが生成されます。

- Linear チケット → チケット本文で種付けされたチャット、チケットキーにちなんで命名されたブランチ、あらかじめ入力されたエージェントプロンプト。
- Slack メッセージ → 会話のコンテキストで種付けされたチャット。返信を起草するか、実際の作業を開始する準備が整っている。
- PR → diff とレビューチェックリストで種付けされたチャット。

「対応すべきものを見つけた」と「エージェントがそれに取り組んでいる」の間に、セットアップの摩擦は一切ありません。

**対応先:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)（「行をクリック → その作業のために種付けされたチャットを生成」）。

## US-3 · チャット内での実際のゲームテスト

> *「チャットは必要に応じて Unity インスタンスを起動し unity/server を実行できるべきで、テストとデバッグができるようにしたい。」*

チャットが実際のゲーム内で動作を検証する必要があるとき、チャットはスロットを取得し、Unity を起動し（画面 2 に配置）、必要に応じてサイドカーサーバーも起動します。エージェントはエディタ内 MCP 経由でゲームを操作します — Play モードに入る、UI をクリックする、スクリーンショットを撮る、ログを読む、状態をアサートする、といった具合です。

スロットの取得は初回だけ遅く（コールドで約 15〜30 秒）、以後の活動は粘着的です（約 50 ms）。

**対応先:** [POPBOT_DESIGN.md → Chat types](POPBOT_DESIGN.md#chat-types)（Client Test / Server Test）、[Slots](POPBOT_DESIGN.md#slots--the-durable-unit)、[MCP automation surface](POPBOT_DESIGN.md#mcp-automation-surface)。

## US-4 · 証跡を伴う自律的なエンドツーエンド完了

> *「エージェントは完全に自律的に作業でき、チケット全体を修正/デバッグして完了させ、修正/変更が要求どおりに機能したことを示す証跡を、検査可能な markdown ドキュメントとして届けられるべきだ。」*

自律モードでは、エージェントは介入なしに 読み取り → 再現 → 修正 → 検証 の完全なサイクルを実行し、最後に `proof.md` アーティファクトを書き出します。証跡には以下が含まれます。

- **Repro** — バグを実証した正確な手順。
- **Before** — 壊れた状態からのスクリーンショット + フィルタ済みログダンプ。
- **Root cause** — エージェントの診断結果。
- **Fix** — 変更内容の diff または要約。
- **After** — 修正後の状態からのスクリーンショット + クリーンなログダンプ。
- **Verification** — repro の再実行結果、今度は成功している。

私は `proof.md` を開くだけで、自分で何かを再実行することなく、その作業が良いものかどうかを判断できます。レビューのための一時停止が必要になるのは、リスクのある操作（`git push`、`gh pr create` など）のときだけです。

**対応先:** [POPBOT_DESIGN.md → Autonomous mode](POPBOT_DESIGN.md#autonomous-mode)、[Proof artifacts](POPBOT_DESIGN.md#proof-artifacts-agent-debug-deliverable)。

## US-5 · サムネイルによる容易なマルチタスク

> *「サムネイルをクリックすることで、エージェント間を簡単にマルチタスクできるようにしたい。」*

サムネイルストリップは並列作業のための主要なナビゲーション面です。コンパクトなプレビューの列 — チャット 1 つにつき 1 つ — により、エージェント間を即座に行き来できます。サムネイルをクリックするとそのチャットが前面に来ますが、他のチャットはバックグラウンドで動き続けます。

サムネイル自体は、単なる識別ではなく状態も伝えます。US-6 を参照してください。

**対応先:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)（サムネイル行）、[PHASING.md](PHASING.md) の Phase 3。

## US-6 · 一目で分かるステータス

> *「エージェントが何をしているか、私からの支援や指示を必要としているかどうかを、一目で把握できるようにしたい。」*

すべてのチャットサムネイルは、クリックして中に入らなくても現在の状態を示します。

| 色 | 意味 |
|---|---|
| 青 | 実行中 |
| 緑 | タスク完了 |
| **黄** | **一時停止 — 私が必要** |
| 赤 | エラー |
| 灰 | アイドル / 未開始 |

黄色は注意を要求する色です。サムネイル行をざっと見るだけで、「誰か詰まっていないか」に 1 秒未満で答えられるべきです。色に加えて、サムネイルは短い進捗ヒント（最後のアクション、現在のステップ）も表示するので、深く見るべきかどうかを判断できます。

**対応先:** [POPBOT_DESIGN.md → Status colors](POPBOT_DESIGN.md#status-colors-chat-thumbnail)。

## US-7 · どこからでも復帰して継続

> *「もはやアクティブでないものも含め、チケットを中断した場所から簡単に復帰して継続できるようにしたい。」*

チャットは永続的です。チャットを閉じても、PopBot を再起動しても、マシンを再起動しても、過去の任意のチャットを再オープンして、中断した場所からまさに続きを始められます。

- 完全なトランスクリプトがチャットカラムに再生されます。
- スロットが再取得されます（または必要ならコールドで立ち上げ直されます）、同じブランチ上で。
- Unity + サイドカーの状態が、設定されていれば関連するフィクスチャ/セーブ blob に復元されます。
- エージェントは私の次のメッセージに応答する前に、最近のトランスクリプトを読み直します — 再起動をまたいでコンテキストが失われることはありません。

チャットを閉じるとそのスロットは解放され、再オープンすると再取得されます。チャットこそが永続的な記録であり、スロットは一時的なインフラです。

**対応先:** [POPBOT_DESIGN.md → Slots](POPBOT_DESIGN.md#slots--the-durable-unit)（スロットとチャットのライフサイクルの違い）、[Tech stack → better-sqlite3](POPBOT_DESIGN.md#tech-stack)（トランスクリプトの永続化）。チャットごとのレコードスキーマは `src/main/persistence/` にあります。

## US-8 · チケットごとの検査: チャット + Unity + ログ + 証跡

> *「チケットの進捗を、内容、実行中のサーバー/Unity インスタンス、関連するログ、完了アーティファクト（markdown）を表示することで簡単に確認できるようにしたい。」*

任意のチャット（アクティブでも一時停止中でも）について、ワンクリックで評価に必要なすべてが表示されます。

- **チャット内容** — エージェントの推論、ツール呼び出し、出力を含む実行中のトランスクリプト。
- **サーバー / Unity のステータス** — スロットは起動しているか、どのブランチか、画面スタックは何か、Unity は Play モードか。
- **関連するログ** — Unity コンソール + サイドカーサーバー、そのチャットのセッションにフィルタされ、同期スクロールされる。
- **完了アーティファクト** — エージェントが作成した `proof.md`（および付随する `before/`、`after/`、`diff.patch`）がインラインでレンダリングされる。

これは「何が起きたかを見せてくれ」というビューです。生のファイアホースではなく — 「これはうまくできているか?」に答えるための、厳選された断面です。

**対応先:** [POPBOT_DESIGN.md → App layout](POPBOT_DESIGN.md#app-layout)（チャットカラム + 下部ログパネル）、[Proof artifacts](POPBOT_DESIGN.md#proof-artifacts-agent-debug-deliverable)。証跡レンダラーは `src/renderer/chat/ProofViewer.tsx`（計画中）にあります。

## US-9 · ジャストインタイムの権限付与

> *「完全に自律的にはやらせるべきでない様々なことについて、エージェントに簡単に許可を与えられるようにしたい。」*

エージェントが常時一時停止リストにある操作（`git push`、`gh pr create`、スロット外での `rm`、許可リストにないホストへのネットワーク呼び出しなど）を行おうとすると、PopBot は一時停止して私に確認します。付与フローは次のとおりです。

- モーダルが、エージェントが**何を**しようとしているか、**なぜ**（エージェントの申告した理由）、そして**コマンド / 引数**を表示します。
- 私は**一度だけ許可**、**このチャット/セッションのみ許可**、**常に許可**（ツールごと、対象ごとの永続的なルール）、または**拒否**を選べます。
- 許可ルールはチャットごとに蓄積され、チャット設定パネルに表示されるので取り消せます。
- ハードコードされた拒否リストは UI から決して上書きできません — [adr/0004](../adr/0004-canusetool-policy-boundary.md) を参照してください。

要点は、自律性がデフォルトでありながら、ターミナルを開いたりエージェントに付きっきりになったりせずに、特定のリスクある操作を摩擦なく承認できることです。

**対応先:** [POPBOT_DESIGN.md → Autonomous mode](POPBOT_DESIGN.md#autonomous-mode)、[adr/0004 — canUseTool policy boundary](../adr/0004-canusetool-policy-boundary.md)。付与ストアは `src/main/agents/policy/` にあります。

## Deviations and additions

このセクションは、ユーザーストーリーがロックされた設計から乖離している箇所にフラグを立てます。実装する際は、ユーザーストーリーを正とし、設計ドキュメントの方を更新してください。

### 第 3 のアテンションソースとしての Slack（US-1）

オリジナルの設計は Linear チケットと未レビューの PR をカバーしています。Slack メッセージはスコープ外でした。US-1 を実現するために:

- 左上のタブグループに、Tickets や Reviews と並んで**Slack パネル**を追加します。
- ソース: 自分宛の Slack DM、@mention、自分がオーナーのチャンネル内のメッセージ。フィルタリングルールはチャット生成ワークフローごとに TBD。
- 認証: Slack OAuth（トークンは `keytar` 経由でキーチェーンに保存）。
- Slack メッセージからチャットを生成すると、会話のコンテキストでエージェントに種付けされます。

これは**まったく新しいサブシステム**です — Slack API クライアントは `src/main/slack/`、パネルは `src/renderer/panels/slack/`。[PHASING.md](PHASING.md) の Phase 3 に、他のパネルと並べて段階的に組み込みますが、後付けではなくファーストクラスのものとして扱ってください。
