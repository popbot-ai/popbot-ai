# PopBot — Feature & Workflow Guide

PopBot は**多数の AI コーディングエージェントを並列で**走らせるためのデスクトップコックピットです。このガイドは、その基盤となる考え方 — なぜ存在するのか、各パーツがどう機能するのか、何が設計を形作ったのか、そして Proof of Play のチームが出荷済みの実際の、アセットの多いプロジェクトでどう使ったのか — を扱います。UI は自分で見つけられるエンジニア向けに書かれています。ここでのポイントは理由であり、台本に従うのではなく自分のワークフローにツールを適応させられるようにすることです。

自分のワークフローに適応させることは、後付けではなく意図された使い方です。PopBot はリファレンス実装として公開されています — 固定された製品としてではなく、チームのために改変するための「型」であり、AI 時代においてソフトウェアが最良の形で作られる方法についての一つの見方を反映しています。すなわち、エージェントの群れを走らせるチームは、決定が固定された一つのツールを採用するより、ツールを所有し作り替える方が概して良い結果を得られる、という見方です。以下の各パーツの背後にある「なぜ」を、それを変えるにはどこを切り込むかの地図として読んでください。[自分たち用に作り替える](#自分たち用に作り替える) に、どのように、どこを、そしてなぜという詳細があります。

## なぜ PopBot を作ったのか

AI コーディングエージェントを 1 体走らせるのは簡単です。**同時に複数**を動かしたいと思った瞬間、3 つの問題が現れます。

1. **隔離。** 2 体のエージェントが同じチェックアウトを編集すると、お互いの作業を壊します。3 体のエージェントに 1 つの作業ツリーというわけにはいきません — そして大規模なゲームプロジェクトでは、3 つの完全なチェックアウトを持つ余裕もありません。
2. **監督。** エージェントは速く、たいていは正しいですが、「たいてい」では `git push`、`p4 submit`、PR を開くといった操作には不十分です。すべてのファイル編集を見張ることなく、取り返しのつかない操作には人間のゲートが必要です。
3. **検証。** コンパイルが通るコードは、動くコードとは限りません。特にゲームでは、唯一本物のテストは*実際に実行し*、クリックして操作してみることです。アプリを見ることができないエージェントは、推測しているだけです。

PopBot は、ライブなゲームを出荷する小さなチームのために、この 3 つすべてを解決するために作られました。核となる発想はこうです。作業の各単位 — チケット、バグ、レビュー — を**チャット**として扱い、各チャットに専用の隔離された**ワークスペース**（必要ならアプリの専用実行コピーも）を与え、**自律的だがゲートされた**形で動かし、フリート全体を 1 つのウィンドウに表示することで、一人が一度に十数体のエージェントを率いられるようにする、というものです。

この設計は、具体的な一連の[ユーザーストーリー](USER_STORIES.md)に基づいて駆動されました。*「エンジニアとして、チケットをクリックすると、エージェントが正しいブランチでその作業を始めてほしい。」* *「レビュアーとして、チェンジリストを開けば、何もチェックアウトせずに本物のレビューを得たい。」* *「リードとして、壁を一目見れば、どのエージェントが自分を必要としているか分かってほしい。」* 以下のすべては、これらに応えるために存在します。各パーツがなぜそのような形をしているかを理解すれば、自分たちのスタック向けにフォークするときに、何を残し、何を置き換えるべきかが分かるはずです。

## 中核となる概念

### エージェントとモデル

すべてのチャットは 1 つの**エージェントバックエンド**によって駆動されます。

- **Claude Code** — Claude Agent SDK 経由。モデル: **Claude Opus**（デフォルト）と **Claude Fable**。
- **Codex** — OpenAI Codex SDK 経由。モデル: **GPT / Codex**。

PopBot はこれらのエージェントを再実装しているのではなく、公式 SDK を通じて**本物のエージェントを駆動しています**。この SDK は、あなたがターミナルで実行するのと同じ **`claude`** と **`codex`** のコマンドラインツールをラップしています。各エージェントのフルパワー — そのツール、スキル、MCP サーバー、サブエージェント — はすべてのチャット内で利用可能であり、PopBot はあなたがインストールしている CLI のバージョンと常に足並みを揃えます。ターミナルの Claude Code で動くなら、ここでも動きます。これは意図的な賭けです。エージェントは急速に進化し、それらをラップまたはフォークしたものは陳腐化してしまいます。CLI を直接駆動することで、PopBot はすべてのアップグレードを無償で継承します。

チャットごとに、バックエンド、**モデル**、**reasoning effort**（`low` → `xhigh` / `max` — effort が大きいほど、より深く考え、より徹底したツール使用が行われますが、コストとレイテンシは高くなります）を選択します。**デフォルト**は、*新規チャット*用と*コードレビュー*用で別々に設定できます（レビューはフィーチャー構築とは異なる深さを求めるため）。タスクに応じてチャットごとに上書きすることもできます。

長時間にわたる作業では、2 つのセッション制御が重要です。

- **セッション途中での切り替え。** 実行中のチャットでモデルや effort を変更すると、PopBot はスレッドを失うことなくエージェントを再構成します。
- **コンテキストを引き継いだ再起動。** このチャットのトランスクリプト（冒頭のターンと最新のターン）を種にした*新しい*エージェントセッションを立ち上げます。セッションが長くなりすぎたり、行き詰まったりしたときに便利です。会話履歴は保持され、エージェントは単にクリーンなランタイムを得るだけです。

各連携の認証情報は**あなたのマシン上にローカルに**、アプリ自身のデータベースに保存されます — このリポジトリには決して保存されません。

### スロット: ウォームで、隔離された、使い捨てのワークスペース

**スロット**は並列性の単位であり、PopBot の中心的な発想です。N 体のエージェントを走らせる素朴な方法は、リポジトリを N 回チェックアウトすることです — しかしこれは共有ツリー上で衝突するか、N ×（チェックアウト時間 + ビルドキャッシュ）のコストがかかります。スロットは「エージェントに、*本物で独立した*作業場所を与えつつ、それがすでに*ウォーム*で*返却しやすい*ようにするにはどうすればよいか」という問いへの答えです。

スロットには 3 つの性質があり、そのどれもが不可欠です。

- **隔離されている。** 各スロットは自身のブランチ（または Perforce ストリーム）上の独自の作業ディレクトリなので、N 体のエージェントが N 個のブランチをゼロ干渉で編集できます。あるエージェントの `git reset` が他のエージェントの作業に触れることはありません。
- **ウォームである。** スロットは、使用をまたいで持続するステートフルなビルド成果物を保持します — ゲームエンジンの場合は独自のインポート/アセットキャッシュ、独自のデータディレクトリを持つ専用の**サイドカーサーバー**、割り当てられた**ポート**、スロットごとのログ、そしてチャットがアクティブな間はライブな**エディタプロセス**。素の作業ディレクトリは隔離された*ソース*を与えてくれますが、スロットは隔離され、すでに*ウォームな*、ビルド・実行・テストのための場所を与えてくれます。
- **使い捨てである。** スロットはプールされます。チャットは自分の生存期間中、空いているスロットを**リース**し、閉じるときに**返却**します。ウォームなワークスペースの作成にはコストがかかりますが、既存のものを再利用するのはほぼ無料です。そのため PopBot はスロットのプールをウォームに保ち、作業をそこで回します。

**エンジン作業において「ウォーム」がすべてを決める理由。** ゲームエンジンは巨大な処理済みアセットキャッシュ（Unity の `Library/`、Unreal の `DerivedDataCache`）を保持しており、これはしばしば数ギガバイトに及び、生成に費用がかかります。新しいチェックアウト、あるいはそれを無効化するブランチ切り替えは、エンジンに**プロジェクトの再インポート**を強制し、これには数分かかることがあります。すべてのタスク、すべてのブランチ切り替えでこれを払っていては、エージェントはコードを書くよりもエンジンを待つ時間の方が長くなってしまいます。スロットは、それぞれに**独自の永続キャッシュ**を与えることで、この税を撤廃します。

- **エージェントをスロットに戻すのに数分ではなく数秒**で済みます — キャッシュはすでにウォームなので、本当に変更されたアセットだけが再処理されます。
- **スロットはエディタを*起動したまま*にできます。** 「粘着的」な再利用（同じスロット、同じブランチ）は、コールドスタートではなくほぼ即座にライブなエディタをエージェントに渡します。
- **10 体のエージェントが 1 つのインポートキャッシュを奪い合うことはありません。** 各スロットが独自のウォームキャッシュを持つため、並列のゲーム作業が単一の再インポートの背後で直列化することはありません。

どのブランチ切り替えの前にも、PopBot は**安全シーケンス**を実行します — 未コミットの作業をスタッシュし、エージェントが所有するコミットを上書きすることを拒否し、切り替えを行い、状態を復元します — これにより、スロットの受け渡しで作業がひそかに失われることはありません。スロットは**スロットプール**モード（再利用、デフォルト）または、ウォームさよりクリーンな状態を優先したいときの**エフェメラル**モード（チャットごとに新しいワークスペース）で動作できます。

> **重要な理由:** 隔離は、「10 体のエージェントを同時に」を破滅的ではなく安全にするものです。ウォームさは、それを*高速*にするものです。使い捨て可能性は、それを*安価*にするものです。どれか一つでも欠ければ、並列エージェントはもう見合わなくなります。

### Copy-on-write: 1 つのリポジトリのディスク容量で無制限にコピー

隔離とウォームさは、スロットの*ファイル*が安価である場合にのみ実現可能です。小さなリポジトリであれば、N 個の git worktree で十分です。しかしテラバイト級のゲームプロジェクト — 巨大なアセットライブラリを持ち、多くのチームでは Git ではなく **Perforce** を使っている — では、N 個の実コピーは数百ギガバイトに達し、それぞれの実体化に数分かかります。これはモデル全体を破綻させます。

そこで、スロットのワークスペースは**copy-on-write なフォルダ**になっています。すべてのスロットはリポジトリの 1 つの**ベースイメージ**を共有し、実際に変更したブロックだけを保存します。実際的な結果は次のとおりです。

- **テラバイト級のツリーの新鮮でライブな完全コピーが数秒で**用意できます — 浅いビューではなく、本物の編集可能なファイルです — そして同じくらい速く解放されます。
- **無制限のコピーが単一リポジトリ分のディスクで済みます。** 1 TB のプロジェクトで 10 体のエージェントを動かしても、10 TB は必要ありません。約 1 TB に加え、各スロットの小さな差分だけで済みます。
- **Windows、macOS、Linux で同じように動作します**（`shado` — PopBot のシャドウワークスペース層経由。Windows では differencing VHDX、それ以外では OS ネイティブの CoW ファイルシステム）。これによって Perforce のツリーがそもそも参加できるようになります。

これは、スロットという発想を「わずかな worktree を持つ Web リポジトリ」から「エージェントのフリートを抱える AAA 級のゲームツリー」までスケールさせるパーツです。最も目立たない機能でありながら、間違いなく最も重要な機能でもあります。安価なコピーがなければ、ウォームで隔離されたスロットは贅沢品にすぎません。それがあれば、デフォルトになります。

### ソースコントロール: Git と Perforce

PopBot はソースコントロールを共通インターフェースの背後にある**プロバイダー**として扱います。なぜなら「隔離されたブランチでエージェントを走らせ、変更をレビューして着地させる」という形は、バックエンドが Git であろうと Perforce であろうと同じだからです。どちらもファーストクラスです。

- **Git** — 隔離のための worktree、チャットごとのブランチ、`gh` CLI 経由の PR、レビュー面としての GitHub。
- **Perforce** — copy-on-write なシャドウワークスペース上のチャットごとのストリーム/ブランチ、作業単位としてのチェンジリスト、レビュー面としての **Helix Swarm**。Swarm レビューは GitHub PR と同じ Reviews 受信箱にピン留めされ、それぞれが独自のレビューチャットを開きます。

以下で見る概念 — ベースブランチ、git/SCM パネル、テンプレート化されたアクション、レビュー受信箱 — は、この共通インターフェースに対して書かれています。「ブランチ」や「PR」という表記は、Perforce を使っている場合は「チェンジリスト」や「Swarm レビュー」と読み替えてください。ワークフローは意図的に同一です。

### 受信箱: 1 つのキュー、複数のソース

受信箱は、統合ではなく*発想*です。**あなたに割り当てられた作業と保留中のレビューが優先度順に並び、それぞれワンクリックでエージェントのチャットになる**というものです。それに供給するものはプラガブルです。

- **チケット** — あなたに割り当てられた **Linear** イシュー、**Jira** イシュー、**GitHub Issues**（GitHub Issues のサポートはより新しく、まだやや実験的です）。1 つをクリックすると、PopBot がブランチに名前を付け、スロットをリースし、チケットを *In Progress* に移動し、その説明でエージェントに種を与えます。
- **レビュー** — あなたのレビューを待っている **GitHub** プルリクエストと **Helix Swarm** チェンジリスト。1 つをクリックすると、リポジトリ不要のレビューチャットが即座に開きます。

ソースを追加してもワークフローは変わりません — 同じキューに行が追加されるだけです。それがポイントです。受信箱をキューとして扱うモデルは汎用的であり、具体的なトラッカーは交換可能なデフォルトにすぎません。

### リポジトリ不要のチャット（コードレビュー用）

すべてのチャットがワークスペースを必要とするわけではありません。**レビュー**は読み取り専用です — 編集はせず、diff と周辺のコードを読んでコメントを投稿します。そのため、レビューチャットは**リポジトリ不要**です。即座に生成され、スロットをリースせず、ワークスペースを消費しません。

これは意図的で重要な分割です。

- **ビルドチャット**（フィーチャー/バグ）はスロットをリースし、ウォームアップに少し時間がかかることがあり、その生存期間中ワークスペースを保持します。
- **レビューチャット**は**即座で無料**です — レビューキューをトリアージするために 5 つ開いても、ビルドチャットは邪魔されずに動き続けます。

これはまた、スロットプールが本当に隔離を必要とする作業のために確保されることも意味します。レビューがビルドからスロットを奪うことは決してありません — プールが RAM とディスクによって制限されている場合に大きく重要な性質です。

### ベースブランチ

チャットが実際にコードを書く*とき*、それは**ベース** — 通常は Git の `develop`/`main`、または Perforce のメインラインストリーム — からフォークします。PopBot はリポジトリごとにベースをデフォルト設定し、前回の選択を記憶しているので、一般的なケースではワンクリックで済み、タスクが必要とするときにはフィーチャーラインやリリースブランチから分岐することもできます。新しいブランチ名は、あなたの規約（例: `<username>/<ticket>-<slug>`）から導出されるため、ブランチは一貫性があり、そのチケットまで追跡可能です。ベースは後続のアクション（「ベースへリベース」、「ベースに対して PR / レビューを開く」、ドリフトチェック）も左右します。

### 永続的でアーカイブ可能なチャット

すべてのチャットはローカルに保存される**永続的なトランスクリプト**です — 文章、ツール呼び出し、diff、権限の決定、すべてです。何も一時的ではありません。

- チャットを**閉じる**とそのスロットは解放されます（他のエージェントのためにワークスペースが解放されます）が、**すべては保持されます**。チャットは**アーカイブ**に移動します。
- アーカイブからチャットを**再オープン**すると、スロットが再リースされ、そのブランチが復元され、エージェントは**全履歴**を保った状態で再開します — 数日後にフィーチャーを再開してレビューフィードバックに対応する際、何も再説明する必要がありません。以前とは*別の*スロットで再オープンする場合、PopBot は事前にそのことをエージェントに伝えるので、エージェントは新しい作業ディレクトリにきれいに順応します。
- アーカイブは名前、チケット、ブランチ、内容を横断して検索可能です。

ロールバックが単に「もう一つメッセージを送る」ことにすぎないため（破壊的な履歴編集は存在しません）、チャットは変更がどのように行われたかの完全で監査可能な物語を蓄積していきます。

## ワークスペースの構造

![PopBot UI anatomy](../../images/anatomy.png)

| 領域 | 内容 |
|---|---|
| **受信箱 — チケットとレビュー** | 割り当てられたチケット（Linear / Jira / GitHub Issues）とあなたを待っているレビュー（GitHub PR / Swarm チェンジリスト）を、優先度順に表示。行をクリックすると、そのコンテキストで種付けされたチャットが生成される。 |
| **スロット** | ウォームなワークスペースのプール。各ピルは、そのスロットが空いているか、あるチャットにリースされているかを示す。 |
| **チャットアーカイブ** | 過去のすべてのチャット。検索可能で、全履歴を保ったまま再オープンできる。 |
| **チャットサムネイル** | 開いているすべてのチャットのライブなスクロールプレビュー — 各エージェントが今何をしているかの実際のビューで、ステータスにより色分けされる: 青 = 実行中、緑 = 完了、黄 = あなた待ち、赤 = エラー、灰 = アイドル。 |
| **チャット** | フォーカスされたエージェントセッション — ストリーミングされる文章、ツール呼び出し、インラインのコード diff。 |
| **チャットごとのターミナル** | そのチャットのワークスペースに固定された埋め込みターミナル。 |
| **SCM パネル** | 作業ツリー/チェンジリストのステータス、最近のコミット、ファイル diff、ワンクリックのコミット / プッシュ / PR / レビュー操作。 |

すべてのチャットが**サムネイルストリップ**にとどまり、**カラムが横に並ぶ**ため、ステータスを探し回る必要はありません。色がシグナルです — 青 = 実行中、緑 = 完了、黄 = あなた待ち、赤 = エラー — 一目で、どのエージェントが動いていて、どれが完了していて、どれが**あなたを待っている**かが分かります。

しかし各サムネイルは単なるステータスライトではなく、**会話のライブプレビュー**でもあります — そのため一目で、各エージェントが実際に*何に*取り組んでいるかを見ることができます。それが**無駄な作業を早期に察知する**ことを可能にしてくれます。エージェントが間違った方向に進んでいるのを見つけ、時間とトークンを浪費する前にリダイレクトできます。「完了した」後に行き止まりを発見するのではなく。フリートを監督することと、フリートに驚かされることの違いです。

### なぜサムネイルなのか、なぜ 1 つのビューなのか

このレイアウトは特定の問題への意図的な答えであり、その理由を述べる価値があります。多くのツールが最も間違えやすい部分だからです。

1 体のエージェントを走らせることは集中作業です。1 つの会話を見守り、応答します。*多数*を走らせることは**モニタリング**作業であり、モニタリングには異なる失敗モードがあります — ボトルネックはあなたのタイピング速度ではなく、あなたの注意力です。静かに脱線したエージェントは、あなたが気づき、理解し、捨てなければならない作業を生み出します。N 体のエージェントがいれば、*気づかないこと*のコストは N に比例してスケールし、自然なインターフェースは気づくことを難しくします。タブは 1 体を除くすべてのエージェントを隠し、起動して待つモデルはすべてを結果が出るまで隠します。

そこで、この設計は 2 つのことにコミットしています。

- **すべてのエージェントは常に可視である。** サムネイルストリップはフリート全体を一度に表示し、各サムネイルはスピナーではなく実際の会話のライブビューです。一歩下がって、一目で十数体のエージェントの状態を把握できるようになっているはずです — どのエージェントが動いていて、どれが詰まっていて、どれが止めたくなるようなことを今にもしそうか。
- **ステータスは色、内容は一目の距離。** 色は「誰が私を必要としているか」を 1 秒未満で答え、ライブプレビューはクリックなしで「これは何をしているか」を答え、横並びのカラムは、他を失うことなくどれか一つに入り込めるようにします。インターフェースは*安価な再チェック*に最適化されています。多数のエージェントがいれば、絶えず再チェックすることになるからです。

その見返りは、**早期に介入する**能力です。自律エージェントにおける高くつく間違いはクラッシュではなく — エージェントが自信満々に 1 時間かけて間違ったものを作ることです。意図を継続的に表面化させるビューは、それを事後的な発見から途中での軌道修正に変えます。それこそが、フリートがタブや通知の裏ではなく、常に画面に表示されている全体の理由です。

## Proof of Play でどう使われたか

PopBot は研究室の実験ではありませんでした。**Proof of Play** のチームによって、出荷済みの実際の、アセットの多いプロジェクトで日常的に構築・使用されました。その出自が設計判断のほとんどを説明しており、このツールが何のためのものかを理解する最も明確な方法です。

実際的な結果は明快でした。ウォームで隔離された copy-on-write なワークスペースというスロットモデルが、大規模なアセットツリー上での並列エージェント作業を実現可能にし、そのおかげでチームはより多くのことを成し遂げられました。複数のエージェントが、衝突したりスイッチのたびにエンジンの再インポート税を払ったりすることなく同時に走ることができたため、並列性がオーバーヘッドに変わるのではなく、スループットが向上しました。

典型的な一日の形はこうです。サムネイルの壁を開いたリードがいて、4、5 体のエージェントが飛んでいます — 何体かはフィーチャーチケットを地道にこなし、1 体はバグを追い、1、2 体はコードレビューをしています。リードは分刻みでコードを書いているわけではありません。**フリートを見守り**、ゲート（プッシュ、PR、リスクのある操作）でだけ、そしてサムネイルが黄色になったりエージェントが目に見えて脱線したりしたときにだけ介入します。チケットはチームの実際のトラッカーから来て、レビューはチームの他のメンバーが着地するのを見る実際の PR とチェンジリストです。

そのゲームプロジェクトが課した厳しい制約こそが、最も重要だと判明した機能そのものでした。

- **アセットツリーは巨大だった**ため、ウォームなスロットと copy-on-write なワークスペースは贅沢品ではありませんでした — それらがなければ、そのツリー上でエージェントのフリートを動かすことは単純に採算が合いませんでした。これが、この 2 つの発想がこのツールの背骨である理由です。
- **エンジンが「動くかどうか」の真実の源泉だった**ため、実行中のゲームを起動して操作できないエージェントは、ほとんどのゲームプレイ作業にとって役に立ちませんでした。だからこそテスト対象アプリとの統合があります。
- **ソースコントロールはゲームには Perforce、ツールには Git だった**ため、プロバイダーに依存しない SCM は選択肢ではなく必須でした。
- **一人が多数のエージェントを率いる必要があった**ため、コックピット全体が、深い単一セッションへの集中ではなく、*一目での監督*に最適化されています。

もしあなたの状況がこれらのどれかと響き合うなら — 大きなツリー、テストすべき実際のアプリ、1 体のエージェントでは手に負えない量の作業 — この設計はあなたのニーズに密接に対応するでしょう。まさにそのために作られたからです。そうでない場合、[自分たち用に作り替える](#自分たち用に作り替える) セクションは、発想を保ちながら具体的な部分を入れ替えることについてのものです。

範囲についての補足: そのプロジェクトは結局商業的な牽引力を得られませんでした。私たちはそうではないと主張しているわけではありません。しかしそれが提起したエンジニアリング上の問題は本物でした — 大規模なアセットツリー、エージェントのフリート、1 つのチーム — そして PopBot のうちそれを解決した部分こそが、ここで文書化されている部分です。このツールの価値は、そのゲームの結果に依存しません。私たちはそれをほのめかすよりも、はっきりと述べる方を選びます。

## End-to-end ワークフロー

### フィーチャーチケット

1. **通知 → 受信箱。** あなたに割り当てられたチケットが**チケット**受信箱に現れます（PopBot は Linear / Jira / GitHub Issues をポーリングし、優先度と期日でランク付けします）。通知ベルがそれをフラグします。
2. **開始はワンクリック。** チケットの行をクリックします。PopBot が、あなたのリポジトリとベース（前回から記憶）をデフォルトにした**新規チャット**ダイアログを開きます — 確認するか、エージェント/モデル/effort を調整します。
3. **スロットの割り当て。** このチャットはコードを書くことになるため、PopBot は**スロットをリース**します。空いているワークスペースを選び、チケットからブランチ名 `you/eng-123-<slug>` を導出し、（最初にスタッシュの安全シーケンスを実行したうえで）ワークスペースをそれに切り替えます。
4. **チケットの自動昇格。** チケットは自動的に **In Progress** に移動します（冪等かつ発火して忘れる方式）。コンテキストスイッチなしにボードが現実を反映します。
5. **エージェント開始。** エージェントは種付けされた最初のメッセージ（チケットのタイトル、説明、ブランチを埋め込んだ、カスタマイズ可能な*start-ticket*テンプレート）を受け取り、コードの探索、編集、コマンドの実行を始めます — すべてそのスロットのワークスペース内で。
6. **スロット内での検証。** ゲームの変更については、エージェントは**そのスロットでアプリを起動し**（2 台目のディスプレイでのエンジンエディタ + サイドカーサーバー）、機能を実行します — UI をクリックして操作し、ログを読み、スクリーンショットを撮ります。動くと推測するのではなく。
7. **ゲートされた終了。** プッシュする準備ができると、エージェントは**一時停止します**（プッシュはゲートされたアクションです）。サムネイルが黄色になります（「あなた待ち」）。
8. **あなたがレビューして出荷する。** **SCM パネル**を開き、diff を読み、**PR をプッシュ**（または**ドラフトをプッシュ**）を押します。このアクションは事前入力された指示をエージェントに送り、エージェントはブランチをプッシュし、あなたのベースに対して PR / Swarm レビューを開きます。

その間ずっと、あなたはそれを見ていませんでした — 他の 2 つのチケットについても同じことをしていたのです。ゲートでだけ介入しました。

### バグチケット

バグのフローは、よりタイトなループを持つフィーチャーフローであり、**並列性**を見せつけます。

1. バグレポートが届きます（チケット、またはバグの説明で手動でチャットを開始します）。
2. チャットを生成する → それが**自身の**スロットとブランチをリースします。進行中のフィーチャーチャットは完全に無傷のままです — 別のワークスペース、別のブランチ。
3. エージェントは**スロットでアプリを実行して**バグを再現し、原因を見つけ、修正し、再実行して再現がなくなったことを確認します。
4. あなたは**サムネイルストリップ**をちらりと見ます。フィーチャーチャットは緑（完了、あなたのプッシュ待ち）、バグチャットは青（実行中）。2 体のエージェント、2 つの隔離されたツリー、ゼロ衝突。
5. 承認のために一時停止したら修正をプッシュします。

### コードレビュー

1. **通知 → Reviews。** チームメイトがあなたのレビューを依頼します。PR（GitHub）またはチェンジリスト（Swarm）が **Reviews** 受信箱に現れます。
2. **即座の、リポジトリ不要のチャット。** クリックすると → **レビューチャット**が即座に開きます — スロットなし、チェックアウトなし、待ち時間なし。*start code review*テンプレート（diff だけでなく周辺のコードを読み、システムをたどり、実際のバグ、レース、エッジケース、セキュリティ、性能の問題を探る）で種付けされています。
3. **本物のレビュー。** エージェントは diff*と*その周辺のコードを読み、正しさについて推論し、GitHub または Swarm に**インラインコメント**と評決（承認 / 変更要求）を投稿します — その後、チャットで問題点をあなたに要約します。
4. **後で再レビュー。** 作者が修正をプッシュした場合、**再レビュー**を押します。PopBot は既存のレビューチャットにフォーカスし、エージェントに**新しいコミットだけ**を見るよう指示し、以前の各スレッドが実際に対応されたかを確認し、レビューを更新させます。

これらはすべて、あなたのビルドチャットが動き続けている間に起こります — レビューがスロットを取ることは決してありません。

### アーカイブされたチャットの再オープン

作業が一度で終わることは稀です。再オープンフローはファーストクラスの機能です。

1. フィーチャーチャットが PR を出荷し、あなたはスロットを解放するためにそれを**閉じました**。それは今**アーカイブ**にあります（トランスクリプトは完全に保持されています）。
2. 2 日後、その変更にレビューコメントが付きます。アーカイブでそのチャットを見つけ（チケット、ブランチ、テキストで検索）、**再オープン**します。
3. PopBot が**スロットを再リース**し、チャットのブランチをワークスペースに復元し、エージェントは**全履歴**を保った状態で再開します — 自分が何を作り、なぜ作ったのかをすでに知っています。以前とは別のスロットに着地した場合、PopBot はそれを新しい作業ディレクトリに順応させます。
4. レビューフィードバックを貼り付けるか要約します。エージェントはそれに対応し、スロットで再テストし、更新をプッシュします — 再オンボーディングも、失われるコンテキストもありません。

ブランチ、トランスクリプト、そして推論のすべてが持続するため、タスクを再開するのに数秒しかかからず、再説明は不要です。

## 統合されたソースコントロールとレビュー

ソースコントロールは、各プロバイダーのネイティブ CLI — GitHub には **`gh`/`git`**、Perforce には **`p4`** と Swarm API — を通じて深く組み込まれているため、エージェントが行うことはすべて、チームが普段見ている場所で見える本物の活動です。

- **Reviews 受信箱。** あなたのレビューを待っている GitHub PR と Swarm チェンジリスト（そしてあなた自身の最近の提出物）が、ワンクリックのチャットソースとして現れます。
- **PR / レビューのステータスチップ。** 変更にリンクされた各チャットは、ライブなステータスチップ — Open / Merged / Closed / Draft — を表示し、クリックすると GitHub や Swarm でそれを開きます。
- **SCM パネル。** どのビルドチャットについても、作業ツリー/チェンジリストのステータス、最近のコミット、ファイルごとの diff を見られます。ファイルをクリックすると、完全な unified-diff のオーバーレイが表示されます。
- **ワンクリックアクション。** テンプレート化された編集可能なアクションが、事前入力された指示をエージェントに送ります: **コミット**、**PR をプッシュ**、**ドラフト PR をプッシュ**、**Make ready**、**CR に対応**（レビューコメントに対応）、**ベースへリベース**。それぞれ `${branch}`、`${baseBranch}`、`${ticket}`、`${prnum}`、`${prurl}` のような変数を展開するので、エージェントは必要なものをちょうど手にできます。
- **あなたのベースに対する作成。** プッシュすると、チャットの設定されたベースに対して、あなたのブランチ規約で名付けられた PR（または Swarm レビュー）が開かれます。

レビューは独立した最適化された経路です（[コードレビュー](#コードレビュー) を参照）。

- **リポジトリ不要で即座** — スロットなし、チェックアウトなし。数秒でレビューのキューをトリアージできます。
- **diff だけでなくコンテキストを読む** — レビューテンプレートは、パッチにゴム印を押すのではなく、周辺のコードを読み、システムをたどり、バグ/レース/エッジケース/セキュリティ/性能を探すようエージェントに指示します。
- **チームが働く場所に投稿する** — GitHub や Swarm へのインラインコメントと提出済みのレビュー。
- **再レビューはスコープされる** — 2 回目のパスでは、エージェントは新しいコミットだけを調べ、レビューを更新する前に、以前の各スレッドが本当に解決されているかを確認します。
- **完全にカスタマイズ可能** — *start code review* と*再レビュー*のプロンプトは編集可能なテンプレートなので、厳密さ、チェックリスト、トーンをチームの基準に合わせて調整できます。レビューの*手順そのもの*（あなたの職場が GitHub や Perforce のレビューをどう進めたいか）はあなたが提供するものです — PopBot は推奨しサンプルを示すことはできますが、あなたのチームの基準はあなたのチームに残ります。

## スロット内でのテスト: テスト対象アプリ

ビルドチャットのスロットは単なるフォルダではありません — 作業を**実行し、検査する**場所です。

- **チャットごとのターミナル。** そのチャットのワークスペースに固定された埋め込みターミナル（xterm + 本物の PTY）。エージェントが作業している間、テストを実行したり、ログを検査したり、手動でコマンドを発行したりできます。チャットを切り替えても持続します。
- **エディタとの連携。** トランスクリプト内のすべての `path/to/file.ts:42` 参照は、チャットのワークスペースを基準に解決され、**VS Code** や **Cursor** で開くクリック可能なリンクです。
- **テスト対象アプリ。** スロットは**実際のアプリケーション**を起動できるので、エージェントは推測するのではなく駆動できます。Web アプリ、CLI、サービスであれば、これはほとんどエージェント自身が行います — スロットのターミナルであなたのビルドとテストのコマンドを実行し、実行中のサーバーにアクセスし、出力を読みます。PopBot はそれらについて何も特別なことを知る必要はありません。エージェントはあなたがするのと同じ方法でそれらを扱います。ゲームの**エンジン**は追加の処理が必要なケースです。エディタは独自のアセットキャッシュを持つ長命な GUI プロセスであり、自然なコマンドライン「実行してチェックする」ループを持たないからです。そこで **Unity** と **Unreal** については、PopBot はライブなエディタ + サイドカーサーバーを起動し、それを 2 台目のディスプレイに配置し、**エディタ内 MCP サーバー**経由でそれをエージェントに公開します。実行中の各エディタはその**スロットから導出された独自の MCP ポート**を得るので — エージェントは常に*自分の*エディタとだけ話し、他のスロットのエディタとは決して話しません — PopBot は各チャットのエージェントをそのエンドポイントに自動的に接続します（メモリ内なので、何もソースコントロールには残りません）。**カスタム**エンジンも同じ仕組みに組み込まれます。PopBot はスロットの識別情報をあなたの起動コマンドに渡すので、エージェントがそれをどう駆動するかはあなたが配線します。どの場合でも、エージェントはアプリを操作できます — UI をクリックし、ログを読み、スクリーンショットを撮り、動作をアサートします — そして PopBot がエディタのライフサイクル（サーバーの起動、ヘルスチェック、エディタの起動、ウィンドウの配置、解放時の停止）を管理し、利用可能な RAM に対して同時インスタンス数を予算配分します。

これは、変更が動くと*思っている*エージェントと、それが動くのを*見た*エージェントの違いです。これのどの部分もゲーム固有ではありません。Web やその他の開発は同等にファーストクラスの用途です。ゲームエンジンは単に、システムが認識しておかねばならない追加の状態（ウォームなアセットキャッシュ、テスト対象アプリとしてのエディタ）を伴うだけであり、その同じ追加の状態こそが、このツールの新しい部分 — ウォームスロット、copy-on-write ワークスペース、エージェントが駆動できる実行中のアプリ — の最も鋭いデモンストレーションにしているのです。

## 権限と安全性

ハードな下限を伴う自律性:

- **自動許可（サイレント）:** **スロットのワークスペース内**での読み取り、編集、シェルコマンド、スロット自身のサービス（エディタ MCP を含む）への呼び出し、内部的なエージェント操作。エージェントはただ作業します。
- **常にゲートされる（あなたのために一時停止する）:** `git push` / `p4 submit` / reset / force、ワークスペースの**外**での操作、PR やレビューを開くこと、スクラッチディレクトリ外での削除、メッセージの送信（Slack/email）、システムやエージェントの設定への変更、許可リストにないホストへのネットワーク呼び出し。
- **それ以外すべて:** あなたに判断を求めます。

何かを承認するとき、それを**一度だけ**、**セッション中だけ**、または**永続的に**（このツール/対象を常に許可）付与できます。MCP サーバーも同じ方法で許可できます — あるスロットのエディタ MCP を一度許可すれば記憶され、その許可は Preferences → Permissions で可視かつ取り消し可能です（PopBot はこの方法で Unity/Unreal のエディタ MCP を自動的に有効化します）。許可はチャットごと、またはグローバルであり、すべて**取り消し可能**です。ハードな拒否の下限（push/submit、ネットワーク、ツリー外）はコードの中に存在し、UI のルールでは上書きできません — そのため、誤って設定された許可によって、エージェントが自力でメインラインに着地することはできません。

## ローカライゼーション

PopBot のインターフェース全体 — メニュー、設定、ダイアログ、すべて — は完全にローカライズされています。アプリは**12 言語**で出荷されます: 英語、スペイン語、フランス語、ドイツ語、日本語、韓国語、簡体字中国語、ブラジルポルトガル語、ロシア語、イタリア語、ポーランド語、ウクライナ語 — 再起動なしで言語メニューからいつでも切り替えられます。PopBot をフォークする場合、各ロケールは単一のメッセージカタログなので、言語の追加や調整は UI 中の探検ではなく、限定された変更で済みます。

## Preferences

すべてがアプリ内で設定されます（設定ファイルの編集は不要です）。

- **エージェント** — 新規チャットとコードレビューで別々の、デフォルトのモデルと reasoning effort。
- **リポジトリ** — フォルダ優先の SCM 対応ウィザードでリポジトリを追加/編集: パス、プロバイダー（Git/Perforce）、ベースブランチまたはストリーム、色、スロットのプレフィックス、ワークスペースディレクトリ、スロットプール対エフェメラルモード。
- **ランタイムとスロット** — プールサイズ（同時に動くエージェントの数）、スロットの事前作成/削除、添付ファイルの保持、copy-on-write ワークスペースのベースイメージ更新。
- **連携** — Linear、Jira、GitHub、Helix Swarm を接続（認証情報はローカルに保存）、プロバイダーごとに設定可能なレビューポーリングレート、保存前のテスト。
- **ソースコントロール** — ブランチ名の規約、デフォルトのベース、編集可能なアクションテンプレート。
- **外部アプリ** — ターミナル（iTerm）、エディタ（VS Code / Cursor）、エンジンのバイナリとエンジンごとのオプション（エディタ MCP のベースポートを含む）、URL ルーティング用の任意の Chrome プロファイル。
- **プロンプトテンプレート** — 種として与えられるすべてのプロンプト（start ticket、start/再レビュー、各アクション）が変数リファレンスカード付きで編集可能。
- **権限** — 永続的な許可をレビューし取り消す。MCP サーバーごとの許可を含む。
- **通知** — トーストの配置とアラートの動作。
- **言語** — インターフェースのロケールを切り替える。

> パネルごとのスクリーンショット付きリファレンスについては、**[Configuration Guide](CONFIGURATION.md)** を参照してください。

## 自分たち用に作り替える

PopBot を適応させることは、主要な意図された用途です。それはリファレンス実装として公開されており、その設計は、AI 時代においてソフトウェアが最良の形で作られる方法についての一つの見方を反映しています。すなわち、チームは動く型を受け取り、それがなぜその形をしているのかを理解し、決定が固定されたツールを採用するのではなく、自分たちのスタック、ツール、規約に合わせてそれを作り替える、という見方です。

その形は汎用的です: **エージェント + 隔離された、ウォームな copy-on-write スロット + キューとしての受信箱 + テスト対象アプリ。** そのパターンは、一度に複数のコーディングエージェントを走らせるほとんどのチームに当てはまります。**MIT ライセンス**であり、フォークされるように構成されています — コードは*小さな共通インターフェースの背後にあるプロバイダー*として整理されているため、他の部分に触れることなく、部品を追加したり入れ替えたりできます。一般的なアプローチは、中核となるアイデアを保ちつつ、具体的な実装を置き換えることです。

以下に、それぞれの*どのように、どこで、なぜ*とともに接続点を挙げます。どれもプラガブルな実装を持つインターフェースです。実際的な進め方は、既存の実装にパターンマッチして、自分自身のものを追加することです。

- **テスト対象アプリを入れ替える。** *なぜ:* 全体のポイントは、あなたのアプリを*実行し検証する*エージェントであり、「あなたのアプリ」は人それぞれ異なるからです。*どこ:* `src/shared/gameEngine.ts`（エンジンの記述子、MCP の配線）と `src/main/ipc/apps.ts`（起動 + ライフサイクル）。Unity と Unreal は 2 つの実装です。**カスタムエンジン**のフックはすでにスロットの識別情報（`POPBOT_SLOT`、導出されたポート）をあなたの起動コマンドに渡しているので、あなたの Web アプリ、CLI、テストハーネスを配線するのは「起動コマンドと、エージェントがそれとどう話すかを埋めるだけ」です。
- **受信箱の接続先を変える。** *なぜ:* 受信箱をキューとして扱う発想が持続的なアイデアであり、具体的なトラッカーは詳細にすぎないからです。*どこ:* `src/main/tickets/` — `provider.ts` の `TicketSource` インターフェースを実装し、あなたのトラッカーのデータを共有 DTO に正規化し、`registry.ts` に登録します（ファイルヘッダーには文字どおり「トラッカーを追加するのはここに 1 行と `*Source.ts` モジュールを足すだけ」と書かれています）。Linear、Jira、GitHub Issues が具体例です。レンダラーは決してプロバイダー id で分岐しないので、UI には触れません。
- **ソースコントロールを追加・入れ替える。** *なぜ:* 「変更を隔離し、レビューし、着地させる」はプロバイダーに依存せず、Git と Perforce は単に 2 つのバックエンドにすぎないからです。*どこ:* `src/main/scm/` — `gitProvider.ts` / `perforceProvider.ts` に倣って `SourceControlProvider` 基底クラス（`provider.ts`）を拡張します。きれいに抽象化されない振る舞いは `if (provider === …)` ではなく**能力（capabilities）を通じて機能検出**されるので、非常に異なる VCS でも、呼び出し側に特殊扱いさせることなく独自のクライアント UI を選択できます。
- **レビュー面を入れ替える。** *なぜ:* レビューはチームがすでに見ている場所に着地すべきだからです。*どこ:* `src/main/reviews/` の背後にあるレビュープロバイダー（`git/reviews.ts` 経由の GitHub PR、`p4/swarmReviews.ts` 経由の Swarm チェンジリスト）。レビューの*手順そのもの* — あなたの職場がレビューをどう進めたいか — は意図的にツールに同梱**されていません**。それはあなたが提供する職場ごとのスキルであり、PopBot は推奨しサンプルを示しますが、あなたの基準を押し付けることは決してありません。
- **アクションとプロンプトを再構成する。** *なぜ:* ブランチ規約、PR/レビューのフロー、エージェントへのブリーフの仕方はチーム固有だからです。*どこ:* コードは不要です — git アクションテンプレートと、種として与えられるすべてのプロンプト（start-ticket、start/re-review）は**Preferences 内で編集可能**で、変数リファレンスカード付きです。厳密さ、チェックリスト、トーンを変更してください。
- **コアは維持する。** *なぜ:* これらは全体を機能させるアイデアであり、変更するのに最も慎重であるべき部分だからです。ウォームスロット、copy-on-write ワークスペース（`src/main/shado/`）、永続的なチャット、ハードコードされた権限フロア、並列エージェントのコックピットは揺るがない背骨です。それ以外のすべては動かされることを意図しています。

プロセス境界、IPC、各サブシステムの所在については、**[Architecture](ARCHITECTURE.md)** ドキュメントを読んでください — 変更したい接続点を見つけるための地図です。オブジェクトモデル（Chat、Slot、AgentSession とそのライフサイクル）については、**[Core Model](CORE_MODEL.md)** を参照してください。

一度に複数のエージェントを走らせるチームにとって、これは分解して別のワークフローの周りに再構築されることを意図した、動く出発点です。
