## なぜ PopBot なのか

AI コーディングエージェントを 1 つ動かすのは簡単です。しかし同時に何体も動かそうとすると、単一エージェントでは発生しない問題が現れます。それぞれの作業を隔離してお互いの成果を上書きしないようにすること、実際に作ったものをテストすること、レビューすること、そして取り返しのつかない操作にゲートをかけ、どのエージェントも監視なしにそれを実行できないようにすることです。

PopBot はそのためのオーケストレーション層です。チケットやレビュー依頼をワンクリックでエージェントセッションに変換し、各エージェントに隔離されたワークスペース（専用の作業コピー、ゲームプロジェクトの場合はテスト対象アプリの専用実行環境も）を与え、デフォルトでは自律的に動かしつつリスクのある操作には人間のゲートをかけ、すべてのトランスクリプト、diff、ターミナル、ログを 1 つのウィンドウに集約します。オペレーターはカラムをざっと見て、ゲートされたアクションを承認し、出荷するだけです。

これは **Proof of Play** の小さなチームによって作られ、出荷済みの実際の、アセットの多いプロダクションプロジェクトで日常的に使われてきました。それがこのツールが証明された環境です。何ギガバイトものアセット、実際のソース管理、実際の締め切り。ウォームで隔離された copy-on-write なワークスペースという「スロット」モデルこそが、そこで並列にエージェントを走らせることを実用的にし、チームが一度にこなせる作業量を増やしました。私たちは PopBot をリファレンス実装として公開・サポートしています。そのまま使う完成品としてではなく、自分たちのスタックとワークフローに合わせて取り込み、作り替えるための「型」としてです。これは、AI 時代におけるソフトウェアの最良の作り方についての一つの見方を反映しています — エージェントの群れを走らせるチームは、固定されたツールを採用するより、ツールを所有し自分で改変する方がうまくいく、という見方です。MIT ライセンスで、フォークしやすいように構成されています。詳しくは [自分たち用に作り替える](#自分たち用に作り替える) を参照してください。

## 特徴

### ライブサムネイル付きマルチチャットビュー

開いているすべてのチャットは常に画面に表示されます — サイドバイサイドの**カラム**の上に並ぶ、**ライブサムネイル**のストリップです。各サムネイルは単なるステータスドットではなく、そのチャットの実際に更新され続けるビューであり、状態（実行中、完了、あなた待ち、エラー）に応じて色分けされます。一目で*すべてのエージェントが何をしているか*が分かり、誰があなたを必要としているかも分かります。そして**間違った方向に進んでいるのを早期に察知**して、時間とトークンを浪費する前にリダイレクトできます。一人で 1 つのウィンドウからエージェントの群れ全体を監督できます。

### ウォームスロット — 再インポート税のない並列エージェント

作業中の各チャットは**スロット**をリースします — 永続的な作業コピーに加え、専用のウォームなビルド状態を持ち、一度作成されたら使い回されます。ゲームエンジンの場合、スロットは自身のホットなアセットキャッシュ（Unity の `Library`、Unreal の DDC）を保持し、エディタを起動したままにできるため、エージェントをそのスロットに戻すのに**数分かかる再インポートではなく、数秒**で済みます。10 体のエージェントが単一のインポートキャッシュを奪い合うことなく、真のブランチ隔離のもとで動作します。[スロットの仕組みへ →](GUIDE.md#スロット-ウォームで隔離された使い捨てのワークスペース)

### 1 つのリポジトリのディスク容量で無制限にコピー

スロットのワークスペースは**copy-on-write なフォルダ**です。すべてのスロットは 1 つのベースイメージを共有し、変更した部分だけを保存します。そのため、**テラバイト級**のゲームツリーの新鮮でライブな完全コピーが**数秒**で用意できます — 浅いビューではなく、本物の編集可能なファイルです — そして無制限にコピーしても、かかるディスク容量は単一リポジトリ分だけです。**Windows、macOS、Linux** で動作し、これによって巨大な Perforce ツリーもフリートに参加できるようになります。[重要性の理由へ →](GUIDE.md#copy-on-write-1-つのリポジトリのディスク容量で無制限にコピー)

### Git と Perforce、レビュー機能内蔵

ソースコントロールは 1 つのインターフェースの背後にある**プロバイダー**です。**Git**（worktree、ブランチ、`gh` 経由の PR）と **Perforce**（シャドウワークスペース上のストリーム、チェンジリスト、**Helix Swarm** レビュー）はどちらもファーストクラスです。*各チャット自身のワークスペース*にスコープされたソースコントロールパネルが、そのブランチのステータス、コミット、ファイルごとの diff を表示します。ワンクリックのテンプレート化されたアクション（**コミット**、**PR をプッシュ**、**Make ready**、**CR に対応**、**ベースへリベース**）が、`${branch}` / `${ticket}` / `${prnum}` を埋め込んだ事前入力済みの指示をそのチャットのエージェントに送ります。

### 1 つの受信箱、複数のソース

ループ全体を 1 か所に。あなたの**受信箱** — **Linear**、**Jira**、**GitHub Issues** から割り当てられたチケットに加え、**GitHub PR** や **Swarm チェンジリスト**としてあなたを待っているレビュー → 隔離されたスロットでの**進行中**のエージェント作業 → **プッシュ**して PR / レビューを開く → 完了したチャットを**アーカイブ** → 後で全履歴を保持したまま**再オープンして再開**。チケットをクリックすると、PopBot がブランチに名前を付け、スロットをリースし、チケットを *In Progress* に移動し、エージェントに種を与えます — その後、マージされた変更まで、そしてそこからまた元に戻るまで運びます。[ワークフローの詳細解説へ →](GUIDE.md#end-to-end-ワークフロー)

## 追加機能

- **本物の Claude Code と Codex — 再実装ではありません。** 各チャットは公式 SDK を通じて*実際の*エージェントを駆動します — ターミナルで実行するのと同じ `claude` と `codex` の CLI で、すべてのツール、スキル、MCP サーバーがそのまま使えます。モデル（Opus / Fable / GPT）と reasoning effort をチャットごとに選択し、セッション途中で切り替え、あるいはそのチャットの履歴を引き継いだ新しいセッションとして再起動できます。
- **自分の作業をテストするエージェント。** スロットは実際のアプリを起動できます — Unity や Unreal では、2 台目のディスプレイ上でライブエディタ + サイドカーサーバーが立ち上がり、スロットごとのポートでエディタ内 MCP サーバー経由でエージェントに駆動されます — エージェントは推測するのではなく、UI をクリックし、ログを読み、変更を検証します。カスタムエンジンにも対応しています。
- **永続的でアーカイブ可能なチャット。** すべてのチャットは永続的なトランスクリプトです。閉じればそのスロットは解放され、後で全履歴を保ったまま再オープンできます。
- **チャットごとのターミナルとクリック可能なコード。** そのチャットのワークスペースに紐づいた埋め込みターミナルと、VS Code や Cursor で開く `file.ts:42` 形式のリンク。
- **自律的、しかし無謀ではない。** エージェントはスロット内の安全な作業を自動実行し、リスクのある操作（`git push` / `p4 submit`、PR を開く、ワークスペース外の操作、ネットワーク呼び出し）ではあなたのために一時停止します。許可はチャットごとに、永続的に、そして取り消し可能です — MCP サーバーも対象です。
- **完全にローカライズ。** インターフェース全体が 8 言語（英語、スペイン語、フランス語、ドイツ語、日本語、韓国語、簡体字中国語、ブラジルポルトガル語）で提供され、言語メニューからいつでも切り替えられます。
- **マルチリポジトリ。** 複数のリポジトリを並行して扱え、それぞれが独自のスロットプール、色、プロバイダー、ブランチ規約を持てます。

## PopBot が他と違う点

エージェント型コーディングツールはいくつかの型に分類される傾向があります。PopBot はそれとは異なる位置にあります。**ウォームなビルド状態とライブな人間の監督のもとで、多数の*本物の*エージェントを並列に走らせるためのローカルコックピット**です。

| 代わりに… | …PopBot は |
|---|---|
| **ターミナルや IDE の中の 1 エージェント** — 一度に 1 つの作業ツリーで 1 つのタスク | **一度に多数のエージェント**、それぞれが専用のウォームなスロットに隔離され、1 つのウィンドウから操縦できるライブなフリートとしてすべて可視化 |
| **非同期のクラウドエージェント** — 不透明でリモート。タスクを送信して PR を待つ | **ローカルでライブ** — 各エージェントの作業を見守り、間違った方向を早期に察知でき、*あなたの実際のアプリ*（2 台目の画面上のエンジンエディタ）を駆動して本物の end-to-end テストができる |
| **DIY の `tmux` + worktree のやりくり** — 並列だが手動で、新しいチェックアウトのたびにエンジンの数分にわたる再インポート税を払う | **管理されたウォームスロット** — 再利用される copy-on-write なワークスペースがアセットキャッシュをホットに保ち、ブランチ/ワークスペースのライフサイクル、SCM パネル、コードレビューをまとめて処理 |
| **エージェントオーケストレーションのフレームワーク** — エージェントシステムを*構築する*ためのツールキット | **仕上がった、意見を持ったアプリ** — あなたの受信箱とレビューループに接続済み。組み立てるためのライブラリではなく、設計段階から人間参加が前提 |

そして重要な点として、PopBot は Claude Code や Codex を置き換えません — それらを**実行するのです**。あなたがすでに信頼している、まさにそのエージェント（そしてまさにその CLI バージョン）を、オーケストレーション、隔離、監督で包んだうえで、一度に何体も使えるようになります。

## ワークスペースの構造

![PopBot UI anatomy](../../images/anatomy.png)

| 領域 | 内容 |
|---|---|
| **受信箱 — チケットとレビュー** | 割り当てられたチケット（Linear / Jira / GitHub Issues）とあなたを待っているレビュー（GitHub PR / Swarm チェンジリスト）を、優先度順に表示。ワンクリックでチャットを生成。 |
| **スロット** | ウォームで隔離されたワークスペースのプール — copy-on-write な作業コピー*に加えて*永続的なビルド状態（ゲームエンジンの場合はホットなアセットキャッシュ）。チャットは作業中に 1 つをリースし、閉じるときに返却する。 |
| **チャットアーカイブ** | 過去のすべてのチャット。検索可能で、全履歴を保ったまま再オープンできる。 |
| **チャットサムネイル** | 開いているすべてのチャットのライブなストリップ — ステータス（実行中 / 完了 / あなた待ち / エラー）で色分け。 |
| **チャット** | フォーカスされたエージェントセッション — 文章、ツール呼び出し、インラインのコード diff がライブでストリーミングされる。 |
| **チャットごとのターミナル** | そのチャットのワークスペースに向けられた埋め込みターミナル、手動コマンド用。 |
| **SCM パネル** | 作業ツリー / チェンジリストのステータス、コミット、ファイル diff、ワンクリックのコミット / プッシュ / PR / レビュー操作。 |

## PopBot のある一日

**フィーチャーチケット。** チケットが受信箱に届く。クリックすると → PopBot が `you/eng-123-…` でチャットを開き、スロットをリースし、チケットを *In Progress* に移動し、エージェントに完全な説明を渡す。エージェントはコードを書き、スロット内でアプリを実行して検証し、プッシュ前にあなたの OK を待って一時停止する。あなたは SCM パネルで diff をレビューし、**PR をプッシュ**を押す。

**並行するバグ。** それが動いている間に、バグレポートが届く。2 つ目のチャットを生成する — 独自のスロット、独自のブランチ — 2 体のエージェントはお互いのツリーに一切触れることなく同時に作業する。サムネイルストリップには両方が表示される。1 つは緑（完了）、もう 1 つは青（実行中）。

**レビュー依頼。** チームメイトの PR（または Swarm チェンジリスト）があなたの Reviews タブに現れる。クリックすると → 即座に**リポジトリ不要**のレビューチャットが開き、エージェントは diff*と*周辺のコードを読み、実際のバグを探し、GitHub や Swarm にインラインレビューを投稿する — その間もあなたの 2 つのビルドチャットは動き続けている。

**翌日また再開する。** 完了したチャットを閉じてスロットを解放する。翌朝、アーカイブからフィーチャーチャットを再オープンしてレビューフィードバックに対応する — エージェントは会話全体とワークスペースをそのまま保った状態で再開する。

→ 完全な解説（フィーチャー、バグ、レビューのフロー、そしてスロット、copy-on-write ワークスペース、再オープンが内部でどう動くか）は **[Feature & Workflow Guide](GUIDE.md)** にあります。

## インストール

署名済みのビルド済みインストーラーは **[popbot.app](https://popbot.app)** で入手できます。

- **macOS** — 署名 & notarize 済みの `.dmg`（Apple silicon）
- **Windows** — 署名済みの `.exe` インストーラー
- **Linux** — `.deb` パッケージ

アプリはリリースチャンネルから自動アップデートされます。自分でビルドしたい場合は [ソースからビルド](#ソースからビルド) を参照してください。

## ソースからビルド

```bash
npm install
npm run dev        # 開発モードでアプリを実行
npm run package    # 自分のプラットフォーム向けの署名済みインストーラーをビルド
```

**必要条件**

- **macOS、Windows、または Linux。** macOS が最も検証されているプラットフォームです（2 台目ディスプレイでのテスト対象アプリのワークフローは macOS のアクセシビリティ API に依存しています）。Windows と Linux もサポートされ、出荷されています — Windows/WSL のセットアップについては [WINDOWS.md](WINDOWS.md) を参照してください。
- **Node 20 以上**（Node 20 / 22 はネイティブモジュールの再コンパイルを回避できます。Windows の注意事項を参照）。
- **`claude`** および/または **`codex`** の CLI（エージェントのバックエンド）、加えて **`git`**、GitHub のフローには **`gh`**。Perforce の場合は **`p4`** CLI。
- 認証情報（Linear、Jira、GitHub、Helix Swarm）は**あなたのマシン上にローカルに**、アプリ自身のデータベースに保存されます — このリポジトリには決して保存されません。
- 任意: ゲームプロジェクト向けの Unity または Unreal エディタ、VS Code / Cursor、iTerm。

## 自分たち用に作り替える

PopBot はリファレンス実装として公開されており、そのまま採用するのではなく、フォークして適応させることを意図しています。その形は汎用的です — **エージェント + 隔離された、ウォームな copy-on-write スロット + キューとしての受信箱 + テスト対象アプリ** — そしてコードは*小さな共通インターフェースの背後にあるプロバイダー*として構成されているため、他の部分に触れずに 1 つの部分を入れ替えられます。**MIT ライセンス**です。一般的なアプローチは、中核となるアイデアを保ちつつ、具体的な実装を置き換えることです。

- **テスト対象アプリを入れ替える。** Unity と Unreal は「エージェントにアプリを実行・検証させる」という考え方の 2 つの実装にすぎません。カスタムエンジンのフックはすでにスロットの識別情報を起動コマンドに渡しているので、それをあなたの Web アプリ、CLI、テストハーネスに向ければよいのです。*(`src/shared/gameEngine.ts`、`src/main/ipc/apps.ts`)*
- **受信箱の接続先を変える。** Linear、Jira、GitHub Issues は具体例です。1 つのインターフェースを実装して登録すれば、トラッカーを追加できます。*(`src/main/tickets/`)*
- **ソースコントロールを追加・入れ替える。** Git と Perforce に加えて、プロバイダーの基底クラスを拡張してください。呼び出し側は決してプロバイダー id ではなく*能力（capabilities）*で分岐します。*(`src/main/scm/`)*
- **アクションとプロンプトを再構成する。** ブランチ規約、PR/レビューのフロー、そして種として与えるすべてのプロンプトは、Preferences 内で編集可能なテンプレートです — コードは不要です。
- **コアは維持する。** ウォームスロット、copy-on-write ワークスペース、永続的なチャット、ハードコードされた権限フロア、並列エージェントのコックピットは、揺るがない背骨です。

**[Feature & Workflow Guide](GUIDE.md)** は各接続点の背後にある理由を説明し、**[Architecture](ARCHITECTURE.md)** ドキュメントはコード内でそれをどこに見つけるかを示します。
