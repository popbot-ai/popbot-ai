# PopBot を設定する

PopBot のすべては、アプリ内の **Preferences**（タイトルバーの歯車、または `⌘,`）を通じて設定されます — 手で編集する設定ファイルはありません。このガイドは、ナビゲーションが並べている順序で各パネルを一通り説明します。これはおおよそ、初めてセットアップする際の順序でもあります。

> 入力する認証情報（Linear、Jira、GitHub、Perforce など）は、**あなたのマシン上にローカルに**、アプリ自身のデータベースに保存されます — このリポジトリには決して保存されません。

## 連携

ここには 2 つの独立したグループがあります。Tickets キューに供給する**チケットソース**と、スロットが起動できる**ゲームエンジン**です。

![Integrations — Linear](../../images/preferences_integrations1.png)

### チケットソース

1 つのアクティブなイシュートラッカーが Tickets キューに供給します。パネル上部のセレクターから選んでください。下の設定フォームがそれに応じて切り替わります。一度にアクティブになれるトラッカーは 1 つだけです。

- **Linear** — API キーを貼り付けます（*linear.app → Settings → API* から取得）。任意で**チームキー**（例: `ENG`）を設定してチケットフィードを 1 つのチームにスコープしたり、**プロジェクト**を選んでさらに絞り込んだりできます。保存するとキーが検証され、どのユーザーとして接続したかが表示されます。
- **Jira** — サイト URL（`https://your-domain.atlassian.net`）、アカウントのメールアドレス、API トークン（*id.atlassian.com → Security → API tokens* から取得）を入力します。任意で**プロジェクト**にスコープしたり、**JQL** フィルタ（例: `labels = backend`）を追加したりできます。保存すると、永続化する前に認証情報が検証されます。
- **GitHub** — GitHub Issues にはここでの認証情報は不要です。プロバイダーは、レビューや git アクションのためにすでに認証済みの `gh` CLI を呼び出し、キューは [リポジトリ](#リポジトリ) で設定されているのと同じリポジトリにまたがります。フォームは、`gh` がインストールされ認証されていることを確認し、何個のリポジトリをカバーしているかを報告するステータスチェックです。

認証情報を持つ各トラッカーは、**保存**時に永続化する前にそれを検証し、*Connected / Not connected* のステータスピルを表示します。

### ゲームエンジン

単一選択のチケットソースとは異なり、エンジンは**独立**しています — Unity、Unreal、カスタムエンジンを同時に有効化できます。有効化された各エンジンは、チャットバーにその Run ボタンを追加し、チャットのスロットワークスペースからエディタを起動します。

- **Enabled** — そのエンジンの Run ボタンをチャットバーに表示（または非表示に）するエンジンごとのチェックボックス。
- **Detected installs / Editor binary**（Unity、Unreal） — PopBot はインストール済みのエディタ（Unity Hub / Epic のインストール）をスキャンし、**再スキャン**リンクを提供します。検出されたバージョンを選ぶか、絶対パスの**エディタバイナリ**を入力してドロップダウンを上書きできます。
- **Run command**（カスタム） — プロジェクトディレクトリで実行される自由形式のシェルコマンドで、**macOS / Linux** と **Windows** の別々のバリアントがあるため、1 つの設定でクロスプラットフォームに対応できます。カスタムエンジンには自動検出はありません。PopBot は `POPBOT_SLOT` 環境変数を通じてスロットの識別情報をあなたのコマンドに渡すので、自分自身の「実行して検証する」フローを配線できます。
- **Project subpath** — エンジンプロジェクトのパスで、ワークスペースルートからの相対パス（Unity のプロジェクトフォルダ、`.uproject` を保持するフォルダ、またはカスタムコマンドが実行される cwd）。ワークスペースルート自体がプロジェクトである場合は空欄のままにします。
- **Use MCP + Base MCP port**（Unity、Unreal） — **Use MCP** チェックボックスがオンのとき、エディタはエージェントが駆動できるようエディタ内 MCP サーバーを指すように起動されます。並列のスロットが決して衝突しないよう、各スロットは**独自のポート**を得ます。ポートは `basePort + (slotId − 1)`（スロット 1 → base、スロット 2 → base + 1、…）です。**Base MCP port** フィールドはスロット 1 のポートを設定し、デフォルトは **Unreal は 8000**、**Unity は 8080**（それぞれのエンジンの MCP プラグインのデフォルトに一致）で、クリアするとそのデフォルトに戻ります。
- **Show project path in title bar**（Unity） — Unity プロジェクトに小さなエディタスクリプトを配置する**タイトルバースクリプトのインストール**ボタン。これにより、開いている各エディタがそのタイトルバーに完全なプロジェクトパスを表示し、スロットのウィンドウを見分けやすくなります。このスクリプトはコミットしても安全です。

> **Slack** と **Sentry** は、配線された受信箱ソースとしてではなく、接続スタブのままなので、今日ここにはパネルとして表示されません。構造的な変更なしに再有効化できます。詳しくは [Feature & Workflow Guide](GUIDE.md) の末尾の注記を参照してください。

## エージェント

新規作成されたチャットのデフォルトモデルの **reasoning effort**（既存のチャットは、チャットコンポーザーで変更するまで自身の設定を保持します）。

![Agents](../../images/preferences_agents.png)

- **Claude** と **Codex** で独立して、さらに次の 2 つで別々に effort を設定します。
  - **新規チャット** — 汎用チャットとチケットチャット。
  - **コードレビュー** — PR レビューチャット、再レビューのフォールバックチャット、レビュー通知。

effort が高いほど、より深い推論とより徹底したツール使用になりますが、コストとレイテンシは高くなります。レビューはフィーチャー構築とは異なる深さを求めることが多いため、この分割があります。

## ランタイムとスロット

このパネルは**添付ファイルの保持**を制御します。（スロットプールのサイジングは現在リポジトリごとになっており、[リポジトリ](#リポジトリ) にあります — そちらの注記を参照してください。）

![Runtime & slots](../../images/preferences_slots.png)

- **Keep attachments for** — チャットに添付したファイルや画像を PopBot 自身のストレージにどれだけ保持するか（デフォルト 60 日、範囲 1〜365）。添付ファイルは PopBot のストレージにコピーされるため、元のファイルが移動した後もチャット履歴から開き続けられます。起動時のスイープがこのウィンドウより古いコピーを削除するため、フォルダが無制限に肥大化することはありません。

> 上のスクリーンショットは、スロットプールのサイジングがリポジトリごとのフローに分割される前のものかもしれません。

## リポジトリ

各チャットは**リポジトリ**内で生きています。このパネルはあなたのリポジトリを一覧表示し、リポジトリごとのソースコントロール、スロット、copy-on-write ワークスペースを設定する場所です。

![Repositories](../../images/preferences_repositories.png)

- **Add Repository** はフォルダ優先のウィザードを開きます。フォルダを選ぶと、PopBot が**そのソースコントロール**（Git または Perforce）を検出し、それに応じて分岐します。その後、id、アクセントカラー、スロットのプレフィックス、スロット数を設定します。
  - **Git** リポジトリは**スロット**モード（再利用されるワークスペースのプール — デフォルトで、`slots × N` と表示される）または**エフェメラル**（チャットごとに新しいワークスペース）を選択します。スロットモードはチャットをまたいでビルドキャッシュをウォームに保ちます。
  - **Perforce** リポジトリは常にスロットモードです。ウィザードは P4 接続をキャプチャし、**ディスクのプリフライト**を実行し、同期済みツリーの凍結された**ベースイメージ**を構築します。その後、スロットはそのベースの copy-on-write な子として作成されます（下記参照）。
- **Copy-on-write ワークスペース。** スロットのワークスペースは、`shado`（PopBot のシャドウワークスペース層）経由でリポジトリの 1 つの**ベースイメージ**を共有し、変更したブロックだけを保存する copy-on-write なフォルダです。Windows では**differencing VHDX**、macOS と Linux ではネイティブの copy-on-write（APFS / reflink）です。テラバイト級のツリーで 10 個のスロットを持っても、かかるディスクはおおよそリポジトリ 1 つ分に各スロットの小さな差分を加えた程度です — これによって大規模な Perforce ツリーがそもそも参加できるようになります。ベースイメージは、Add-Repository ウィザードの一手順として一度だけ構築されます。
- **モードは恒久的です。** リポジトリのスロット対エフェメラルのモードは作成時に固定されます。切り替えると、進行中のチャットのワークスペースが孤立してしまいます。
- リポジトリを**編集**して、アクセントカラー、デフォルトのベースブランチ（Git）、Perforce のエージェント作業ディレクトリを変更したり、**Resize slots**（そのリポジトリのすべてのチャットが閉じられていることを条件に、プールを一度に 1 ワークスペースずつ拡大・縮小）したりできます。
- リポジトリを**削除**します。まだそれを参照しているチャットがある場合、確認ダイアログが警告します。

複数のリポジトリが並行して動作し、それぞれが独自のスロットプールとアクセントカラーを持ちます（この色は、そのリポジトリのスロットピルに色を付け、一目でチャットを見分けられるようにします）。各リポジトリカードには、そのソースコントロールプロバイダーとモードが表示されます。

## ソースコントロール

グローバルなソースコントロール設定と、編集可能なアクションテンプレート。リポジトリのプロバイダーはフォルダごとに検出され、両方が同時に使われることもあるため、Git と Perforce のパネルは並べて表示されます。

![Source control](../../images/preferences_source_control.png)

- **Change-view file limit**（共通） — 変更ビューにリストが打ち切られるまでに表示する最大ファイル数。Git と Perforce の両方に適用されます。

**Git**

- **Branch username** — 新しいブランチのプレフィックス: `<username>/<ticket>-<slug>`。
- **Action templates** — SCM パネルがエージェントに送る **Commit**、**Push PR**、**Push draft PR**、**Make ready**、**Address CR**、**Rebase onto base** のプロンプト。それぞれ `${name}` マクロ（`${branch}`、`${baseBranch}`、`${ticket}`、`${prnum}`、`${prurl}`…）をサポートします。

**Perforce**

- **Connection defaults** — `p4` バイナリのパス、デフォルトのサーバーポート、デフォルトのユーザー。Add-Repository → Perforce 接続ステップを事前入力します。
- **Transfer / submit options** — 並列同期スレッドの数、submit 時に未変更ファイルを revert するかどうか。
- **Swarm review poll interval** — Reviews パネルがあなたのレビューを待っているチェンジリストについて Helix Swarm をポーリングする頻度。これは**GitHub のポーリングとは独立**しており、**30 秒の下限**があります。共有 Perforce/Swarm サーバーへの負荷を大規模に軽減するには、この値を上げてください。
- **Perforce action templates** — Perforce パネルがエージェントに送る **CR**（Helix Swarm レビューを開く/更新する）、**Run tests**、**Review & commit** のプロンプトで、それぞれ `${name}` マクロを持ちます。

## 外部アプリ

チャットのアイコン行から PopBot が起動するデスクトップアプリで、すべてそのチャットのスロットワークスペースに向けられます。

![External apps](../../images/preferences_external_apps.png)

- **Terminal** — ターミナルアイコンのランチャーが開くターミナル（例: iTerm2）。
- **Terminal shell (Windows)** — アプリ内ターミナルパネルが使うシェル: PowerShell、コマンドプロンプト、または PowerShell 7。変更後に開いたターミナルに適用されます。
- **Code editor** — VS Code または Cursor。Edit ツールの行にあるクリック可能な `file.ts:42` リンクにも使われます。
- **Git client** — デフォルトは GitHub Desktop。
- **Chrome profile for URLs** — リンクを開く際に、（プロファイルの*ディレクトリ*名で指定した）特定の Chrome プロファイルに固定し、常にあなたの仕事用アカウントで開かれるようにします。

> エンジンのバイナリとその MCP オプションは、ここではなく [Integrations → Game engines](#連携) で設定します。

## プロンプトテンプレート

チャットが生成されたときに PopBot が送る最初のメッセージ。すべてのテンプレートは編集可能で、利用可能な `${name}` マクロのリファレンスカード付きです。（SCM パネルのアクションテンプレートは [ソースコントロール](#ソースコントロール) にあります。）

![Prompt templates](../../images/preferences_prompt_templates.png)

- **Start ticket** — チケット（Linear、Jira、GitHub Issues のいずれでも）からチャットを生成したときに発火します。マクロには `${ticketid}`、`${tickettitle}`、`${markdown}`、`${branch}`、`${slot}` が含まれます。
- **Start code review** — レビュー（GitHub PR または Helix Swarm チェンジリスト）からチャットを生成したときに発火します。デフォルトでは、エージェントにレビュースキルを使い、周辺のコードを（diff だけでなく）読み、チャットを読み取り専用として扱うよう指示します。
- **Re-review** — 既存のレビューチャットを再レビューしたときに発火し、エージェントを新しいコミットだけにスコープします。

これらをチューニングして、チームの規約、チェックリスト、トーンをエンコードしてください。

## コードレビュー

**Reviews** 受信箱の制御。このキューは、あなたのレビューを待っている GitHub PR と Helix Swarm チェンジリストを表面化させます。すでにレビュー済みの PR は自動的に除外されます。

![Code reviews](../../images/preferences_code_reviews.png)

- **Search cache window** — **+ Add** ピッカーが最近のチケットや PR をあいまい検索でマッチさせる遡り日数（大きいほど検索性が上がりますが、更新はやや遅くなり API 予算も多く消費します）。あなたに割り当てられたチケットは、このカットオフに関わらず常に含まれます。
- **Ignore by title** — PR をキューから除外する部分文字列（1 行に 1 つ、大文字小文字を区別しない）。
- **Ignore by GitHub author** — ミュートするボット/作者のログイン（1 行に 1 つ、例: `renovate[bot]`）。

> レビューの**ポーリングレート**はここではなく、プロバイダーごとに設定されます。Helix Swarm のポーリング間隔は [ソースコントロール → Perforce](#ソースコントロール) にあり、GitHub のポーリングとは独立しているため、GitHub を遅くすることなく共有 Perforce/Swarm サーバーを保護できます。

## 通知

アラートがどう表面化するか。

![Notifications](../../images/preferences_notifications.png)

- **VIP names** — そのメッセージが常に緊急優先度に引き上げられる人。表示名の大文字小文字を区別しない部分文字列としてマッチするので、名前は具体的にしておいてください。
- **Toast placement** — *中央上部、消える際にベルへ飛ぶ*（デフォルト）、または従来の右上コーナーのトースト。トグルは即座に適用されます。
- **Test new-item flow** — 実際のキュー項目のうちいくつかを一時的に NEW としてフラグ付けし、チップ/ピップの挙動をプレビューします（何も永続化されません）。これは一時的な開発補助です。

## 権限

各エージェントツールのグローバルなデフォルトと、自律モードの下限。

![Permissions](../../images/preferences_permissions.png)

- 各ツール（**Bash**、**Read**、**Write**、**Edit**、**Grep**、**Glob**、**WebFetch**、**WebSearch**、…）について: **Ask**（毎回確認する — デフォルト）、**Allow**（自動承認）、**Deny**（自動拒否）。
- **MCP サーバーごとの許可。** スロットのエディタ MCP サーバー（Unity、Unreal、またはエージェントが読み込む任意の MCP サーバー）も、同じ 3 つの方法で許可できます。あるスロットのエディタ MCP を一度許可すれば記憶され、その許可はここで可視かつ取り消し可能です — 生のネームスペースではなく `unityEditor → all tools` / `unrealEditor → all tools` として表示されます。PopBot はこの方法で Unity と Unreal のエディタ MCP を自動的に有効化します。ワイルドカードと異なるツールごとのルールは、オーバーライドとして保持されます。
- チャットごとのルール（許可カードの*Allow this chat* / *Deny this chat* から設定）はこれらのグローバル設定を上書きするので、他のすべての場所で許可しているツールでも、単一のチャットでロックダウンできます。

> ハードな拒否の下限 — `git push` / `p4 submit`、許可リストにないホストへのネットワーク、ワークスペース外のあらゆる操作 — はコードの中に存在し、ここでは上書き**できません**。誤って設定されたルールによって、エージェントが自力でメインラインに着地することはできません。

## 言語

PopBot のインターフェースは完全にローカライズされています。

- **Display language** — 言語メニューからインターフェースのロケールを切り替えます。各言語は、その言語自身の名前で一覧表示されます。出荷されているロケールは、英語、スペイン語、フランス語、ドイツ語、中国語（簡体字）、日本語、韓国語、ポルトガル語（ブラジル）です。ほとんどのテキストとメニューはすぐに更新されます。一部のシステム文字列は再起動後に更新が完了します。新しいウィンドウとアプリメニューもこの言語を使用します。
