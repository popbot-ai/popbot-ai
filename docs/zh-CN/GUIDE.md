# PopBot——功能与工作流指南

PopBot 是一个用于**并行运行多个 AI 编码智能体**的桌面驾驶舱。本指南介绍它所依托的理念——它为何存在、各部分如何协作、是什么塑造了这套设计，以及 Proof of Play 的团队如何在一个已上线的真实资产密集型项目中使用它。本文面向能够自行摸索出 UI 用法的工程师；这里的重点在于背后的推理过程，好让你能把这个工具改造成适合自己工作流的样子，而不是照本宣科。

把它改造以适应你自己的工作流是一种预期用途，而非事后补充。PopBot 是作为参考实现发布的——是一个可为你的团队改造的模板，而不是一个决策已被固定死的成品——这反映了我们对 AI 时代软件构建方式的一种看法：运行智能体舰队的团队，通常更适合拥有并重新塑造这个工具，而不是采用一个决策已被别人定死的工具。请把下文中每一部分背后的"为什么"当作一张地图，标示出如果要改造它，应该从哪里下手。[打造你自己的版本](#打造你自己的版本)详细介绍了怎么改、在哪改、为什么改。

## 我们为什么构建 PopBot

运行单个 AI 编码智能体很容易。而一旦你想要**同时运行不止一个**，就会出现三个问题：

1. **隔离性。** 两个智能体编辑同一份检出（checkout）会破坏彼此的工作。你不能让三个智能体共用一棵工作树——而在大型游戏项目上，你甚至负担不起三份完整检出。
2. **监督。** 智能体又快又基本靠谱，但对于 `git push`、`p4 submit` 或打开 PR 这类操作，"基本靠谱"还不够好。你需要在不可逆操作上设一道人工关卡——同时又不必对每一次文件编辑都亲自盯着。
3. **验证。** 能编译通过的代码不等于能工作的代码。对游戏来说尤其如此，唯一真正的测试就是*运行它*并点击体验一遍。一个看不到应用界面的智能体只能靠猜。

PopBot 就是为一个正在交付一款上线中的游戏的小团队而构建的，用来同时解决这三个问题。其核心洞见是：把每一个工作单元——一个工单、一个缺陷、一次评审——都当作一个**聊天（chat）**，为每个聊天配备自己隔离的**工作区**，以及（在需要时）自己独立运行的一份应用副本，默认让它们**自主运行但设有门禁**，并把整支舰队汇总呈现在同一个窗口中，让一个人也能同时带领十几个智能体。

这套设计是由一组具体的[用户故事](USER_STORIES.md)驱动的：*"作为一名工程师，我点击一个工单，一个智能体就会在正确的分支上开始处理它。"* *"作为一名评审者，我打开一个变更列表，就能获得一次真正的评审，而无需检出任何东西。"* *"作为一名负责人，我扫一眼那面缩略图墙，就知道哪些智能体需要我介入。"* 下文的一切都是为了服务这些目标而存在的。如果你理解了每一部分为何被设计成这个样子，你就会知道当你为自己的技术栈二次开发这个工具时，哪些部分该保留、哪些该替换。

## 核心概念

### 智能体与模型

每个聊天都由一个**智能体后端（agent backend）**驱动：

- **Claude Code** — 通过 Claude Agent SDK。模型：**Claude Opus**（默认）和 **Claude Fable**。
- **Codex** — 通过 OpenAI Codex SDK。模型：**GPT / Codex**。

PopBot 并不会重新实现这些智能体——它是在**驱动真实的智能体**，通过它们各自的官方 SDK，这些 SDK 封装的正是你在终端里会运行的那套 **`claude`** 和 **`codex`** 命令行工具。每个智能体的全部能力——工具、技能（skill）、MCP 服务器和子智能体（subagent）——在每个聊天中都可用，而且 PopBot 会与你机器上安装的那些 CLI 版本始终保持同步。只要在终端版 Claude Code 里能跑通的东西，在这里也能跑通。这是一个刻意的取舍：智能体进化很快，任何对其进行封装或分叉的方案都会逐渐过时。通过直接驱动这些 CLI，PopBot 可以免费继承它们的每一次升级。

在每个聊天中，你可以选择后端、**模型**和**推理强度**（`low` → `xhigh` / `max`——强度越高意味着思考越深入、工具使用越彻底，但成本/延迟也越高）。你可以设置合理的**默认值**——对*新聊天*和*代码评审*分别设置，因为评审所需的深度和功能构建不一样——并可以在具体任务需要时按聊天覆盖这些默认值。

有两个会话控制项对长时间运行的工作很关键：

- **会话中途切换。** 更改一个进行中聊天的模型或推理强度；PopBot 会重新配置智能体而不丢失上下文线索。
- **带上下文重启。** 启动一个*全新*的智能体会话，并用该聊天的记录（开头几轮加上最近几轮）作为初始上下文——当一个会话变得冗长或卡住时很有用。对话历史被完整保留，智能体只是获得了一个干净的运行时环境。

各项集成所需的凭据**存储在你本机**，位于应用自有的数据库中——绝不存于本仓库。

### 卡槽：热备、隔离、一次性的工作区

**卡槽（slot）**是并行处理的基本单元，也是 PopBot 的核心理念。运行 N 个智能体最朴素的做法是检出 N 份仓库——这要么会在共享的项目树上产生冲突，要么代价高达 N ×（检出时间 + 构建缓存）。卡槽正是"如何给智能体一个*真实、独立*的工作场所，同时又*已经是热备的*、还*便于回收*"这一问题的答案。

卡槽有三个属性，每一个都至关重要：

- **隔离（Isolated）。** 每个卡槽都是自己独立分支（或 Perforce 流）上的独立工作目录，因此 N 个智能体编辑 N 个分支，互不干扰。一个智能体的 `git reset` 碰不到另一个智能体的工作成果。
- **热备（Warm）。** 卡槽会保留跨多次使用而持续存在的有状态构建产物——对于游戏引擎而言，是它自己的导入/资产缓存；一个拥有自己数据目录的专属**边车服务器（sidecar server）**；已分配的**端口**；每卡槽的日志；以及在聊天处于活跃状态时，一个实时运行的**编辑器进程**。一个普通的工作目录只能给你隔离的*源代码*；而一个卡槽给你的是一个隔离、且*已经热备好*的构建、运行、测试场所。
- **一次性（Disposable）。** 卡槽是池化管理的。一个聊天在其生命周期内**租用**一个空闲卡槽，并在关闭时**归还**它。创建一个热备工作区代价很高；重复使用一个则几乎免费，因此 PopBot 会维护一个热备卡槽池，让工作在其中循环流转。

**为什么"热备"对引擎类工作来说是决定性的。** 一个游戏引擎会维护一个庞大的已处理资产缓存——Unity 的 `Library/`、Unreal 的 `DerivedDataCache`——通常有好几个 GB，生成代价高昂。一次全新检出，或一次让该缓存失效的分支切换，都会迫使引擎**重新导入整个项目**，可能耗时数分钟。如果每个任务、每次分支切换都要付出这个代价，你的智能体花在等待引擎上的时间就会超过写代码的时间。卡槽通过为每个卡槽提供**自己持久化的缓存**来消除这个代价：

- **把一个智能体切回它的卡槽只需几秒钟，而不是几分钟**——缓存已经是热的，只有真正发生变化的资产才需要重新处理。
- **卡槽可以让编辑器保持*运行状态*。** 一次"粘性（sticky）"复用（同一个卡槽、同一个分支）几乎能立即为智能体提供一个实时编辑器，而不是冷启动。
- **十个智能体不会争抢同一个导入缓存。** 每个卡槽都有自己的热备缓存，因此并行的游戏相关工作永远不会串行卡在单一的重新导入过程后面。

在任何分支切换之前，PopBot 都会运行一套**安全流程**——它会暂存（stash）未提交的工作、拒绝覆盖智能体拥有的提交、执行切换、再恢复状态——因此一次卡槽交接绝不会悄无声息地丢失工作。卡槽可以运行在**卡槽池**模式（复用，默认）或**一次性**模式（每个聊天使用全新工作区）下，如果你更愿意用热备状态换取一份干净的初始环境。

> **为什么这很重要：** 隔离性是让"同时十个智能体"从灾难性变为安全可控的关键。热备状态是让它*快*的关键。一次性则是让它*便宜*的关键。去掉其中任何一个，并行智能体就不再值得投入了。

### 写时复制：单个仓库磁盘上的无限份拷贝

只有当卡槽的*文件*足够廉价时，隔离性和热备状态才负担得起。在一个小仓库上，N 个 git 工作树没什么问题。但在一个 TB 级的游戏项目上——拥有庞大的资产库，而且在很多团队中用的是 **Perforce** 而非 Git——N 份真实拷贝会占用数百 GB，每份都要花上数分钟才能生成。这会让整个模型彻底失效。

因此，卡槽的工作区是一个**写时复制（copy-on-write）文件夹**。每个卡槽都共享同一份仓库的**基础镜像（base image）**，只存储它实际改动的数据块。实际效果是：

- **一份 TB 级项目树的全新、实时、完整拷贝可以在几秒钟内就绪**——不是浅层视图，而是真实可编辑的文件——释放起来同样迅速。
- **无限份拷贝的成本仅相当于一个仓库的磁盘占用。** 十个智能体在一个 1 TB 的项目上工作，不需要 10 TB 磁盘；他们只需要约 1 TB，外加每个卡槽自己的小增量。
- **它在 Windows、macOS 和 Linux 上表现一致**（通过 `shado`——PopBot 的影子工作区层——在 Windows 上使用差异化 VHDX，在其他平台上使用原生的写时复制文件系统），这也正是让 Perforce 项目树得以参与其中的关键。

正是这一部分，让卡槽这一理念从"一个只有少数几个工作树的 Web 仓库"扩展到"一支智能体舰队面对的 3A 级游戏项目树"成为可能。它也是最不起眼、但可以说是最重要的功能：没有廉价拷贝，热备隔离卡槽只是一种奢侈品；有了它，它们才能成为默认配置。

### 源代码管理：Git 与 Perforce

PopBot 把源代码管理当作隐藏在同一接口后的**提供方（provider）**，因为"在一个隔离分支上运行智能体，然后评审并落地这个改动"这件事，无论后端是 Git 还是 Perforce，形状都是一样的。两者都是一等公民：

- **Git** — 用于隔离的工作树、按聊天划分的分支、通过 `gh` CLI 发起的 PR，评审在 GitHub 上进行。
- **Perforce** — 每个聊天在写时复制的影子工作区之上使用流/分支，变更列表作为工作单元，评审在 **Helix Swarm** 上进行。Swarm 评审会和 GitHub PR 一起固定显示在同一个评审收件箱中，各自打开自己的评审聊天。

下文你会看到的这些概念——基线分支、git/源代码管理面板、模板化操作、评审收件箱——都是围绕这个通用接口编写的。凡是文字中写"分支"或"PR"的地方，如果你用的是 Perforce，请理解为"变更列表"或"Swarm 评审"；这个工作流是刻意做成完全一致的。

### 收件箱：一个队列，多个来源

收件箱是一种*理念*，而不是一种具体的集成：**你已分配的工作和待处理的评审，按优先级排序，每一项只需一次点击就能变成一个智能体聊天。** 为它提供数据的来源是可插拔的：

- **工单** — 分配给你的 **Linear** issue、**Jira** issue 和 **GitHub Issues**（GitHub Issues 支持较新，目前仍有一定实验性质）。点击其中一项，PopBot 就会命名一个分支、租用一个卡槽、把工单移动到*进行中*状态，并用它的描述为智能体提供初始信息。
- **评审** — 等待你评审的 **GitHub** 拉取请求和 **Helix Swarm** 变更列表。点击其中一项，一个无需仓库的评审聊天会立即打开。

新增一个数据来源并不会改变工作流——它只是往同一个队列里加了几行数据。这正是关键所在：以收件箱为队列的模型是通用的，具体的任务跟踪系统只是可以互换的默认选项。

### 无仓库聊天（用于代码评审）

不是每个聊天都需要工作区。**评审**一个改动是只读的——你不编辑，你只是阅读 diff 和周边代码，然后发表评论。因此评审聊天是**无仓库的**：它们瞬间生成，不租用任何卡槽，也不占用任何工作区。

这是一个刻意为之、非常重要的区分：

- 一个**构建聊天**（功能/缺陷）会租用一个卡槽，可能需要片刻才能热身完成，并在其生命周期内持有一个工作区。
- 一个**评审聊天**是**即时且免费的**——你可以同时打开五个评审聊天来梳理你的评审队列，而你的构建聊天不受任何干扰地继续运行。

这也意味着你的卡槽池被保留给真正需要隔离的工作。评审永远不会挤占构建工作的卡槽资源——当卡槽池受限于内存和磁盘时，这一点非常重要。

### 基线分支

当一个聊天确实需要编写代码时，它会从一个**基线（base）**分叉出去——在 Git 上通常是 `develop`/`main`，在 Perforce 上则是主干流。PopBot 会为每个仓库设置默认基线，记住你上一次的选择，让常见场景只需一次点击，并允许你在任务需要时从某条功能分支或发布分支上分叉。它会根据你的约定推导出新分支的名称——例如 `<username>/<ticket>-<slug>`——从而让分支保持一致且可追溯回其对应的工单。基线还驱动着后续操作："变基到基线（rebase onto base）"、"针对基线打开 PR / 评审"以及漂移检查，都以它为基准。

### 持久化、可归档的聊天

每个聊天都是本地存储的一份**持久化记录**——文字、工具调用、diff、权限决策，一应俱全。没有任何东西是短暂的。

- **关闭**一个聊天会释放其卡槽（把工作区腾给其他智能体使用），但会**保留一切**。该聊天会移动到**归档**中。
- 从归档中**重新打开**一个聊天会重新租用一个卡槽、恢复其分支，智能体会带着**完整历史**继续工作——你可以在数天之后重新捡起某个功能，处理评审反馈，而无需重新解释任何东西。如果它落到了一个*不同*的卡槽上，PopBot 会预先告知智能体这一点，让它干净利落地重新适应新的工作目录。
- 归档可以按名称、工单、分支和内容进行搜索。

由于回滚只是"再发一条消息"这么简单（没有破坏性的历史编辑），一个聊天会积累出一份完整、可审计的改动实现过程记录。

## 工作区剖析

![PopBot UI anatomy](../../images/anatomy.png)

| 区域 | 是什么 |
|---|---|
| **收件箱——工单与评审** | 已分配的工单（Linear / Jira / GitHub Issues）和等待你处理的评审（GitHub PR / Swarm 变更列表），按优先级排序。点击一行即可生成一个以其上下文为初始信息的聊天。 |
| **卡槽** | 热备工作区的池子。每个胶囊状标签都显示某个卡槽是空闲的，还是被某个聊天租用了。 |
| **聊天归档** | 每一个过去的聊天，均可搜索、可重新打开，且保留完整历史。 |
| **聊天缩略图** | 所有打开聊天的实时滚动预览——真实展示每个智能体当前正在做什么，按状态用颜色区分：蓝色 = 运行中，绿色 = 已完成，黄色 = 需要你处理，红色 = 出错，灰色 = 空闲。 |
| **聊天** | 聚焦的智能体会话——实时流式呈现的文字、工具调用和内联代码 diff。 |
| **每聊天终端** | 一个固定在该聊天工作区上的内嵌终端。 |
| **源代码管理面板** | 工作树/变更列表状态、近期提交记录、文件 diff，以及一键式的提交 / 推送 / PR / 评审操作。 |

因为每个聊天都一直显示在**缩略图条**上，而且各**列并排排列**，你永远不需要满屏寻找状态信息。颜色就是信号——蓝色 = 运行中，绿色 = 已完成，黄色 = 需要你处理，红色 = 出错——扫一眼就能知道哪些智能体在工作、哪些已经完成、哪些在**等你处理**。

但每个缩略图同时也是**对话的实时预览**，而不仅仅是一盏状态灯——因此一眼就能看出每个智能体*实际*在做什么。这正是让你能**尽早发现无用功**的关键：发现某个智能体走上了错误的方向，在它耗费大量时间和 token 之前及时纠正它，而不是在它"完成"之后才发现这是一条死路。这就是"监督一支舰队"和"被舰队打个措手不及"之间的区别。

### 为什么要用缩略图，为什么要统一到一个视图里

这种布局是针对一个具体问题的刻意解答，值得说明其背后的推理，因为这正是大多数工具做错的地方。

运行一个智能体是一项专注型任务：你盯着单个对话并作出回应。而运行*多个*智能体则是一项**监控**任务，监控有着不同的失败模式——瓶颈不在于你的打字速度，而在于你的注意力。一个悄悄跑偏的智能体会产生你必须去注意、理解、然后丢弃的工作成果。当有 N 个智能体时，"没注意到"这件事的代价会随 N 增长，而常规界面恰恰让"注意到"这件事变得困难：标签页一次只能显示一个智能体，而"启动后等待结果"的模式则会把所有智能体都藏起来，直到它们冒出一个结果。

因此这套设计坚持两点：

- **每个智能体始终可见。** 缩略图条一次性展示整支舰队，每个缩略图都是真实对话的实时视图，而不是一个转圈的加载动画。你应该能够退后一步，一眼扫过就把十几个智能体的状态尽收眼底——哪些在推进，哪些卡住了，哪些即将做出你想要阻止的事情。
- **状态是一种颜色，内容只需一瞥即可获得。** 颜色能在一秒之内回答"谁需要我？"；实时预览无需点击就能回答"这个智能体在做什么？"；并排排列的各列则让你可以深入其中任何一个，而不会失去对其他智能体的关注。这套界面是为*廉价的反复检查*而优化的，因为智能体一多，你就会不断地反复检查。

由此带来的回报是能够**尽早介入**。自主智能体最昂贵的错误不是崩溃——而是一个智能体信心满满地花上一个小时构建了错误的东西。一个持续呈现意图的视图，能把这种情况从"事后才发现"变成"中途纠正"。这正是这支舰队要始终显示在屏幕上，而不是藏在标签页或通知背后的全部原因。

## 在 Proof of Play 的实际使用情况

PopBot 不是一个实验室里的实验品。它是由 **Proof of Play** 团队在一个已上线的真实资产密集型项目中构建并每天使用的。这一起源解释了大多数设计选择，也是理解这个工具用途的最清晰方式。

实际效果很直接：卡槽模型——热备、隔离、写时复制的工作区——让并行的智能体工作在一棵庞大的资产树上变得可行，团队因此完成了更多工作。多个智能体可以同时运行，不会相互冲突，也不必在每次切换时付出引擎重新导入的代价，因此吞吐量提高了，而不是让并行处理变成了额外开销。

典型的一天大致是这样的：一位负责人打开那面缩略图墙，同时有四五个智能体在运行——有几个在攻克功能工单，一个在追查一个缺陷，一两个在做代码评审。这位负责人并不是每分钟都在写代码；他们在**盯着这支舰队**，只在关卡处（一次推送、一个 PR、一次高风险操作）以及某个缩略图变黄、或某个智能体明显跑偏时才介入。工单来自团队真实的任务跟踪系统；评审是团队其他成员会看到落地的真实 PR 和变更列表。

那个游戏项目所强加的硬性约束，恰恰正是最终被证明最重要的那些功能：

- **资产树规模巨大**，因此热备卡槽和写时复制工作区不是锦上添花——没有它们，在那棵项目树上运行一支智能体舰队根本负担不起。这正是这两个理念成为该工具骨干的原因。
- **引擎是判断"能否工作"的唯一真相来源**，因此一个无法启动并驱动运行中游戏的智能体，对大多数玩法相关的工作来说毫无用处。这正是被测应用集成存在的原因。
- **游戏用 Perforce 做源代码管理，工具链则用 Git**，因此不依赖具体提供方的源代码管理支持并非可有可无的选项。
- **需要一个人来带领多个智能体**，因此整个驾驶舱都是为"一眼看全局的监督"而优化的，而不是为深度的单会话专注而优化。

如果你的情况和这些有相通之处——庞大的项目树、需要测试的真实应用、超出单个智能体处理能力的工作量——这套设计会与你的需求高度契合，因为它正是为此而生的。如果不是，[打造你自己的版本](#打造你自己的版本)这一节讲的就是如何保留理念、替换具体做法。

关于适用范围的说明：那个项目最终没有取得商业上的成功，我们也不打算说它成功了。但它所面对的工程问题是真实存在的——一棵庞大的资产树、一支智能体舰队、一个团队——而 PopBot 中解决了这个问题的部分，正是本文档所记录的部分。这个工具的价值并不取决于那款游戏的商业结果，我们宁愿把这一点说清楚，也不愿含糊其辞地暗示更多。

## 端到端工作流

### 一个功能工单

1. **通知 → 收件箱。** 一个分配给你的工单出现在**工单**收件箱中（PopBot 会轮询 Linear / Jira / GitHub Issues，按优先级和截止日期排序）。通知铃铛会标记它。
2. **一键开始。** 点击该工单所在行。PopBot 会打开一个默认指向你的仓库和基线（记住了你上一次的选择）的**新建聊天**对话框——确认，或调整智能体/模型/推理强度。
3. **卡槽分配。** 因为这个聊天将会编写代码，PopBot 会**租用一个卡槽**：它挑选一个空闲工作区，根据工单推导出分支名 `you/eng-123-<slug>`，并将该工作区切换到这个分支上（先执行暂存安全流程）。
4. **工单自动推进。** 工单会自动被移动到**进行中**（幂等、无需等待确认），让你的看板无需切换上下文就能反映真实状态。
5. **智能体启动。** 智能体收到一条预填的初始消息（你可自定义的*启动工单*模板，填入工单标题、描述和分支），然后开始工作：探索代码、进行编辑、运行命令——这一切都在其卡槽的工作区内完成。
6. **在卡槽中验证。** 对于游戏相关的改动，智能体会**在其卡槽中启动应用**（一个引擎编辑器 + 第二块屏幕上的边车服务器）并实际操作这个功能——点击浏览 UI、读取日志、截图——而不是靠猜测它是否可行。
7. **有门禁的收尾。** 当准备好推送时，智能体会**暂停**（推送是一个受门禁控制的操作）。缩略图会变成黄色（"需要你处理"）。
8. **你来评审并发布。** 打开**源代码管理面板**，阅读 diff，然后点击**推送 PR**（或**推送草稿**）。该操作会向智能体发送一条预填好的指令，智能体随即推送分支并针对你的基线打开 PR / Swarm 评审。

在这整个过程中，你并没有一直盯着它——你在同时处理另外两个工单。你只在关卡处介入了一次。

### 一个缺陷工单

缺陷处理流程就是循环周期更紧凑的功能处理流程，它也展示了**并行处理能力**：

1. 一份缺陷报告到达（可以来自一个工单，也可以是你手动用缺陷描述启动一个聊天）。
2. 生成一个聊天 → 它会租用**属于自己的**卡槽和分支。你正在进行中的功能聊天完全不受影响——不同的工作区、不同的分支。
3. 智能体**通过在其卡槽中运行应用**来复现这个缺陷，找出原因，修复它，然后重新运行以确认问题已经消失。
4. 你瞄一眼**缩略图条**：功能聊天是绿色的（已完成，等待你推送），缺陷聊天是蓝色的（运行中）。两个智能体，两棵隔离的项目树，零冲突。
5. 当缺陷聊天暂停等待批准时，推送这个修复。

### 一次代码评审

1. **通知 → 评审。** 一位同事请求你进行评审。该 PR（GitHub）或变更列表（Swarm）出现在**评审**收件箱中。
2. **即时的无仓库聊天。** 点击它 → 一个**评审聊天**立即打开——不租卡槽、不检出、无需等待。它会以*启动代码评审*模板作为初始信息（阅读周边代码而不只是 diff；追溯相关系统；寻找真正的缺陷、竞态条件、边界情况、安全和性能问题）。
3. **真正的评审。** 智能体阅读 diff **以及**周边代码，对正确性进行推理，并在 GitHub 或 Swarm 上发布**内联评论**和一个结论（批准 / 请求修改）——然后在聊天中为你总结出需要留意的问题。
4. **稍后再次评审。** 如果作者推送了修复，点击**重新评审**：PopBot 会聚焦到既有的评审聊天，并让智能体**只**关注新的提交，验证之前每一条评论是否真的得到了解决，然后更新其评审结论。

这一切都发生在你的构建聊天持续运行的同时——评审从不占用卡槽。

### 重新打开一个已归档的聊天

工作很少是一次性完成的。重新打开这个流程是一等公民功能：

1. 一个功能聊天推送了它的 PR；你**关闭**它以释放卡槽。它现在位于**归档**中（记录完整保留）。
2. 两天后，这个改动收到了评审意见。在归档中找到该聊天（按工单、分支或文字内容搜索）并**重新打开**它。
3. PopBot 会**重新租用一个卡槽**，把该聊天的分支恢复到工作区中，智能体会带着**完整历史**继续工作——它已经知道自己构建了什么、为什么这么构建。如果它落到了与之前不同的卡槽上，PopBot 会帮它适应新的工作目录。
4. 粘贴或总结评审反馈。智能体会处理这些意见，在卡槽中重新测试，然后推送更新——无需重新引导，也不会丢失上下文。

由于分支、记录和推理过程都被持久保留，重新捡起一项任务只需几秒钟，而不需要重新解释一遍。

## 集成的源代码管理与评审

源代码管理是通过每个提供方的原生 CLI 深度接入的——GitHub 用 **`gh`/`git`**，Perforce 用 **`p4`** 和 Swarm API——因此智能体所做的一切都是你团队能在常规场所看到的真实活动。

- **评审收件箱。** 等待你评审的 GitHub PR 和 Swarm 变更列表（以及你自己最近提交的内容），会作为一键式的聊天来源出现。
- **PR / 评审状态徽章。** 每个链接到某个改动的聊天都会显示一个实时状态徽章——开放中 / 已合并 / 已关闭 / 草稿——点击即可在 GitHub 或 Swarm 中打开它。
- **源代码管理面板。** 对于任意构建聊天，可以查看工作树/变更列表状态、近期提交记录，以及逐文件 diff。点击某个文件即可显示完整的统一 diff 浮层。
- **一键式操作。** 模板化、可编辑的操作会向智能体发送预填好的指令：**提交（Commit）**、**推送 PR（Push PR）**、**推送草稿 PR（Push draft PR）**、**标记就绪（Make ready）**、**处理评审意见（Address CR）**（处理评审评论）、**变基到基线（Rebase onto base）**。每一个都会展开诸如 `${branch}`、`${baseBranch}`、`${ticket}`、`${prnum}` 和 `${prurl}` 之类的变量，让智能体获得恰好所需的信息。
- **针对你的基线创建。** 推送会针对该聊天所配置的基线打开 PR（或 Swarm 评审），并按照你的分支命名约定命名。

评审是一条独立、经过优化的路径（参见[一次代码评审](#一次代码评审)）：

- **无需仓库、即时完成**——不租卡槽，不检出。几秒钟内即可梳理完一整队评审。
- **阅读上下文，而不只是 diff**——评审模板会引导智能体阅读周边代码、追溯相关系统，并寻找缺陷/竞态条件/边界情况/安全/性能问题，而不是走过场式地盖章通过。
- **发布在你团队实际工作的地方**——在 GitHub 或 Swarm 上发布内联评论和一份提交的评审结论。
- **重新评审的范围是限定的**——在第二次评审时，智能体只检查新的提交，并在更新评审结论之前确认之前的每一条评论都真的得到了解决。
- **完全可定制**——*启动代码评审*和*重新评审*的提示词都是可编辑的模板，因此你可以调整严谨程度、检查清单和语气以符合你团队的标准。评审*流程本身*（你的团队希望如何完成一次 GitHub 或 Perforce 评审）由你自己提供——PopBot 会提出建议并可以提供一个样例，但这个标准始终掌握在你的团队手中。

## 在卡槽中测试：被测应用

一个构建聊天的卡槽不只是一个文件夹——它是一个可以**运行并检查**工作成果的场所：

- **每聊天终端。** 一个固定在该聊天工作区上的内嵌终端（xterm + 真实的 PTY）。可以运行测试、检查日志，或在智能体工作时手动执行命令。切换聊天时它会持续保留。
- **编辑器集成。** 记录中每一个 `path/to/file.ts:42` 形式的引用都是一个可点击的链接，会针对该聊天的工作区在 **VS Code** 或 **Cursor** 中打开对应位置。
- **被测应用。** 一个卡槽可以启动**真实的应用**，让智能体去驱动它，而不是靠猜。对于一个 Web 应用、一个 CLI 或一个服务来说，这大多是智能体自己就能完成的事——它在卡槽的终端中运行你的构建和测试命令，访问运行中的服务器，读取输出结果。PopBot 不需要对这些应用了解任何特殊之处；智能体会像你自己操作一样来处理它们。游戏**引擎**是需要额外处理的情况，因为编辑器是一个长期运行的 GUI 进程，有自己的资产缓存，也没有天然的命令行式"运行并检查"循环。因此对于 **Unity** 和 **Unreal**，PopBot 会启动一个实时编辑器 + 边车服务器，把它放在第二块屏幕上，并通过一个**编辑器内 MCP 服务器**将其暴露给智能体。每个运行中的编辑器都会获得**从其卡槽派生出的专属 MCP 端口**——因此一个智能体只会与*自己*的编辑器通信，永远不会碰到另一个卡槽的编辑器——PopBot 会自动把每个聊天的智能体连接到相应的端点（这一连接是内存中的，因此不会写入源代码管理系统）。**自定义**引擎接入的是同一套机制：PopBot 会把卡槽身份信息传递给你的启动命令，由你来配置智能体如何驱动它。在所有这些情况下，智能体都能实际操作该应用——点击 UI、读取日志、截图、断言行为——而 PopBot 负责管理编辑器的生命周期（启动服务器、进行健康检查、启动编辑器、放置其窗口、在释放时将其关闭），并根据可用内存来控制并发实例的数量。

这就是"智能体*认为*自己的改动可行"和"智能体已经*亲眼见证*它可行"之间的区别。这一点丝毫不局限于游戏领域——Web 及其他开发同样是一等公民的使用场景。游戏引擎只是恰好携带了额外的状态（热资产缓存、作为被测应用的编辑器），系统必须对此有所感知，而正是这份额外状态，让它们成为展现这个工具新颖之处最鲜明的例子：热备卡槽、写时复制工作区，以及一个智能体可以驱动的运行中应用。

## 权限与安全

自主性设有硬性底线：

- **自动允许（静默执行）：** 在**卡槽工作区内**的读取、编辑和 shell 命令，对卡槽自身服务的调用（包括其编辑器 MCP），以及智能体内部操作。智能体只管工作。
- **始终受门禁（暂停等你处理）：** `git push` / `p4 submit` / 重置 / 强制操作，任何**工作区之外**的操作，打开 PR 或评审，删除临时目录之外的内容，发送消息（Slack/邮件），触碰系统或智能体配置，以及对未列入白名单主机的网络调用。
- **其余一切：** 提示你来决定。

当你批准某项操作时，可以授予**一次性**、**本次会话**或**永久性**（始终允许该工具/目标）的权限。MCP 服务器也可以用同样的方式授权——批准一次某个卡槽的编辑器 MCP，它就会被记住，该授权在"偏好设置 → 权限"中可见且可撤销（PopBot 会以这种方式自动启用 Unity/Unreal 编辑器 MCP）。授权可以按聊天或全局设置，且全部**可撤销**。硬性拒绝底线（推送/提交、网络访问、工作区之外的操作）写在代码里，UI 规则无法覆盖它——因此一条配置错误的授权规则不可能让智能体自行将改动落地到主干。

## 本地化

PopBot 的整个界面——菜单、设置、对话框，一切——都完全本地化了。该应用发布了**十二种语言**：英语、西班牙语、法语、德语、日语、韩语、简体中文、巴西葡萄牙语、俄语、意大利语、波兰语和乌克兰语——可随时从语言菜单切换，无需重启。如果你对 PopBot 进行二次开发，每种语言都是一个独立的消息目录，因此添加或调整一种语言是一次范围明确的改动，而不是在整个 UI 里大海捞针。

## 偏好设置

一切都在应用内配置（无需编辑配置文件）：

- **智能体** — 默认模型和推理强度，对新聊天和代码评审分别设置。
- **仓库** — 通过一个以文件夹为起点、能感知源代码管理系统的向导来添加/编辑仓库：路径、提供方（Git/Perforce）、基线分支或流、颜色、卡槽前缀、工作区目录、卡槽池模式还是一次性模式。
- **运行时与卡槽** — 池大小（同时运行多少个智能体）、预创建/删除卡槽、附件保留期、写时复制工作区的基础镜像刷新。
- **集成** — 连接 Linear、Jira、GitHub 和 Helix Swarm（凭据存储在本地）；按提供方可配置的评审轮询频率；保存前先测试连接。
- **源代码管理** — 分支命名约定、默认基线，以及可编辑的操作模板。
- **外部应用** — 终端（iTerm）、编辑器（VS Code / Cursor）、引擎二进制文件及各引擎选项（包括编辑器 MCP 基础端口）、可选的用于 URL 路由的 Chrome 配置文件。
- **提示词模板** — 每一个初始提示词（启动工单、启动/重新评审，以及每个操作）都可编辑，并附有变量参考卡片。
- **权限** — 审阅并撤销永久授权，包括按 MCP 服务器设置的许可。
- **通知** — 提示消息的位置和提醒行为。
- **语言** — 切换界面语言。

> 若需要逐面板的详细参考及配图说明，请参见**[配置指南](CONFIGURATION.md)**。

## 打造你自己的版本

改造 PopBot 是一种主要的预期用途。它是作为参考实现发布的，其设计反映了我们对 AI 时代软件构建方式的一种看法：团队拿到一个可运行的原型，理解它*为什么*是这个样子，然后围绕自己的技术栈、工具和约定对其进行重塑，而不是采用一个决策已被固定死的工具。

它的形态是通用的：**智能体 + 隔离的热备写时复制卡槽 + 作为队列的收件箱 + 被测应用。** 这种模式适用于大多数同时运行多个编码智能体的团队。它采用 **MIT 许可**，且结构上便于二次开发——代码组织为*位于小型通用接口之后的各个提供方（provider）*，因此可以添加或替换某一部分而不必触碰其余部分。总体思路是：保留核心理念，替换具体实例。

下面列出了这些可替换的接缝，以及每一个的*怎么改、在哪改、为什么改*。每一个都是一个带有可插拔实现的接口；实际做法是参照现有实现进行模式匹配，然后添加你自己的实现。

- **替换被测应用。** *为什么：* 整个理念的核心就是让智能体*运行并验证*你的应用，而"你的应用"对每个人来说都不一样。*在哪：* `src/shared/gameEngine.ts`（引擎描述符、MCP 接线）和 `src/main/ipc/apps.ts`（启动 + 生命周期）。Unity 和 Unreal 是两种实现；**自定义引擎**钩子已经把卡槽身份信息（`POPBOT_SLOT`、派生端口）传递给了你的启动命令，因此接入你的 Web 应用、CLI 或测试框架就只是"填写启动命令，以及智能体如何与它通信"这么简单。
- **把收件箱指向别处。** *为什么：* 以收件箱为队列是持久不变的理念；具体的任务跟踪系统只是一个细节。*在哪：* `src/main/tickets/`——在 `provider.ts` 中实现 `TicketSource` 接口，把你的任务跟踪系统的数据规范化为共享的 DTO，并在 `registry.ts` 中注册它（该文件的头部注释明确写道："添加一个任务跟踪系统，只需在这里加一行，再加上它的 `*Source.ts` 模块"）。Linear、Jira 和 GitHub Issues 就是现成的实现范例。渲染进程从不根据提供方 id 做分支判断，因此你不需要触碰 UI。
- **添加或替换源代码管理。** *为什么：* "隔离一个改动、评审它、落地它"这件事与具体提供方无关；Git 和 Perforce 只是两种后端而已。*在哪：* `src/main/scm/`——参照 `gitProvider.ts` / `perforceProvider.ts`，扩展 `SourceControlProvider` 基类（`provider.ts`）。无法干净抽象的行为，是通过**能力（capabilities）检测**来处理的，而不是 `if (provider === …)` 这样的判断，因此即使是一个非常不同的版本控制系统，也可以选择接入自己的客户端 UI，而不需要调用方为它做特殊处理。
- **替换评审平台。** *为什么：* 评审应该落地在你团队本来就会去查看的地方。*在哪：* `src/main/reviews/` 背后的评审提供方（GitHub PR 通过 `git/reviews.ts`，Swarm 变更列表通过 `p4/swarmReviews.ts`）。*评审流程本身*——你的团队希望如何完成一次评审——被刻意地**没有**内置到工具中；这是一项由你提供的、因团队而异的技能（skill），因此 PopBot 只提出建议并给出样例，从不强加你的标准。
- **重新配置操作和提示词。** *为什么：* 分支命名约定、PR/评审流程，以及你如何向智能体交代任务，都是因团队而异的。*在哪：* 不需要写代码——git 操作模板和每一个初始提示词（启动工单、启动/重新评审）都**可以在"偏好设置"中编辑**，并附有变量参考卡片。可以调整严谨程度、检查清单和语气。
- **保留核心。** *为什么：* 这些是让整套系统得以运转的理念，也是你应该最不愿意去改动的部分。热备卡槽、写时复制工作区（`src/main/shado/`）、持久化聊天、硬编码的权限底线，以及并行智能体驾驶舱，是这套系统持久不变的主干。其他一切都是可以变动的。

关于进程边界、IPC，以及每个子系统所在的位置，请阅读**[架构文档](ARCHITECTURE.md)**——这是一张地图，帮你找到你想要改动的那道接缝。关于对象模型（Chat、Slot、AgentSession 及其生命周期），请参见**[核心模型](CORE_MODEL.md)**。

对于同时运行多个智能体的团队来说，这是一个可运行的起点，本身就是为了被拆解、并围绕不同的工作流重新搭建而设计的。
