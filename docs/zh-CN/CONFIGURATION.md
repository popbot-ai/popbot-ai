# 配置 PopBot

PopBot 中的一切都在应用内通过**偏好设置**（标题栏中的齿轮图标，或 `⌘,`）进行配置——没有需要手动编辑的配置文件。本指南按照导航列出的顺序逐一介绍每个面板，这个顺序大致也是你第一次搭建时应该设置它们的顺序。

> 你输入的凭据（Linear、Jira、GitHub、Perforce 等）**存储在你本机**的应用自有数据库中——绝不存于本仓库。

## 集成

这里包含两组独立的设置：为工单队列提供数据的**工单来源**，以及卡槽可以启动的**游戏引擎**。

![Integrations — Linear](../../images/preferences_integrations1.png)

### 工单来源

单个活跃的问题跟踪系统会为工单队列提供数据。从面板顶部的选择器中选择它；下方的配置表单会随之切换。同一时间只有一个任务跟踪系统处于活跃状态。

- **Linear** — 粘贴一个 API 密钥（来自 *linear.app → Settings → API*）。可以选择性地设置一个**团队密钥（Team key）**（例如 `ENG`），把工单数据限定到一个团队，还可以选择一个**项目**进一步缩小范围。保存时会验证该密钥，并显示它连接的是哪个账号。
- **Jira** — 输入你的站点 URL（`https://your-domain.atlassian.net`）、账号邮箱，以及一个 API 令牌（来自 *id.atlassian.com → Security → API tokens*）。可以选择性地限定到某个**项目**，并添加一个 **JQL** 过滤条件（例如 `labels = backend`）。保存前会先验证凭据，再进行持久化保存。
- **GitHub** — GitHub Issues 在这里不需要任何凭据：该提供方会直接调用你已经为评审和 git 操作认证过的 `gh` CLI，其队列涵盖的仓库范围与[仓库](#仓库)下配置的仓库相同。此处的表单只是一个状态检查，用于确认 `gh` 已安装并已认证，并报告它涵盖了多少个仓库。

每个配置了凭据的任务跟踪系统都会在**保存**时先验证凭据再持久化，并显示一个"已连接 / 未连接"的状态标签。

### 游戏引擎

与单选的工单来源不同，各引擎是**相互独立**的——你可以同时启用 Unity、Unreal 和一个自定义引擎。每个启用的引擎都会在聊天工具栏中添加一个**运行**按钮，用于从该聊天的卡槽工作区启动其编辑器。

- **已启用** — 一个按引擎设置的复选框，用于显示（或隐藏）该引擎在聊天工具栏中的运行按钮。
- **检测到的安装 / 编辑器二进制文件**（Unity、Unreal）——PopBot 会扫描已安装的编辑器（Unity Hub / Epic 安装），并提供一个**重新扫描**链接；可以选择检测到的某个版本，或输入一个绝对路径的**编辑器二进制文件**来覆盖下拉列表中的选择。
- **运行命令**（自定义）——一条在项目目录中运行的自由格式 shell 命令，分别提供 **macOS / Linux** 和 **Windows** 两种变体，因此一份配置就能跨平台使用。自定义引擎没有自动检测功能；PopBot 会通过一个 `POPBOT_SLOT` 环境变量把卡槽身份信息传递给你的命令，以便你搭建自己的"运行并验证"流程。
- **项目子路径** — 引擎项目相对于工作区根目录的路径（Unity 项目文件夹；存放 `.uproject` 的文件夹；或者自定义命令运行时所在的工作目录）。如果工作区根目录本身*就是*项目所在位置，则留空。
- **使用 MCP + 基础 MCP 端口**（Unity、Unreal）——当**使用 MCP** 复选框开启时，编辑器启动时会指向一个编辑器内 MCP 服务器，以便智能体可以驱动它。每个卡槽都会获得**专属端口**，确保并行卡槽之间永不冲突：端口计算方式为 `basePort + (slotId − 1)`（卡槽 1 → 基础端口，卡槽 2 → 基础端口 + 1，……）。**基础 MCP 端口**字段设置的是卡槽 1 的端口；默认值为 **Unreal 的 8000** 和 **Unity 的 8080**（分别匹配各引擎 MCP 插件的默认值），清空后会恢复为该默认值。
- **在标题栏中显示项目路径**（Unity）——一个**安装标题栏脚本**按钮，会向你的 Unity 项目中放入一个小型编辑器脚本，让每个打开的编辑器都在其标题栏中显示完整的项目路径，从而更容易区分各个卡槽窗口。该脚本可以安全地提交到版本控制中。

> **Slack** 和 **Sentry** 目前仍只是连接占位符，而不是接通的收件箱数据来源，因此目前没有作为面板显示在这里。它们可以在不做结构性改动的情况下重新启用；参见[功能与工作流指南](GUIDE.md)结尾处的说明。

## 智能体

新创建聊天的默认模型**推理强度**（已有的聊天会保留自己原有的设置，直到你在聊天输入框中改变它）。

![Agents](../../images/preferences_agents.png)

- 分别为 **Claude** 和 **Codex** 独立设置推理强度，并按以下两类分别设置：
  - **新聊天** — 普通聊天和工单聊天。
  - **代码评审** — PR 评审聊天、重新评审的兜底聊天，以及评审通知。

推理强度越高意味着思考越深入、工具使用越彻底，但成本和延迟也越高。评审所需的深度往往和功能构建不一样——这正是要分开设置的原因。

## 运行时与卡槽

此面板控制**附件保留期**。（卡槽池大小现在按仓库单独设置，位于[仓库](#仓库)之下——详见那里的说明。）

![Runtime & slots](../../images/preferences_slots.png)

- **附件保留时长** — 你附加到聊天中的文件和图片在 PopBot 自有存储中保留多长时间（默认 60 天，可设置范围为 1–365）。附件会被复制到 PopBot 的存储中，因此即使原始文件被移动，它们也能继续从聊天记录中打开；启动时的清理任务会删除超出这一时限的副本，防止该文件夹无限增长。

> 上面的截图可能拍摄于卡槽池大小设置被拆分为按仓库配置之前。

## 仓库

每个聊天都存在于一个**仓库**中。此面板列出你的各个仓库，也是配置每个仓库的源代码管理、卡槽和写时复制工作区的地方。

![Repositories](../../images/preferences_repositories.png)

- **添加仓库**会打开一个以文件夹为起点的向导：选择一个文件夹，PopBot 会**检测其源代码管理系统**（Git 或 Perforce）并据此分支处理。然后你可以设置 id、强调色、卡槽前缀和卡槽数量。
  - **Git** 仓库可以选择**卡槽**模式（一个可复用的工作区池——默认模式，显示为 `slots × N`）或**一次性**模式（每个聊天使用全新工作区）。卡槽模式能让构建缓存在各聊天之间保持热备状态。
  - **Perforce** 仓库始终使用卡槽模式。向导会采集 P4 连接信息，运行一次**磁盘预检**，并为已同步的项目树构建一个冻结的**基础镜像**；随后各卡槽将作为该基础镜像的写时复制子项被创建（见下文）。
- **写时复制工作区。** 一个卡槽的工作区是一个写时复制文件夹，通过 `shado`（PopBot 的影子工作区层）与其他卡槽共享同一份仓库的**基础镜像**，只存储自己改动的数据块：在 Windows 上是**差异化 VHDX**，在 macOS 和 Linux 上是原生的写时复制机制（APFS / reflink）。在一棵 TB 级的项目树上运行十个卡槽，大致只需一个仓库的磁盘占用，外加每个卡槽自己的小增量——这也正是让庞大的 Perforce 项目树得以参与其中的关键。基础镜像会在"添加仓库"向导的某一步中构建一次。
- **模式是永久性的。** 一个仓库是"卡槽模式"还是"一次性模式"在创建时就已固定；切换模式会让进行中聊天的工作区变成孤儿数据。
- **编辑**一个仓库可以更改其强调色、默认基线分支（Git）或 Perforce 智能体工作目录，也可以**调整卡槽数量**（一次增减一个工作区，前提是该仓库下所有聊天都已关闭）。
- **删除**一个仓库；如果仍有聊天引用它，确认对话框会给出警告。

多个仓库可以并排运行，每个都有自己的卡槽池和强调色（该颜色会给该仓库的卡槽标签染色，方便你一眼分辨各个聊天属于哪个仓库）。每张仓库卡片都会显示其源代码管理提供方和模式。

## 源代码管理

全局的源代码管理设置和可编辑的操作模板。Git 和 Perforce 面板并排显示，因为一个仓库的提供方是按文件夹检测的，两者可能同时在用。

![Source control](../../images/preferences_source_control.png)

- **变更视图文件数量上限**（共享）——变更视图中在列表被截断之前最多显示的文件数量。同时适用于 Git 和 Perforce。

**Git**

- **分支用户名** — 新分支的前缀：`<username>/<ticket>-<slug>`。
- **操作模板** — 源代码管理面板向智能体发送的提示词，用于**提交（Commit）**、**推送 PR（Push PR）**、**推送草稿 PR（Push draft PR）**、**标记就绪（Make ready）**、**处理评审意见（Address CR）**和**变基到基线（Rebase onto base）**。每一个都支持 `${name}` 形式的宏（`${branch}`、`${baseBranch}`、`${ticket}`、`${prnum}`、`${prurl}`……）。

**Perforce**

- **连接默认值** — `p4` 二进制文件路径、默认服务器端口，以及默认用户，这些会预填到"添加仓库 → Perforce 连接"步骤中。
- **传输/提交选项** — 并行同步线程数，以及提交时是否还原未改动的文件。
- **Swarm 评审轮询间隔** — 评审面板轮询 Helix Swarm 以获取待你评审的变更列表的频率。这**独立于 GitHub 的轮询**，且有一个**30 秒的下限**；在大规模场景下，可以调高这个值以减轻共享 Perforce/Swarm 服务器的负载。
- **Perforce 操作模板** — Perforce 面板向智能体发送的提示词，用于 **CR**（打开/更新一个 Helix Swarm 评审）、**运行测试**和**评审并提交**，每个都带有 `${name}` 形式的宏。

## 外部应用

PopBot 从聊天的图标行中启动的桌面应用，全部指向该聊天的卡槽工作区。

![External apps](../../images/preferences_external_apps.png)

- **终端** — 终端图标启动器打开的是哪个终端（例如 iTerm2）。
- **终端 Shell（Windows）** — 应用内终端面板所使用的 shell：PowerShell、命令提示符或 PowerShell 7。适用于该改动之后新打开的终端。
- **代码编辑器** — VS Code 或 Cursor；也用于工具调用行中可点击的 `file.ts:42` 链接。
- **Git 客户端** — 默认使用 GitHub Desktop。
- **用于打开 URL 的 Chrome 配置文件** — 将链接固定在某个特定的 Chrome 配置文件中打开（按其配置文件*目录*名称指定），确保链接始终在你的工作账号中打开。

> 引擎二进制文件及其 MCP 选项是在[集成 → 游戏引擎](#集成)下配置的，而不是在这里。

## 提示词模板

PopBot 在聊天生成时发送的第一条消息。每个模板都可编辑，并附有一张该模板可用的 `${name}` 宏参考卡片。（源代码管理面板的操作模板位于[源代码管理](#源代码管理)之下。）

![Prompt templates](../../images/preferences_prompt_templates.png)

- **启动工单** — 当你从一个工单生成聊天时触发，无论其来源是什么（Linear、Jira 或 GitHub Issues）。宏包括 `${ticketid}`、`${tickettitle}`、`${markdown}`、`${branch}` 和 `${slot}`。
- **启动代码评审** — 当你从一次评审（一个 GitHub PR 或一个 Helix Swarm 变更列表）生成聊天时触发。默认设置会指示智能体使用评审技能（skill）、阅读周边代码（而不只是 diff），并把该聊天当作只读处理。
- **重新评审** — 当你对一个既有的评审聊天进行重新评审时触发；它会把智能体的关注范围限定在新的提交上。

调整这些模板，以固化你团队的约定、检查清单和语气。

## 代码评审

**评审**收件箱的控制项。该队列会呈现等待你评审的 GitHub PR 和 Helix Swarm 变更列表；你已经评审过的 PR 会被自动剔除。

![Code reviews](../../images/preferences_code_reviews.png)

- **搜索缓存窗口** — **+ 添加**选择器进行模糊匹配时，向前追溯多少天的近期工单和 PR（数值越大，可搜索的范围越广，但刷新会略慢，API 预算消耗也更多）。分配给你的工单始终会被包含在内，不受此截止时间限制。
- **按标题忽略** — 会将某个 PR 从队列中剔除的子字符串（每行一个，不区分大小写）。
- **按 GitHub 作者忽略** — 需要屏蔽的机器人/作者登录名（每行一个，例如 `renovate[bot]`）。

> 评审的**轮询频率**是按提供方分别配置的，而不是在这里：Helix Swarm 的轮询间隔位于[源代码管理 → Perforce](#源代码管理)之下，独立于 GitHub 的轮询，因此可以在不拖慢 GitHub 轮询的情况下保护共享的 Perforce/Swarm 服务器。

## 通知

提醒信息如何呈现。

![Notifications](../../images/preferences_notifications.png)

- **VIP 姓名** — 其消息始终会被提升为紧急优先级的人员。匹配方式是显示名称中不区分大小写的子字符串，因此请保持姓名足够具体。
- **提示消息位置** — *顶部居中，关闭时飞向铃铛图标*（默认），或经典的右上角提示消息。该开关会立即生效。
- **测试新项目流程** — 临时将队列中的几个真实项目标记为"新"，以预览徽标/圆点提示的行为（不会持久化任何数据）。这是一个临时的开发辅助功能。

## 权限

每个智能体工具的全局默认设置，以及自主模式下的权限底线。

![Permissions](../../images/preferences_permissions.png)

- 对每个工具（**Bash**、**Read**、**Write**、**Edit**、**Grep**、**Glob**、**WebFetch**、**WebSearch**……）：**询问**（每次都提示——默认设置）、**允许**（自动批准）或**拒绝**（自动驳回）。
- **按 MCP 服务器设置许可。** 一个卡槽的编辑器 MCP 服务器（Unity、Unreal，或智能体加载的任何 MCP 服务器）可以按同样的三种方式获得许可。批准一次某个卡槽的编辑器 MCP，该许可就会被记住，且这里可以查看和撤销该授权——显示为 `unityEditor → all tools` / `unrealEditor → all tools`，而不是原始的命名空间。PopBot 会以这种方式自动启用 Unity 和 Unreal 编辑器 MCP；如果某条按工具设置的规则与通配符规则不同，会作为一条覆盖规则保留下来。
- 按聊天设置的规则（通过权限卡片上的*允许此聊天*/*拒绝此聊天*设置）会覆盖这些全局设置，因此单个聊天可以锁定一个你在其他地方已经全局允许的工具。

> 一道硬性拒绝底线——`git push` / `p4 submit`、对未列入白名单主机的网络访问、任何工作区之外的操作——写在代码里，这里**无法**覆盖它，因此一条配置错误的规则不可能让智能体自行将改动落地到主干。

## 语言

PopBot 的界面已完全本地化。

- **显示语言** — 从语言菜单切换界面语言，菜单中每种语言都以其自身的名称列出。已发布的语言包括英语、西班牙语、法语、德语、简体中文、日语、韩语和葡萄牙语（巴西）。大多数文字和菜单会立即更新；少数系统字符串会在重启后完成更新。新窗口和应用菜单也会使用这一语言。
