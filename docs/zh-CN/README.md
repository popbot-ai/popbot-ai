## 为什么选择 PopBot

运行单个 AI 编码智能体很简单。而同时运行多个智能体会带来单个智能体不会遇到的问题：要保持它们的工作相互隔离，避免互相覆盖；要真正测试它们构建出的东西；要对其进行评审；还要对不可逆操作设置门禁，确保没有智能体在无人监督的情况下执行此类操作。

PopBot 就是为此而生的编排层。它把工单和评审请求转化为一键式的智能体会话，为每个智能体提供一个隔离的工作区（拥有自己的工作副本——对于游戏项目，还拥有自己正在运行的被测应用），默认自主运行智能体，同时对高风险操作设置人工把关，并将每一份记录、diff、终端和日志汇总到同一个窗口中。操作者只需浏览各个列，批准被门禁的操作，然后发布即可。

它由 **Proof of Play** 的一个小团队构建，并在一个已上线的真实资产密集型生产项目中每天使用。这正是它得到验证的环境：数十 GB 的资产、真实的源代码管理、真实的截止日期。卡槽模型——热备、隔离、写时复制（copy-on-write）的工作区——正是让并行运行智能体在该项目中变得可行的关键，也提升了团队在同一时间能完成的工作量。我们发布并维护 PopBot 作为参考实现：它不是一个可以照单全收的成品，而是一个可以拿来为你自己的技术栈和工作流重新塑造的模板。这也反映了我们对 AI 时代软件构建方式的一种看法——运行智能体舰队的团队，更适合拥有并改造工具，而不是采用一个固定不变的工具。它采用 MIT 许可，且组织结构便于二次开发；参见[打造你自己的版本](#打造你自己的版本)。

## 核心功能

### 带实时缩略图的多聊天视图

每个打开的聊天都会一直显示在屏幕上——并排**列**上方是一条**实时缩略图**条。每个缩略图都是该聊天的真实、持续更新的视图（而不只是一个状态点），并按状态用颜色区分：运行中、已完成、等待你处理、出错。你可以一眼看出*每个智能体在做什么*、谁需要你——并且可以**尽早发现走偏的方向**，在浪费时间和 token 之前就进行纠正。一个人就能从一个窗口监督一整支舰队。

### 热备卡槽——无需重新导入税的并行智能体

每个工作中的聊天都会租用一个**卡槽（slot）**——一份持久化的工作副本，外加它自己的热备构建状态，一次创建、重复使用。对于游戏引擎来说，这意味着卡槽保留着自己的热资产缓存（Unity 的 `Library`、Unreal 的 DDC），并可以让编辑器保持运行，因此把智能体切回它的卡槽只需**几秒钟，而不是几分钟的重新导入**。十个智能体在真正的分支隔离下运行，而不会互相争抢同一个导入缓存。[卡槽的工作原理 →](GUIDE.md#卡槽热备隔离一次性的工作区)

### 单个仓库磁盘上的无限份拷贝

一个卡槽的工作区是一个**写时复制（copy-on-write）文件夹**：每个卡槽共享同一个基础镜像，只存储自己所改动的部分。因此，一份**TB 级**游戏项目树的全新、实时、完整拷贝可以在**几秒钟**内就绪——是真实可编辑的文件，而不是浅层视图——而无限份拷贝所占的磁盘空间只相当于一个仓库。它在 **Windows、macOS 和 Linux** 上都可用，也正是这一点让庞大的 Perforce 项目树得以加入这支舰队。[为什么这很重要 →](GUIDE.md#写时复制单个仓库磁盘上的无限份拷贝)

### Git 与 Perforce，内置评审功能

源代码管理是隐藏在同一接口后的**提供方（provider）**：**Git**（工作树、分支、通过 `gh` 发起的 PR）和 **Perforce**（基于影子工作区的流、变更列表、**Helix Swarm** 评审）都是一等公民。一个作用域限定在*每个聊天自己工作区*内的源代码管理面板，会显示该分支确切的状态、提交记录和逐文件 diff。一键式的模板化操作（**提交（Commit）**、**推送 PR（Push PR）**、**标记就绪（Make ready）**、**处理评审意见（Address CR）**、**变基到基线（Rebase onto base）**）会向该聊天的智能体发送预填好的指令，其中 `${branch}` / `${ticket}` / `${prnum}` 均已自动填充。

### 一个收件箱，多个来源

整个流程汇聚一处：你的**收件箱**——来自 **Linear**、**Jira** 和 **GitHub Issues** 的已分配工单，以及以 **GitHub PR** 和 **Swarm 变更列表**形式等待你处理的评审 → 隔离卡槽中**进行中**的智能体工作 → **推送**并打开 PR / 评审 → **归档**已完成的聊天 → 之后**重新打开并重启**，且保留完整历史。点击一个工单，PopBot 就会命名分支、租用卡槽、把工单移动到*进行中（In Progress）*状态，并为智能体提供初始信息——然后一路推进到变更合并，再回到收件箱。[端到端工作流演示 →](GUIDE.md#端到端工作流)

## 更多功能

- **真正的 Claude Code 和 Codex——而非重新实现。** 每个聊天都通过官方 SDK 驱动*真实的*智能体——正是你在终端里运行的那套 `claude` 和 `codex` 命令行工具，所有工具、技能（skill）和 MCP 服务器都完整保留。可按聊天选择模型（Opus / Fable / GPT）和推理强度，可在会话中途切换，也可以用该聊天的历史记录重新初始化一个全新会话。
- **能测试自己工作成果的智能体。** 一个卡槽可以启动真实的应用——对于 Unity 和 Unreal，是在第二块屏幕上运行的实时编辑器 + 边车（sidecar）服务器，由智能体通过**每卡槽专属端口**上的编辑器内 MCP 服务器进行驱动——因此智能体会点击浏览 UI、读取日志、验证自己的改动，而不是靠猜测。也支持自定义引擎。
- **持久化、可归档的聊天。** 每个聊天都是一份持久化记录；关闭它可以释放其卡槽，之后重新打开时完整历史依然保留。
- **每聊天专属终端和可点击代码。** 一个固定在该聊天工作区上的内嵌终端，以及可在 VS Code 或 Cursor 中打开的 `file.ts:42` 链接。
- **自主，但绝不鲁莽。** 智能体会在自己的卡槽内自动运行安全的工作，遇到任何有风险的操作都会暂停等你——`git push` / `p4 submit`、打开 PR、任何工作区之外的操作、网络调用。授权按聊天、持久且可撤销——MCP 服务器也不例外。
- **完全本地化。** 整个界面提供十二种语言（英语、西班牙语、法语、德语、日语、韩语、简体中文、巴西葡萄牙语、俄语、意大利语、波兰语、乌克兰语），可随时从语言菜单切换。
- **多仓库。** 并排驱动多个仓库，每个仓库都有自己的卡槽池、颜色、提供方和分支约定。

## PopBot 的与众不同之处

智能体化编码工具往往可以归为几类。PopBot 处在一个不同的位置：一个**用于并行运行多个*真实*智能体的本地驾驶舱，具备热备构建状态和实时人工监督。**

| 相比于…… | ……PopBot |
|---|---|
| **在终端或 IDE 中运行单个智能体**——一次只在单个工作树里处理单个任务 | **同时运行多个智能体**，每个都隔离在自己的热备卡槽中，全部作为一支实时舰队呈现，可从一个窗口统一操控 |
| **异步云端智能体**——不透明且远程；提交一个任务，等待一个 PR | **本地且实时**——观察每个智能体工作，尽早发现走偏的方向，而且它还能驱动*你的真实应用*（第二屏幕上的引擎编辑器）进行真正的端到端测试 |
| **手工用 `tmux` + 工作树来回折腾**——并行但全靠手动，而且每次全新检出都要付出引擎数分钟的重新导入税 | **托管的热备卡槽**——重复使用、写时复制的工作区，让资产缓存保持热备状态，分支/工作区生命周期、源代码管理面板和代码评审都为你打理好 |
| **智能体编排框架**——用于*构建*智能体系统的工具包 | **一款成品化、有主见的应用**，与你的收件箱和评审流程直接对接——设计上就是人在回路（human-in-the-loop），而不是一个需要自行拼装的库 |

关键的一点是：PopBot 并不是要取代 Claude Code 或 Codex——它是在**运行它们**。你得到的正是你已经信赖的那些智能体（以及你现有的 CLI 版本），只不过可以同时运行多个，并且外面包裹着编排、隔离和监督能力。

## 工作区剖析

![PopBot UI anatomy](../../images/anatomy.png)

| 区域 | 是什么 |
|---|---|
| **收件箱——工单与评审** | 已分配的工单（Linear / Jira / GitHub Issues）和等待你处理的评审（GitHub PR / Swarm 变更列表），按优先级排序。一键即可生成一个聊天。 |
| **卡槽** | 热备、隔离工作区的池子——一个写时复制的工作副本，*加上*持久化的构建状态（对于游戏引擎而言，是它自己的热资产缓存）。聊天在工作期间租用一个卡槽，关闭时归还。 |
| **聊天归档** | 每一个过去的聊天，均可搜索、可重新打开，且保留完整历史。 |
| **聊天缩略图** | 所有打开聊天的实时条带——按状态用颜色区分（运行中 / 已完成 / 需要你处理 / 出错）。 |
| **聊天** | 聚焦的智能体会话：文字、工具调用和内联代码 diff，实时流式呈现。 |
| **每聊天终端** | 一个指向该聊天工作区的内嵌终端，用于手动执行命令。 |
| **源代码管理面板** | 工作树 / 变更列表状态、提交记录、文件 diff，以及一键式的提交 / 推送 / PR / 评审操作。 |

## PopBot 的一天

**一个功能工单。** 一个工单出现在你的收件箱中。点击它 → PopBot 在 `you/eng-123-…` 上打开一个聊天，租用一个卡槽，把工单移动到*进行中*，并把完整描述交给智能体。它编写代码，在自己的卡槽中运行应用进行验证，并在推送前暂停等待你的确认。你在源代码管理面板中审阅 diff，然后点击**推送 PR**。

**并行处理一个缺陷。** 就在那个任务运行的同时，一份缺陷报告进来了。生成第二个聊天——它有自己的卡槽、自己的分支——两个智能体同时工作，彼此的项目树永远不会互相触碰。缩略图条同时显示两者：一个绿色（已完成），一个蓝色（运行中）。

**一个评审请求。** 一位同事的 PR（或 Swarm 变更列表）出现在你的评审标签页中。点击它 → 一个即时的**无需仓库（repoless）**评审聊天随即打开，智能体读取 diff*以及*周边代码，寻找真正的缺陷，并在 GitHub 或 Swarm 上发布内联评审——与此同时，你那两个构建聊天仍在继续运行。

**第二天继续处理。** 关闭已完成的聊天以释放它们的卡槽。第二天早上，从归档中重新打开该功能聊天以处理评审反馈——智能体会带着完整的对话记录和工作区状态恢复工作。

## 安装

已签名的预构建安装程序可在 **[popbot.app](https://popbot.app)** 获取：

- **macOS** — 已签名并经过公证的 `.dmg`（Apple 芯片）
- **Windows** — 已签名的 `.exe` 安装程序
- **Linux** — `.deb` 软件包

该应用会从其发布渠道自动更新。若想自行构建，请参见[从源代码构建](#从源代码构建)。

## 从源代码构建

```bash
npm install
npm run dev        # 以开发模式运行该应用
npm run package    # 为你的平台构建一个已签名的安装程序
```

**环境要求**

- **macOS、Windows 或 Linux。** macOS 是使用最充分的平台（第二屏幕被测应用工作流依赖 macOS 辅助功能 API）；Windows 和 Linux 同样受支持并已发布——Windows/WSL 的搭建说明见 [WINDOWS.md](WINDOWS.md)。
- **Node 20+**（Node 20 / 22 可避免原生模块重新编译；详见 Windows 相关说明）。
- **`claude`** 和/或 **`codex`** CLI（智能体后端），以及 **`git`**，若使用 GitHub 相关流程还需要 **`gh`**。若使用 Perforce，则需要 **`p4`** CLI。
- 凭据（Linear、Jira、GitHub、Helix Swarm）**存储在你本机**的应用自有数据库中——绝不存于本仓库。
- 可选：用于游戏项目的 Unity 或 Unreal 编辑器；VS Code / Cursor；iTerm。

## 打造你自己的版本

PopBot 是作为参考实现发布的，意在被二次开发和改造，而非照单全收。它的形态是通用的——**智能体 + 隔离的热备写时复制卡槽 + 作为队列的收件箱 + 被测应用**——代码组织为*位于小型通用接口之后的各个提供方（provider）*，因此团队可以替换其中一部分而不必触碰其余部分。它采用 **MIT 许可**。总体思路是：保留核心理念，替换具体实例：

- **替换被测应用。** Unity 和 Unreal 是"让智能体运行并验证应用"这一理念的两种实现。自定义引擎钩子已经把卡槽身份信息传递给了你的启动命令——把它指向你的 Web 应用、CLI 或测试框架即可。*（`src/shared/gameEngine.ts`、`src/main/ipc/apps.ts`）*
- **把收件箱指向别处。** Linear、Jira 和 GitHub Issues 只是实现范例；只需实现一个接口并注册它，即可添加一个新的任务跟踪系统。*（`src/main/tickets/`）*
- **添加或替换源代码管理。** 与 Git 和 Perforce 一样，扩展提供方基类；调用方永远根据*能力（capabilities）*分支处理，而不是根据提供方 id。*（`src/main/scm/`）*
- **重新配置操作和提示词。** 分支命名约定、PR/评审流程，以及每一个初始提示词都是"偏好设置"中可编辑的模板——无需编写代码。
- **保留核心。** 热备卡槽、写时复制工作区、持久化聊天、硬编码的权限底线，以及并行智能体驾驶舱，是这套系统持久不变的主干。

**[功能与工作流指南](GUIDE.md)**解释了每个可替换接缝背后的思路；**[架构文档](ARCHITECTURE.md)**则说明了在代码中何处能找到它们。
